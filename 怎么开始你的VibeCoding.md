
# 什么是Harness？
### 1.进化历程
- **Prompt 阶段（青铜）：** 猜谜语。每次都要重写长长的指令，看 AI 心情给结果。
    
- **Context 阶段（白银）：** 喂资料。你会把文件贴给 AI，或者用 `@` 符号，但 AI 还是会偶尔“串味”。
    
- **Harness 阶段（黄金）：** **建规则**。你不仅给它资料，还给它装上了“护栏”和“加速器”。
  ![[Image_enj4pqenj4pqenj4.png]]
  
	Prompt 是求 AI 办事，Context 是给 AI 资料，而 Harness 是给 AI 盖了一座实验室。
	


### 2.Harness核心定义

当你发现agent犯了一个错，我们花时间去==**工程化**==一个解决方案，让agent不再犯这个错误。
为agent套上马具约束它做该做的事情的整个流程

### 5.具体流程是怎么走的
1. 感知 (Perception) 
   - **概念：** AI 进入你的项目，通过扫描代码库和读取你准备的 **Harness 文档**（如 `ARCHITECTURE.md`）来理解现状。
    
- **在 Vibe Coding 中：**
    - 普通模式：AI 只看你贴给它的那 50 行代码。
    - 感知模式：AI 通过 Trae 的全库索引，知道你的 Service 是怎么定义的，知道你的 `references/` 里有哪些 API 约束。
        
- **个人感悟：** Harness这个词是马具，这一步就是给你的马拴上马具（约束）。让它知道该怎么走
  
  2. 规划 (Planning) 

- **概念：** 在动手写第一行代码前，先根据“意图”拆解步骤。
    
- **在 Vibe Coding 中：**
    
    - 这就是你在 `PLANS.md` 或 `exec-plans/active/` 里让 AI 先输出的那份 **Step-by-Step**。（Trae也有规划模式）
        
    - 比如：“第一步修改 Model 增加字段，第二步更新 Service 逻辑，第三步补全单元测试。”
        
- **个人感悟：** 规划完了代码，agent就开始执行了，这是最后一道控制的地方了，比如说一匹马跑在路上我要怎么跑，如果错了的话那就走错路了。
    

 3. 执行 (Execution) 

- **概念：** AI 真正开始修改文件，将规划中的抽象文字变成代码。
    
- **在 Vibe Coding 中：**
    
    - 这就是 Trae 的 **SOLO 模式** 的时候。它不再是一个文件一个文件地改，而是基于计划，**跨文件联动修改**。这里也可以是调用工具比如tools，skills什么的
        
- **个人感悟：** **执行应该是枯燥且自动化的。** 如果这一步还需要人类手动 复制粘贴，那就是工程化的失败。这点很重要如果马跑偏了是不是我们要反思路径错了？改路径。工具错了改工具
    

4. 验证 (Verification)

- **概念：** 检查执行结果是否符合预期，如果不符合，立即修正。
    
- **在 Vibe Coding 中：**
    
    - AI 运行控制台里的测试命令，观察报错信息。
        
    - **关键点：** 它根据 `RELIABILITY.md` （错误处理、重试、日志记录的硬性要求）里的标准进行自查。如果报错了，它会自动回到“感知”阶段（感知错误原因），重新“规划”修复路径，再次“执行”。
        
- **个人感悟** 验证自己是否跑偏的，数据回环，重复规划并且再次执行

### 6.简单的多智能体协作

**别让“球员”兼任“裁判”**
核心分工，不要让写代码的agent和代码审查使用同一个角色。分工明确
最简单的是就是Trae开启多个智能体。或者codex中开启不同的子agent。等等
- **Trae 模式：** 你可以开启多个 Solo 任务，或者在不同窗口给 AI 设定不同的身份。或者设定不同的agent角色
    
- **Codex或Cursor 模式：** 先让一个 Agent 生成代码，然后换一个全新的 Session，把代码和 `RELIABILITY.md` 丢给另一个 Agent 说：**你是最严苛的代码审计员**

**核心痛点：** 为什么我们自己写代码容易出 Bug？因为人类大脑在写逻辑时会产生“盲点”。AI 也一样。如果同一个 Agent 既负责写代码，又负责检查代码，它会陷入“逻辑自洽”的幻觉——它觉得自己的逻辑完美无缺。
**角色分工的精细化与权责边界明确**

不仅仅局限于我们的开发单一角色，而是拆分为更专业的角色，每个角色仅拥有对应的权限，仅读取对应领域的文档，避免角色混乱：
- 产品智能体：负责把需求转化为验收标准，校验交付结果是否符合业务预期；
- 架构智能体：负责方案规划，校验架构合规性；
- 开发智能体：负责代码实现；
- 测试智能体：负责单测、集成测试、场景验证；
- 安全智能体：负责安全合规校验，敏感逻辑审计；
- 运维智能体：负责部署、可观测性配置、线上问题排查。
个人感悟：agent是通过文档握手的
写代码的看 `product-specs/`（需求）
查代码的看 `QUALITY_SCORE.md`（标准）
具体的md可以从github上抄。然后根据自己的需求改进

按当前项目的 product spec 和 active exec plan 启动多 agent 协作：
- 一个 agent 提炼需求
- 一个 agent 提炼计划
- 一个 agent 实现
- 一个 agent 验证
- 验证通过后整理 PR
不要偏离文档。
### 7. 开始 Vibe Coding （MPV）
刚刚上面说了一堆东西，看见那么多的md文档不清楚是干嘛用的，其实我们可以从最小可行性开始
#### 一.最小可行性建立规则
- 1. 写一个.traerules 规则
- 2.创建ARCHITECTURE.md （架构）
- 3.创建product-specs/（需求）
- 4. 创建exec-plans/ （规划）
剩下的，等 AI 犯错的时候，我们再往里填。
- **AGENTS.md**（<150 行，仅索引）
- **ARCHITECTURE.md** + 配套 Lint
- **docs/** 结构化知识库
- **Agent 工具权限**（git / 运行 / 测试 / 日志）
- **自动清理任务**（垃圾回收）
- **人类不写代码**
  
  Agent 必须具备权限：

- Git 操作、PR 创建 / 合并
- CI/CD 执行
- 日志查询（LogQL）
- 浏览器自动化（DevTools）
- 本地脚本调用
  以上再trae里都是有的
#### 二.从“对话式”转向“文档式”纠错
- **普通方式（无效迭代）：**
    
    - 人：“你这个接口没加鉴权，重写！”
        
    - AI：“对不起，加了。”（下次换个接口，它又忘了）。
        
- **Vibe Coding 方式（工程化积累）：**
    
    - 人：**不骂 AI**。直接打开 `SECURITY.md`（数据脱敏、接口鉴权等敏感规范），增加一行：`“所有 /api/v1 下的接口必须引用 authxxxx”`。
        
    - 人：对 AI 说：“重新读取安全文档，修复你的计划。”
        
    - **感悟：** 你改的是文档，沉淀的是资产。你这次花的 1 分钟，保住了以后 100 次不再犯错。
#### 三. Agent 自愈 (Self-Healing Loop)

虽然trae里面有solo模式。也有让agent跑起来验证。但是有一点我觉得不好的地方就是trae里面没有记录错误的东西。为什么错误的话下次可能还会再犯错
准备好执行环境的，叫agent写好日志，输出后让agent去翻references
当agent自写代码修复好代码而不是通过对话引导。那么harness实现了闭环
重点**自愈边界**：
针对航路的话业务对资金、汇率极度敏感
如果涉及到这部分的话千万别让ai自己去改。一定要标注好人工检查！！！！！！！非常重要

### 8.长周期任务治理

基于我个人使用的是qwen3.5和deepseek模型。这俩回答的话经济，但是遇到的问题是一个任务没有做拆解的话上下文长度直接拉爆了。导致出现降智的情况。harness的话我们如何解决这个问题呢。如果从成本上考虑价格低的模型是否能完成我我们的开发任务呢？？


如果你有一个很长的需求的话，别一次性处理完成，而是分阶段去做，因为上下文有限的话ai会发散如果太长的话甚至不能回滚
一般我们使用ULW（循环推进），**“切香肠开发法”**
把大的需求切换成更小块小块的验收小结果，每段都有明确目标，证据，存档点能回滚回来
我在使用trae的时候经常遇到这个傻子问题。如何解决的？
比如约定
- 每一轮盯着一个改动点，动手前先说清楚化验证标准
- 每轮结束必须提交证据：改动点 +diff+通过验证的日志
- 最多尝试多少轮。超过了停下来。
- 总结做了什么，结论是什么，把结论放到下一轮，不要搬动大日志（太占用上下文了）
这么做完之后确实好点了。
如果 AI 报错三次还在原地踏步，说明你的 `RELIABILITY.md`（错误处理、重试、日志记录的硬性要求） 或 `.traerules` 没写清楚
定时清空对话。

### 10. 大型历史项目如何VibeCoding
- 1. 善用LLMs.txt
历史的内容比如我写某个mq调用或者是crud的话写好当前的示范例子。当然可以丢给ai去做，压缩一下规范然后写到references/这个目录里面供外部引用具体的话可以让md知道引用references/xxxx.txt，这样减少了上下文。
 - 2.微服务的话项目模块化。每个微服务项目单独的一套规则。
 - 3.祖传代码定义觉对禁区，你可千万别动。就算是错了你也别动让我手动改。
 - 4.新功能补齐测试单元测试。
 - 5. 团队小组共享文档。需求的话写完归档doc里面的话有部分是可以共用的，个人需求写的话规划完之后就归档。通过git去追溯就好了。
- 6.按业务域拆分**知识卡片库**：针对核心业务域（如票价计算、退改规则、订单支付），在`docs/business-domains/`下生成单独的标准，核心业务规则、上下游依赖、关联核心表、历史坑点、绝对不能改的逻辑、可修改范围、代码入口。AI 处理对应业务需求时，仅加载对应域，无需全量扫描代码。
 -  7. 老代码的话我个人还是建议先不偿还技术债，避免AI模仿老代码一些坑，及时制定规范，老坑做标记，但是千万别让ai去进行模仿，这样会加快技术腐烂。如新技术的话还是建议单独的服务然后做好harness约束。
