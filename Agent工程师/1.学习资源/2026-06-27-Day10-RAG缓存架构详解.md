# RAG 缓存架构详解

> Re-ranking 文稿提到的「缓存」不是简单的 key-value，而是一套多级缓存架构。

---

## 一、缓存在 RAG 中的位置

```
用户查询 ──→ [查询改写]
                │
                ▼
           [  缓存层  ]  ←── 命中 → 直接返回缓存结果（零延迟）
                │ 未命中
                ▼
           [ Embedding 粗筛 ]  →  Top-50
                │
                ▼
           [ Cross-encoder 精排 ]  →  Top-5
                │
                ▼
           [ LLM 生成答案 ]
                │
                ▼
           [ 写回缓存 ]  ←── 下次同样的查询快速命中
```

---

## 二、四层缓存架构

### 第一层：查询结果缓存（最关键）

**缓存什么**：完整的检索结果（Top-5 文档块 + LLM 生成答案）

**Key 设计**：
```
cache_key = f"rag:query:{hash(query)}:{hash(semantic_normalized)}"
```
- 先对查询做语义归一化（同义词归一、简繁转换、去除停用词）
- 再取 hash 作为 key
- 避免「重置密码」「怎么重置密码」「重置密码怎么操作」各占一条

**Value 设计**：
```json
{
  "answer": "重置密码的步骤是...",
  "sources": [
    {"doc_id": "doc_123", "chunk": "重置密码需要...", "score": 0.95},
    {"doc_id": "doc_456", "chunk": "密码找回流程...", "score": 0.87}
  ],
  "created_at": "2026-06-27T10:00:00Z",
  "ttl": 3600
}
```

**TTL 策略**：
| 知识库类型 | TTL | 理由 |
|-----------|-----|------|
| 静态文档（产品文档、FAQ） | 24h-7d | 几乎不变 |
| 半动态（周更新知识库） | 1h-6h | 平衡实时性和缓存命中 |
| 动态（实时新闻、聊天记录） | 5-30min | 需要较新数据 |

---

### 第二层：Embedding 向量缓存

**缓存什么**：问题的 Embedding 向量（避免重复调用 Embedding 模型）

```
cache_key = f"rag:embedding:{hash(query)}"
value = [0.123, -0.456, ...]  # 768维向量
```

**为什么需要这一层**：Embedding 模型虽然快（~10ms），但如果每秒 1000 次查询，10ms × 1000 = 10 秒 CPU 开销，加一层缓存可以省掉 60% 的重复编码。

---

### 第三层：Cross-encoder 分数缓存

**缓存什么**：(query, doc) 配对的 score

```
cache_key = f"rag:rerank:{hash(query)}:{hash(doc_chunk)}"
value = 0.92  # Cross-encoder 输出的分数
```

**为什么需要这一层**：同一个查询可能出现在多个会话中，同一个文档块可能被多次配对。缓存了配对分数之后，第二次命中就不用跑 Cross-encoder 了。

**配合预置缓存**：可以离线跑一批高频 (query, doc) 对的分数预热缓存。

---

### 第四层：LLM 生成缓存（选做）

**缓存什么**：相同的 query + context 生成的完整答案

**风险**：LLM 生成可能有微小差异，适合确定性场景（如 FAQ 类的固定答案）

---

## 三、Key 设计的核心问题：怎么判断「同一个查询」？

直接比较字符串是不够的，因为用户有很多变体：

| 用户实际输入 | 归一化后 |
|-------------|---------|
| 「重置密码」 | 重置密码 |
| 「怎么重置密码啊」 | 重置密码 |
| 「如何重置密码」 | 重置密码 |
| 「reset password」 | 重置密码 |
| 「密码怎么改」 | 重置密码 |

**做法**：
1. **语义归一化**：先用轻量级 LLM 或同义词表做归一化
2. **MinHash/LSH 去重**：对相似但不完全一样的查询做模糊匹配
3. **冷却期缓存**：5分钟内完全相同的查询直接命中，5分钟外的做语义归一再查

## 四、具体代码骨架

```python
import hashlib
import json
import time
from redis import Redis

redis = Redis(host="localhost", port=6379, db=0)

class RAGCache:
    def __init__(self):
        self.default_ttl = 3600  # 1小时

    def _semantic_normalize(self, query: str) -> str:
        """语义归一化：同义词替换 + 停用词过滤 + 简繁转换"""
        # 实际接入 NLU 模型或同义词表
        normalized = query \
            .replace("怎么", "") \
            .replace("如何", "") \
            .replace("啊", "") \
            .replace("呀", "") \
            .strip()
        return normalized

    def _make_cache_key(self, query: str) -> str:
        normalized = self._semantic_normalize(query)
        return f"rag:query:{hashlib.md5(normalized.encode()).hexdigest()}"

    def get(self, query: str) -> dict | None:
        """从缓存获取"""
        key = self._make_cache_key(query)
        cached = redis.get(key)
        if cached:
            console.log(f"[缓存命中] {query}")
            return json.loads(cached)
        return None

    def set(self, query: str, result: dict, ttl: int = None):
        """写入缓存"""
        key = self._make_cache_key(query)
        redis.setex(
            key,
            ttl or self.default_ttl,
            json.dumps(result)
        )

    def invalidate(self, doc_ids: list[str]):
        """知识库更新时，清除涉及该文档的缓存"""
        # 实现方式1：遍历缓存键（不适合大量缓存）
        # 实现方式2：用 Redis 的 tag 模式
        for doc_id in doc_ids:
            redis.sadd("rag:invalidated_docs", doc_id)
```

## 五、分级检索 × 缓存联动（文稿的完整架构）

```
                  ┌─────────────────────────────────────┐
                  │          用户查询                      │
                  └─────────────────────────────────────┘
                              │
                              ▼
                  ┌─────────────────────────────────────┐
                  │  查询改写 + 语义归一化                  │
                  └─────────────────────────────────────┘
                              │
                              ▼
                  ┌─────────────────────────────────────┐
                  │       缓存查询（第一层）                │
                  └─────────────────────────────────────┘
                     ↓ 命中              ↓ 未命中
                  ┌──────────┐    ┌─────────────────────┐
                  │ 直接返回  │    │  分类器判定查询等级    │
                  │ 零延迟    │    └─────────────────────┘
                  └──────────┘     ↓          ↓          ↓
                                🔵常见   🟡中等    🔴复杂
                                   │          │          │
                                   ▼          ▼          ▼
                             ┌────────┐ ┌────────┐ ┌──────────────┐
                             │ 缓存   │ │仅粗筛  │ │ 粗筛+精排    │
                             │ 兜底   │ │(无     │ │ 完整链路      │
                             │ 返回   │ │Rerank) │ │              │
                             └────────┘ └────────┘ └──────────────┘
                                                   │
                                                   ▼
                                             ┌──────────────┐
                                             │ 写回缓存      │
                                             └──────────────┘
```

**各等级的实际表现**：

| 等级 | 占比 | 路径 | 延迟 | 缓存策略 |
|------|------|------|------|---------|
| 🔵 常见（FAQ） | ~60% | 直接命中缓存 | <1ms | TTL 长（24h） |
| 🟡 中等 | ~20-30% | Embedding 粗筛 | ~50ms | 写入缓存（TTL 1h） |
| 🔴 复杂 | ~10% | 完整链路 | ~300-500ms | 写入缓存（TTL 1h） |

## 六、缓存失效策略（难点）

RAG 缓存比 HTTP 缓存更难的地方：**知识库是动态的**。

### 方案 A：TTL 过期（最简单）
- 每个缓存设过期时间
- 适合更新频率稳定的知识库
- ⚠️ 问题：TTL 内知识库更新了，用户拿到的还是旧结果

### 方案 B：主动失效（变更感知）
```python
def on_knowledge_updated(updated_doc_ids: list[str]):
    # 清除涉及这些文档的所有缓存
    for key in redis.scan_iter("rag:query:*"):
        cache_entry = json.loads(redis.get(key))
        cache_sources = [s["doc_id"] for s in cache_entry["sources"]]
        if any(doc_id in updated_doc_ids for doc_id in cache_sources):
            redis.delete(key)
```

### 方案 C：Cache-Aside + 写穿透
- 知识库更新时，同时标记相关查询的缓存为「陈旧」
- 下次命中时，后台异步刷新（Stale-While-Revalidate 模式）

```
用户查询 → 命中陈旧缓存
          ├─ 立即返回旧结果（老数据也比空结果好）
          └─ 后台异步重新检索 + 更新缓存
```

## 七、面试回答模板

**面试官问**：「RAG 系统里缓存怎么设计？」

**回答框架**：
1. **四层缓存**：查询结果 → Embedding 向量 → Cross-encoder 分数 → LLM 生成（按重要性排列）
2. **Key 设计**：不做简单的字符串匹配，先做语义归一化（同义词替换、去停用词）再 hash
3. **TTL 策略**：静态库 24h，动态库 5-30min，配合主动失效
4. **分级联动**：60% 常见查询直接走缓存零延迟，20-30% 仅粗筛，10% 才走完整链路
5. **陈旧缓存处理**：Stale-While-Revalidate，先返回旧数据再后台刷新
