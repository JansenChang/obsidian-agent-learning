# Day 11 — Agentic RAG 与 GraphRAG

## 🎯 核心概念

| 概念 | 一句话 | 等级 |
|------|--------|:---:|
| **Agentic RAG** | LLM 自主决定多轮检索，不是检一次就完 | 🟠 |
| **Corrective RAG** | 检完后让 LLM 评估结果质量，不够好就改写重搜 | 🟠 |
| **Self-RAG** | 每条 chunk 自评「是否需要我」「我够不够回答」再决定是否纳入 | 🟡 |
| **HyDE** | 先根据问题生成一篇假想答案，用这篇答案去搜 | 🟡 |
| **Multi-Hop** | 第一次检索的结果作为线索，顺着线索再搜，层层深入 | 🟠 |
| **GraphRAG** | 从文档中提取实体和关系建知识图谱，按图结构检索 | 🟠 |
| **实体抽取** | 从文本中识别出人、物、地点、概念等实体 | 🟡 |
| **关系抽取** | 识别实体之间的语义关系（A 属于 B、A 包含 B） | 🟡 |

---

## 🧱 第一部分：Agentic RAG — 从一次性检索到对话式检索

### 传统 RAG 的硬伤

```
用户："君心似我心讲什么"
传统 RAG：全库搜 chunk → 拼碎片 → 拼凑回答
Agentic RAG：识别意图 → 搜标题定位文档 → 取完整文档 → 概述
```

传统 RAG 执行单次搜索。找不到就是找不到，不会尝试换一种方式。用户查一个产品价格，系统扫向量库，没搜到直接说没有。但人查价格会先翻产品列表，定位到产品页，再找价格栏目。

Agentic RAG 允许模型在搜索和生成之间反复，像人一样逐步缩小范围。

### 核心循环

```
用户问题
    ↓
Agent 判断：「我需要检索吗？」
    ↓ 是
执行检索策略（查询改写/多路搜索/定向搜索）
    ↓
拿到结果 → Agent 评估：「这些结果够回答吗？」
    ↓ 不够                            ↓ 够了
改写查询，再搜一次                   LLM 生成答案
    ↓ 仍不够
再改再搜（最多 N 次）
    ↓ 达上限
返回已有结果 + 注明局限性
```

Agentic RAG 依赖三个关键组件：
- **Judgment（判断）**：当前信息够不够？
- **Tool Selection（选工具）**：下一步搜哪里？BM25、向量库、文档标题索引、Web 搜索？
- **Memory（记忆）**：已经搜过了什么？避免重复搜同一个东西。

### 🟠 Corrective RAG

```
用户问题 → 检索 → LLM 评估检索结果质量
                      ↓ 好                      ↓ 差（缺失/不相关）
                用结果生成                改写查询，重新检索
                                              ↓
                                        再次评估
```

Corrective RAG 的关键是一步退路：让 LLM 对自己搜到的东西做一个质量判断。如果得分低，不硬着头皮生成。

**实现要点**：
- 评估标准：检索结果是否包含核心实体？是否与问题领域一致？
- 改写策略：补全缺失关键词、换同义词、改问法
- 最大重试次数：通常 1-2 次，多了延迟失控

### 🟡 Self-RAG

Self-RAG 精度更高。每条 chunk 生成两个辅助 token：
- **isRel（是否相关）**：这个 chunk 和问题有关吗？
- **isSup（是否支撑）**：这个 chunk 能支撑答案吗？

```
检索到 10 条 chunk
    ↓
第 1 条：isRel=否 → 丢弃
第 2 条：isRel=是 → isSup=否 → 保留但注明「不完全支撑」
第 3 条：isRel=是 → isSup=是 → 高优先级纳入
    ↓
用精选后的 chunk 生成答案
```

Self-RAG 开销大——每条 chunk 都要过 LLM 做自评。适合对精度要求极高的场景（医疗、法律）。

### 🟡 HyDE（Hypothetical Document Embeddings）

```
用户：「异步编程怎么优化？」
    ↓
LLM 生成一篇假想文档：
「异步编程可以通过以下方式优化：减少 await 次数、合并协程、使用 asyncio.run...」
    ↓
用这篇假想文档去向量库检索
    ↓
搜到真实文档 → 用真实文档回答
```

HyDE 的思路：用户的问题太短，信息量不够。先让 LLM 按照问题生成一段「答案风格」的文本，再拿这段更长的文本去搜，命中率更高。

**适用场景**：用户查询很短（2-5 个字）、用户自己说不清楚、领域术语和日常用语不匹配。

### 🟠 Multi-Hop RAG

```
Q1：「2024 年诺贝尔化学奖得主是谁？」
  → 搜到获奖者：David Baker、Demis Hassabis、John Jumper
Q2：「Demis Hassabis 是哪家公司的创始人？」
  → 搜到：DeepMind
Q3：「DeepMind 的 AlphaFold 解决了什么问题？」
  → 搜到：蛋白质结构预测
最终回答：「...」 
```

Multi-Hop 的核心是**信息链**。第一次检索的结果里包含了一个实体，这个实体又是下一次检索的起点。像解谜游戏，每解开一局拿到下一条线索。

**实现前提**：LLM 需要有能力从检索结果中识别「线索实体」，并据此构造下一轮查询。

## 🧱 第二部分：GraphRAG — 从文档检索到知识图谱检索

### 传统 RAG 解决不了的场景

```
问题：「华为和腾讯在云计算领域的合作有哪些？」
传统 RAG：搜"华为 腾讯 云计算 合作"
          结果：找到了"华为云"的某段 + "腾讯云"的某段
          但没有哪篇文章同时提到两者的合作 → 答案拼不完整
          
GraphRAG：知识图谱里有「华为—合作→腾讯」「华为—主营→云计算」「腾讯—主营→云计算」
          检索时沿着实体关系走，找到两者的交叉点
```

GraphRAG 把文档中的实体和关系抽取出来，建成图结构。检索时不仅搜文档语义，还搜实体之间的关系网。

### GraphRAG 的核心流程

```
┌─────────────────────────────┐
│  文档集                        │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│  实体抽取                     │
│  LLM 从每篇文档中提取：         │
│  人物、公司、产品、概念、地点...  │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│  关系抽取                     │
│  实体之间的关系：               │
│  A→母公司→B                   │
│  C→竞争对手→D                 │
│  E→投资→F                    │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│  构建图索引                    │
│  实体 = 节点，关系 = 边          │
│  存储到图数据库（Neo4j/ArangoDB）│
│  同时为实体生成 Embedding        │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│  检索流程                      │
│  1. 用户问题 → 识别实体         │
│  2. 在图中定位实体节点          │
│  3. 沿关系扩散（1-3跳）         │
│  4. 聚合子图内容                │
│  5. LLM 基于子图生成答案        │
└─────────────────────────────┘
```

### 微软 GraphRAG 方案（主要实现）

微软 2024 年开源的 GraphRAG 是业界最完整的实现：

1. **文档分组**：把文档分社区，社区内实体关系更密集
2. **社区摘要**：每个社区生成一段摘要，描述这个社区里的核心实体和关系
3. **分层索引**：底层是原始实体关系，上层是社区摘要
4. **检索优化**：根据问题粒度选择检索层——全局问题搜社区摘要，局部问题搜实体关系

**优势**：
- 能回答跨文档的全局问题（「我们的产品在哪些方面受到好评？」）
- 能回答统计类问题（「哪个模块的 Bug 最多？」）
- 能回答对比类问题（「A 和 B 的定位有什么区别？」）

**劣势**：
- 建图成本高（GPT-4 处理 100 万 token 约 $100）
- 实体抽取质量影响下游检索
- 不适合高频更新的知识库（每次更新都要重新抽取实体）

## ⚡ 工程实践关键问题

### 1. 什么时候停止循环？

| 策略 | 做法 | 优缺点 |
|------|------|--------|
| **固定次数** | 最多搜 2-3 次 | 简单，但可能不够或浪费 |
| **置信度阈值** | 评估分数 > 0.8 就停止 | 灵活，但需要调阈值 |
| **边际收益** | 新结果不带来新信息就停 | 最合理，但实现复杂 |

### 2. 延迟怎么控制？

Agentic RAG 延迟 = 检索次数 × (检索 + 评估 + 改写) 时间

**优化方向**：
- 第一次检索走缓存（很多常见问题一次就够了）
- 检索之间共享 Context（不重复传输已知信息）
- 并行搜索（如果 Multi-Hop 不依赖上一步结果）

### 3. GraphRAG 的建图频率？

| 知识库类型 | 建图频率 | 理由 |
|-----------|---------|------|
| 静态文档 | 一次性建图 | 几乎不变 |
| 周更新 | 每周增量重建 | 只处理新文档的实体 |
| 实时数据 | 不适用 | 建图成本太高 |

## ❓ 面试高频考点

1. **Agentic RAG 和传统 RAG 的核心区别？** → 单次 vs 多轮，被动 vs 主动，检完就生成 vs 检完自我评估
2. **Corrective RAG 怎么知道检得好不好？** → LLM 自己打分（自评），或专门训练一个评估模型
3. **Self-RAG 的两个辅助 token 是什么？** → isRel（是否相关）和 isSup（是否支撑）
4. **Multi-Hop 的实现前提？** → LLM 能识别线索实体并构造下一轮查询
5. **GraphRAG 解决了什么传统 RAG 解决不了的问题？** → 跨文档关联、对比、全局统计类问题
6. **GraphRAG 和知识图谱有什么区别？** → GraphRAG 的实体从文档自动抽取，传统知识图谱需要人工构建
7. **Agentic RAG 延迟怎么优化？** → 缓存 + 并行检索 + 收敛判断 + 分级策略

| Relation Extraction | 关系抽取 | 识别实体间的语义关系 |
