# Day 9 — Chunking 策略与 Hybrid Search

## 🎯 核心概念

| 概念 | 一句话 | 等级 |
|------|--------|:---:|
| **Chunking（分块）** | 将长文档切分为适合检索的段落单元 | 🟡 |
| **固定长度分块** | 按固定 token 数切分，最简单但可能截断语义 | 🟡 |
| **递归分块** | 按分隔符优先级逐级切分（段落→句子→词），工程最优折中 | 🟡 |
| **语义分块** | 检测相邻句子的 Embedding 相似度断崖作为切点 | 🟠 |
| **Overlap（块重叠）** | 相邻块之间保留重复内容，防止边界信息丢失 | 🟡 |
| **BM25**（Best Match 25） | 基于词频和逆文档频率的关键词检索算法 | 🟡 |
| **Sparse Retrieval（稀疏检索）** | 基于关键词匹配的检索，BM25 是代表 | 🟡 |
| **Dense Retrieval（稠密检索）** | 基于 Embedding 向量的语义检索 | 🟡 |
| **Hybrid Search（混合搜索）** | 稀疏 + 稠密检索结果融合，互补盲区 | 🟡 |
| **RRF**（Reciprocal Rank Fusion） | 混合搜索中融合多路排序结果的算法 | 🟠 |

---

## 🧱 为什么 Chunking 是 RAG 的基石

RAG 三段式 Pipeline 中，索引层的第一步就是 Chunking。它直接决定后续检索质量的上限。

**块太大的代价：**
- 噪声多 —— 用户问一句话，返回 2000 字的块，只有 10% 相关
- LLM 注意力稀释 —— 信息淹没在无关内容中

**块太小的代价：**
- 上下文断裂 —— 「它解决了这个问题」——「它」指什么？块里没有上文
- 检索碎片化 —— 一个完整答案分散在 5 个块里，但只搜到其中 2 个

> **核心权衡：语义完整性 vs 检索精度。512~1024 token 是常见起点。**

---

## ✂️ 三种 Chunking 策略

### 策略 1：固定长度分块（Fixed-size Chunking）

最简单粗暴的方式——每 N 个字符/token 切一刀。

```
原文: "RAG 是检索增强生成。它解决了 LLM 的知识截止日期问题。
      具体做法是先从外部知识库检索相关文档..."

固定长度（每 20 字）:
  块1: "RAG 是检索增强生成。它解决了 LLM"     ← "LLM" 后面被截断
  块2: "的知识截止日期问题。具体做法是先从外部" ← 缺少主语
```

**优点：** 实现简单，计算量小
**缺点：** 无视语义边界，经常把一句话拦腰斩断

**适用场景：** 快速原型、对精度要求不高的场景

---

### 策略 2：递归分块（Recursive Character Text Splitting）

LangChain 的默认策略。按分隔符优先级逐级尝试：

```
优先级: "\n\n"（段落） → "\n"（换行） → "。"（句子） → "，"（从句） → ""（字符）

走查:
  文档 → 先按段落切
        → 某段超过 chunk_size？
           → 降级按句号切
              → 某句还超过？
                 → 降级按逗号切
```

**优点：** 尽量保持语义完整 + 保证每块不超过上限（工程最优折中）
**缺点：** 纯基于分隔符，不理解语义

**面试要点：** 这是 LangChain `RecursiveCharacterTextSplitter` 的实现原理，面试常考。

---

### 策略 3：语义分块（Semantic Chunking）

让 Embedding 决定在哪切——检测「语义断崖」。

```
逐句做 Embedding，计算相邻句子的余弦相似度:

  ① "RAG 是检索增强生成。"        ] 
  ② "它解决了知识截止日期问题。"    ] 相似度 0.91 ← 高，不切
  ③ "先从外部知识库检索文档。"      ] 相似度 0.88 ← 高，不切
  ───────── 相似度骤降至 0.23 ───────── ← 语义断崖！切！
  ④ "Jansen 最近在学 Agent 面试。"  ]
  ⑤ "他每天通勤时听预习音频。"      ] 相似度 0.85 ← 高，不切

结果: 块1 = ①②③（RAG 话题） / 块2 = ④⑤（学习话题）
```

**优点：** 语义最完整，不会切断逻辑链
**缺点：** 每句话做 Embedding → 索引速度慢，成本高

**适用场景：** 精度要求极致的场景，先用递归分块跑通后再考虑升级

---

### 三种策略对比

| 维度 | 固定长度 | 递归分块 | 语义分块 |
|------|:---:|:---:|:---:|
| 实现复杂度 | ⭐ | ⭐⭐ | ⭐⭐⭐ |
| 语义完整性 | 差 | 中 | 优 |
| 索引速度 | 快 | 快 | 慢 |
| 推荐场景 | 快速原型 | **默认首选** | 高精度场景 |

## 🔗 Overlap（块重叠）— 防止边界信息丢失

即使切分策略再聪明，边界处总有信息会被切断。Overlap 是最后的保护网。

```
没有 Overlap:
  块1: "...模型的上下文窗口从 128K 扩展到"  ← 扩展到多少？丢了！
  块2: "1M token，这让它能处理更长的文档。" ← 1M token！但没有上文

有 Overlap（20% 重叠）:
  块1: "...模型的上下文窗口从 128K 扩展到 1M token，这让它能"
  块2: "扩展到 1M token，这让它能处理更长的文档。具体来说..."
       ↑ 重叠部分确保衔接
```

**典型值：chunk_size 的 10-20%。** 比如 chunk_size=500, overlap=50~100。

**面试金句：** 「Overlap 不是越多越好——多了浪费存储和检索带宽。10-20% 是工程实践中的 sweet spot。」

## 🔀 Hybrid Search（混合搜索）

### 为什么单一语义搜索有盲区

Embedding 检索很强，但不是万能的：

| 场景 | Embedding 的表现 | 原因 |
|------|:---:|------|
| 「Python 是什么」 | ✅ 好 | 语义清晰 |
| 「怎么优化 Python 性能」 | ✅ 好 | 语义清晰 |
| 「Jansen」这个名字 | ❌ 差 | 专有名词，训练语料里几乎没出现过 |
| 「API-2024-Q3-v2.doc」 | ❌ 差 | 文件名/编号，不是自然语言 |
| 「错误码 E10023」 | ❌ 差 | 精确匹配需求，语义帮不上忙 |

**这就是 BM25 登场的地方。**

---

### BM25（Best Match 25）— 关键词检索

基于「词频 × 逆文档频率」的经典算法。核心逻辑：

- **词频（TF）**：这个词在文档中出现多少次？越多越相关——但有**饱和效应**：出现 5 次和出现 50 次差不多
- **逆文档频率（IDF）**：这个词在整个知识库里稀不稀有？越稀有越重要

```
用户问: "Jansen 的 Agent 项目"

BM25 搜索:
  "Jansen" → IDF 很高（专有名词，罕见）→ 权重极高
  "Agent" → IDF 中等（常见词）
  "项目" → IDF 低（非常常见）→ 权重低

→ BM25 知道 "Jansen" 是最关键的搜索词
→ 而 Embedding 可能根本不认识 "Jansen" 这个词
```

---

### 混合搜索 = BM25 + Embedding

| 维度 | BM25（稀疏检索） | Embedding（稠密检索） |
|------|-----------------|---------------------|
| 匹配方式 | 关键词精确匹配 | 语义相似度 |
| 擅长 | 专有名词、编号、精确查询 | 同义改写、模糊语义、长文本 |
| 盲区 | 「Python」搜不到「编程语言」 | 「E10023」搜不到精确的错误码 |
| 速度 | 极快（倒排索引） | 快（向量索引） |
| 代表性工具 | Elasticsearch | Milvus/Qdrant/Pinecone |

**两者互补：** BM25 抓精确匹配，Embedding 抓语义理解。合在一起覆盖所有检索场景。

---

### RRF（Reciprocal Rank Fusion）— 结果融合

两路检索各返回一个排序列表，怎么合并？最简单的就是 RRF：

```
BM25 排序:          Embedding 排序:
  1. 文档A             1. 文档C
  2. 文档B             2. 文档A
  3. 文档D             3. 文档B

RRF 融合公式（简化理解）:
  每个文档的最终得分 = 1/(BM25排名+k) + 1/(Embedding排名+k)

  文档A: 1/(1+60) + 1/(2+60) = 0.0164 + 0.0161 = 0.0325  ← 最高
  文档C: 1/(100+60) + 1/(1+60) = 0.0062 + 0.0164 = 0.0226
  文档B: 1/(2+60) + 1/(3+60) = 0.0161 + 0.0159 = 0.0320

最终排序: 文档A → 文档B → 文档C
```

> **RRF 的优点：** 不需要归一化两路分数（BM25 和余弦相似度的数值范围完全不同），只看排名，简单有效。

## 🔗 知识连接

### Chunking × Tokenization（Day 3）

| 维度 | Tokenization | Chunking |
|------|-------------|----------|
| 切分对象 | 文本 → token | 文档 → 段落块 |
| 粒度 | 字符/子词级 | 段落级 |
| 目的 | 让模型能「读」文本 | 让检索能「定位」信息 |
| 共同点 | 都是文本预处理的第一步，错误的切分会导致后续全链路崩塌 | |

### Overlap × Lost in the Middle

| 维度 | Overlap | Lost in the Middle |
|------|---------|-------------------|
| 解决的问题 | 块边界处信息被切断 | 长上下文中间位置被忽略 |
| 机制 | 物理上让相邻块包含重复内容 | LLM 注意力分布的固有缺陷 |
| 层级 | 索引层防御 | 生成层缺陷 |

### BM25 × Embedding

不是替代关系，是互补关系。就像「查字典」和「问老师」——一个按字母顺序精确查找，一个理解你的意思帮你找。

## ❓ 面试三问

1. **固定分块、递归分块、语义分块各有什么优缺点？什么时候用哪个？**
   → 固定 = 原型阶段，快但差；递归 = 默认首选，平衡最好；语义 = 精度极致，但贵

2. **为什么需要 Hybrid Search？单一 Embedding 检索有什么盲区？**
   → Embedding 对专有名词、编号、精确查询无能为力。BM25 互补。用户可能用完全不同的词问同一个问题（语义搜索强），也可能用精确的关键词搜（BM25 强）

3. **Overlap 设多少合适？太大或太小有什么问题？**
   → 10-20% 是 sweet spot。太小 → 边界信息丢失；太大 → 存储翻倍 + 检索噪声
