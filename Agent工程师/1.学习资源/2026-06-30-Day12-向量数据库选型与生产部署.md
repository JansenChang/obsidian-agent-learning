# Day 12 — 向量数据库选型与生产部署

## 🎯 核心概念

| 概念 | 一句话 | 等级 |
|------|--------|:---:|
| **ANN** | 近似最近邻搜索，向量数据库的底层技术 | 🟡 |
| **HNSW** | 多层图索引，召回率最高但要吃内存 | 🟡 |
| **IVF** | 按组搜索，省内存但精度略低 | 🟡 |
| **DiskANN** | 把索引放 SSD 上，适合十亿级规模 | 🟠 |
| **Milvus** | 最成熟的独立向量数据库，架构复杂 | 🟡 |
| **Qdrant** | Rust 实现，单机性能好部署简单 | 🟡 |
| **Chroma** | 轻量级，pip install 就能用，只适合原型 | 🟡 |
| **PGVector** | PostgreSQL 扩展，零额外运维成本 | 🟡 |
| **Pinecone** | 纯托管方案，不用自己部署 | 🟡 |
| **先写后建** | 先用 Flat 写入，攒够再建索引 | 🟠 |

---

## 🧱 三种索引

| 索引 | 工作方式 | 内存 | 精度 | 适合 |
|------|---------|------|------|------|
| **HNSW** | 多层图结构，逐层下钻 | 高（大） | 最高（98%+） | 内存够、追求精度 |
| **IVF** | 聚类分组，只搜邻近组 | 低 | 略低 | 内存有限、可容忍精度损失 |
| **DiskANN** | 图结构存 SSD | 很低 | 略低于 HNSW | 十亿级以上 |

---

## 🗄️ 五个数据库对比

| 数据库 | 优点 | 缺点 | 适合谁 |
|--------|------|------|--------|
| **Milvus** | 最成熟、索引全、分布式 | 架构太重 | 有大运维团队 |
| **Qdrant** | Rust 性能好、部署简单 | 生态不如 Milvus | 推荐首选 |
| **Chroma** | 安装即用 | 功能最少、不稳 | 只做原型 |
| **PGVector** | 零运维、事务一致 | 性能不如专用库 | 已用 PostgreSQL |
| **Pinecone** | 托管、零运维、性能好 | 付费、数据在外 | 不想碰服务器 |

---

## ⚡ 生产部署五件事

1. **容量规划**：估算值翻一倍留余量（五百万向量 ≈ 32GB HNSW）
2. **索引构建**：先 Flat 写入，攒够再 optimize （先写后建）
3. **备份策略**：Qdrant 快照 / PGVector pg_dump / Milvus 导出重建
4. **监控指标**：P99 延迟 <100ms、召回率 >90%、索引构建进度
5. **版本升级**: 等新版本发布一个月再升，先测试环境验证

## ❓ 面试高频考点

1. **HNSW 和 IVF 的区别？** → 图结构 vs 分组，内存 vs 精度取舍
2. **什么时候用 DiskANN？** → 十亿级以上，内存不够时
3. **生产环境建索引的推荐做法？** → 先写后建，不要实时建
4. **Qdrant 和 Milvus 怎么选？** → 团队能力决定，推荐 Qdrant 起步
5. **向量数据库的备份怎么做？** → 快照 / pg_dump / SDK 导出

## 🏷️ 术语表

| 术语 | 全称 | 一句话 |
|------|------|--------|
| ANN | Approximate Nearest Neighbor（近似最近邻） | 向量检索的基础技术 |
| HNSW | Hierarchical Navigable Small World（分层可导航小世界图） | 多层图索引，召回率最高 |
| IVF | Inverted File Index（倒排文件索引） | 先聚类再组内搜索，省内存 |
| DiskANN | Disk-based Approximate Nearest Neighbor | 基于 SSD 的图索引，适合十亿级 |
| Quantization | 量化 | 把 Float32 压缩成 Float16/Uint8 省内存 |
| nprobe | — | IVF 搜索时搜的组数，越大越准越慢 |
