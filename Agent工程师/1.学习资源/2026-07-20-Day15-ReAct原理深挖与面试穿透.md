# Day 15 — ReAct 原理深挖与面试穿透

## 🎯 核心概念

| 概念 | 一句话 | 等级 |
|------|--------|:---:|
| **接地（Grounding）** | 工具调用把 LLM 的推理从"漂浮"拉回"地面"，防止幻觉传播 | 🟡 |
| **错误传播** | CoT 的自循环推理——第一步错，后面全错，无外部验证 | 🟡 |
| **ReAct 四组对比实验** | Standard / CoT(Reason Only) / Act Only / ReAct | 🟠 |
| **CoT 幻觉** | LLM 不知道答案时不会说不知道，而是推理出"看起来合理但错误"的答案 | 🟡 |
| **推理-行动互补** | 推理提供方向（地图），行动提供精度（坐标） | 🟡 |
| **ReAct 三种失败模式** | 推理幻觉传播 / 工具沼泽 / 循环发散 | 🟠 |
| **ReAct 提示词四层设计** | 角色定义→推理行动约束→工具使用指南→安全边界 | 🟠 |
| **Agent 评估三层次** | 组件评估 / 轨迹评估 / 端到端评估 | 🟡 |
| **轨迹调试** | Agent 异常时先看轨迹而非代码，逐轮检查 Thought/Action/Observation | 🟠 |
| **Supervisor vs Peer-to-Peer** | Multi-Agent 两种编排模式：中心调度 vs 对等通信 | 🟠 |

---

## 📊 ReAct 四组对比实验

原论文在四个任务上测试四种配置：

| 配置 | 说明 | 优势场景 | 劣势场景 |
|------|------|---------|---------|
| **Standard** | 直接回答，不推理不调工具 | 极简单问答 | 任何需要推理的场景 |
| **CoT (Reason Only)** | 有推理，无工具 | 百科知识类（LLM内部有知识） | 需要实时信息，会产生幻觉 |
| **Act Only** | 有工具，无推理 | 简单查询（天气/股价） | 需要多步规划 |
| **ReAct** | 推理+工具交替 | 开放式多步问题 | 推理开销大 |

**反直觉结论：** 百科知识类任务上 CoT > ReAct。因为问题在 LLM 内部知识范围内，工具调用引入噪音和多一轮出错概率。

---

## 🔄 推理与行动的互补机制

```
CoT 的问题：自循环推理 → 错误传播
  LLM 知识 → 错误事实1 → 基于事实1推理 → 错误事实2 → ...
  无外部验证 ❌

Act Only 的问题：缺乏规划
  调工具A → 拿到结果 → 发现应该先调工具B → 重新来 → 效率低 ❌

ReAct 的解法：
  Thought（规划方向）→ Action（获取事实）→ Observation（验证修正）→ 新 Thought
  推理 = 地图，行动 = 坐标。两者相互锚定。
```

---

## 🚨 ReAct 三种失败模式

| 失败模式 | 表现 | 解法 |
|---------|------|------|
| **推理幻觉传播** | 错误推理→选错工具→不相关信息→推理更偏 | 提前刹车提示："请先检查已有信息是否能回答问题" |
| **工具沼泽** | 已有足够信息仍继续调工具 | 相关性评估，连续调非关键工具时打断 |
| **循环发散** | 越查离原始问题越远 | 每次 Thought 强制回顾原始问题 |

## 🆚 ReAct vs Function Calling

| 维度 | ReAct | Function Calling |
|------|-------|------------------|
| 调用模式 | 多轮循环 | 单轮 |
| 工具选择 | 动态自主决定 | 一次性决定 |
| 适用场景 | 开放式多步推理 | 明确工具映射 |
| 代码实现 | while 循环 + 每轮 LLM 调用 | 单次 LLM 调用 |

**判断标准：** 用户问题 → 解决工具映射是否明确？明确→FC够了；需多步推理→ReAct。

## 📝 ReAct 提示词四层设计

| 层级 | 作用 | 示例 |
|------|------|------|
| 1. 角色定义 | 指导 Thought 方向 | "你是一个智能客服" |
| 2. 推理行动约束 | 控制 Action 频率 | "每次只调用必要的工具，有足够信息后直接回答" |
| 3. 工具使用指南 | 提升 Action 准确性 | "查询订单时使用订单查询工具，失败可重试一次" |
| 4. 安全边界 | 保护 Observation 后行为 | "只能查询，不能修改。退款需引导用户到退款页面" |

**优先级：越往下越高。安全边界 > 一切。**

---

## 📐 Agent 评估三层次

| 层次 | 评估什么 | 回答什么问题 |
|------|---------|-------------|
| **组件评估** | 每个工具单独测试 | 工具是否可靠？ |
| **轨迹评估** | 整个推理-行动链是否合理 | Agent 是否聪明？ |
| **端到端评估** | 最终答案正确性 | Agent 是否有用？ |

**核心洞察：** 推理全错但碰巧结果对的 Agent 和推理全对结果也对的 Agent 在端到端评估里得分相同，但前者不可信赖，后者可复现。生产系统以不可信赖更致命。

## 🐛 故障排查黄金法则

1. **看轨迹** → 拉出最近 100 条完整 ReAct 轨迹，找失败模式
2. **复现** → 用代表性失败轨迹在本地重建
3. **分层修复** → 工具层 > LLM推理层 > 循环控制层
4. **黄金法则** → 永远先修最底层的问题。工具返回格式错误，修工具而非改 Prompt

## 🎤 面试应答模板

### 被问「讲一下 ReAct 的原理」
> ReAct 是 Reasoning + Acting 的缩写。核心思想是让 LLM 在推理过程中穿插行动。模型先思考下一步需要什么信息，然后执行行动调用工具或搜索，获得观察结果后再继续推理，如此循环直到得出结论。与纯 CoT 的区别在于它能够与外部环境交互，用真实信息修正推理，避免幻觉。论文在 HotpotQA 和 Fever 等任务上验证了 ReAct 比纯推理或纯行动方案都有显著提升。

### 被问「ReAct 在实际中怎么用」
> 最典型的应用是 LangChain 中的 ReAct Agent。在 prompt 中定义可用的工具列表，模型按照 Thought-Action-Observation 的格式进行多轮推理。实际部署时需要注意：设计清晰的工具描述、提供合适的 few-shot 示例、设置最大步数防止无限循环、以及做好错误处理。

### 被问「ReAct vs CoT 的本质区别」
> CoT 不与环境交互，只做内部推理，容易产生幻觉——模型不知道答案时也会推理出看似合理的错误结论。ReAct 通过工具调用获取外部真实信息，用 Observation 修正推理。一句话总结：CoT 只有 Thought，ReAct 是 Thought + Action + Observation 的交替循环。
