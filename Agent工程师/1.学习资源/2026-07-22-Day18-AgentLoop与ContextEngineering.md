# Day 18 — Agent Loop 与 Context Engineering

## 🎯 核心概念

| 概念 | 一句话 | 等级 |
|------|--------|:---:|
| **Context Engineering** | 在有限的上下文窗口中最大化信息价值的系统方法论 | 🔴 |
| **Write（写入策略）** | 决定什么内容进上下文，按什么结构分层 | 🔴 |
| **Select（选择策略）** | 窗口有限时，用三维加权排名决定留什么 | 🔴 |
| **Compress（压缩策略）** | 摘要/事实提取/分层压缩，触发条件与压缩比控制 | 🔴 |
| **Isolate（隔离策略）** | 角色隔离 + Neutral Observation + 会话沙箱 | 🔴 |
| **系统提示词分层** | 角色→任务→约束→知识，优先级倒金字塔 | 🔴 |
| **前缀缓存** | 不变前缀只编码一次，多用户共享 KV Cache | 🟠 |
| **静态前缀分离** | 通用前缀可缓存 + 差异化后缀动态注入 | 🟠 |

---

## 🔄 四大策略全景

```
用户消息到达
    │
    ▼
┌─ Select ──────────────────────┐
│ token > 70%? → 三维加权排名    │
│ 时间×重要性×相关性 → 标记淘汰  │
└───────────────────────────────┘
    │
    ▼
┌─ Compress ────────────────────┐
│ 淘汰消息先压缩再丢弃           │
│ 摘要(5:1) / 事实提取 / 分层    │
│ 压缩后仍超→提高压缩比再压      │
└───────────────────────────────┘
    │
    ▼
┌─ Isolate ─────────────────────┐
│ 工具返回 → Neutral Observation │
│ 角色标签校验 + 上下文沙箱       │
└───────────────────────────────┘
    │
    ▼
┌─ Write ───────────────────────┐
│ 不变前缀(可缓存) → 动态后缀    │
│ System Prompt / User / History │
└───────────────────────────────┘
    │
    ▼
   LLM 调用
```

## 📝 系统提示词四层设计

| 层 | 内容 | 示例 | 可变性 |
|----|------|------|:--:|
| 1 角色 | 你是谁 | "你是一个智能订单客服" | 不变 |
| 2 任务 | 你能做什么 | "查询订单/跟踪物流/解答政策" | 不变 |
| 3 约束 | 你不能做什么 | "只能查询，不能修改" | 不变 |
| 4 知识 | 你知道什么 | "品牌信息/退换货政策/物流方" | 可动态注入 |

**优先级：倒金字塔。4 > 3 > 2 > 1。安全约束压倒一切。**

---

## 🗜️ 压缩手段对比

| 手段 | 压缩比 | 适用场景 | 风险 |
|------|:---:|------|------|
| **摘要压缩** | 5:1 | 客套话/确认/重复 | 摘要 LLM 可能总结错 |
| **事实提取** | 20:1 | 大量工具返回数据 | 丢失叙述过程 |
| **分层压缩** | 动态 | 全场景 | 实现复杂，逐层维护 |

**触发条件：** token 超 70% 窗口 → 首轮轻度（摘要）→ 仍超 → 深度（事实提取）

## ⚡ KV Cache 优化

```
无缓存：每用户每轮都编码全部上下文
  6次调用 × ~2000 token = ~12000 token 编码开销

前缀缓存：System Prompt 只编码一次
  6次调用 → 1次编码1000 token + 6次编码动态部分

静态前缀分离：
  [System Prompt 通用部分 800 token] ← 缓存
  [用户差异化配置 200 token]          ← 每次编码
  [对话历史 ...]                       ← 每次编码
```
