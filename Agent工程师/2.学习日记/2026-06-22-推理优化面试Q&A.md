# 推理优化面试 Q&A 速查

## PagedAttention（🟡 必考）

**Q：vLLM 比 HF 推理快在哪？**
→ PagedAttention（显存利用率 20% → 90%+）+ Continuous Batching（GPU 利用率 50% → 95%+）= 吞吐 **2-4x**

**Q：为什么显存利用率低？**
→ 内部碎片（预分配 2048 只用几百）+ 外部碎片（释放块大小不匹配）。PagedAttention = 固定大小页 + 按需分配 = 零碎片。

**Q：多请求共享 System Prompt 怎么处理？**
→ Copy-on-Write。共享页共用，要修改时复制私有副本。类似 Linux fork()。

**Q：PagedAttention ≠ 新 Attention 算法**
→ 是 KV Cache **内存管理算法**，不是 Attention 计算算法。

## Continuous Batching（🟡 必考）

**Q：Static vs Continuous？**
| Static | Continuous |
|--------|-----------|
| 等凑满一批才跑 | 每轮动态调整 batch |
| 整批同生共死 | 谁完成谁退，立刻补新 |
| 打包延迟 + 尾随效应 | GPU 永不空等 |

**Q：依赖什么底层能力？**
→ PagedAttention。请求动态进出需要 O(1) 分配/释放 KV Cache。

**Q：Prefill 和 Decode 能混跑吗？**
→ vLLM **分离调度**——Prefill 单独算完再进 CB 循环。混跑会拖慢 Decode。

## Speculative Decoding（🟠 加分）

**Q：原理？**
→ 小模型猜 N 个候选 token → 大模型一次性验证 → 拒绝采样保证**精度无损**（数学等价于原模型自回归）。

**Q：加速最明显的场景？**
| ✅ 好用 | ❌ 不好用 |
|--------|---------|
| 代码/SQL/JSON → 2-3x | 长文本创作 |
| 翻译 → 1.5-2x | 草稿模型太小 |
| Chat 短回复 → 2x | 高并发批处理 |

**Q：部署？**
→ `vllm serve --speculative-model xxx --num-speculative-tokens 5`
→ 只传 token ID（整数数组），所以要求**词表兼容**

**Q：为什么大厂推？**
→ 精度无损 + 降本。一行配置省一半 GPU 费用。
