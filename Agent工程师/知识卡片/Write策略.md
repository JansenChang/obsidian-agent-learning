# Write 策略

> 一句话：决定什么内容以什么顺序进入 LLM 的上下文窗口——本质是上下文分层写入，利用"不变前缀可缓存"原则最大化 KV Cache 命中率，是 Context Engineering 中每次 LLM 调用都会执行的策略。

## 核心机制

### 上下文分层结构

```
┌─────────────────────────────────────────────┐
│ System Prompt（角色+任务+约束）  静态前缀     │
│ ├─ 层1: 角色定义                            │ ← 不变，可缓存
│ ├─ 层2: 任务描述                            │ ← 不变，可缓存
│ └─ 层3: 约束规则                            │ ← 不变，可缓存
├─────────────────────────────────────────────┤
│ 知识注入层                       动态后缀     │
│ └─ 层4: RAG检索/用户记忆/动态参数           │ ← 每次变化
├─────────────────────────────────────────────┤
│ 对话历史                         动态后缀     │
│ └─ 用户消息 + Assistant回答 + 工具调用记录   │ ← 持续增长
├─────────────────────────────────────────────┤
│ 当前用户消息                     动态后缀     │
│ └─ 最新消息                                  │ ← 每次不同
└─────────────────────────────────────────────┘
```

### 核心原则：不变在前，可变在后

Write 策略的核心洞察：KV Cache 对前缀连续匹配的 token 可复用。因此：

- **静态前缀放最前**：角色、任务、约束 → 所有用户共享，只编码一次
- **差异化配置紧随其后**：用户级知识注入（偏好/记忆/RAG结果）
- **动态历史追加末尾**：每轮增长，按需压缩

### 写入顺序对性能的影响

| 写入方式 | KV Cache 行为 | 每次调用开销 |
|---------|:------------:|:-----------:|
| 每次全部重写 | 零缓存，完整编码 | ~2000 token 重编码 |
| 静态前缀 + 动态追加 | 前缀命中，后缀增量 | ~1000 token 增量编码 |
| 静态+分层差异化 + 动态 | 最优化 | ~200 token 增量编码 |

## 面试要点

1. **"分层写入"是面试关键词**——能讲清楚四层写入（角色→任务→约束→知识→历史→用户消息），面试官就知道你理解的不只是"往 messages 数组里塞东西"
2. **Write 与 KV Cache 的关系**——Write 策略决定了前缀缓存能省多少计算量，写在最前面的不变内容是性能优化的核心
3. **为什么约束放在知识层前面？**——约束（安全规则）优先级高于知识，即使在知识注入中包含与约束冲突的内容，约束层的"安全压倒一切"设计也必须生效
4. **Write 不是调一次就完**——Agent 循环中每轮都执行 Write，对话历史会不断追加，这是 Select/Compress 被触发的根源
5. **与 LangChain 的关系**——LangChain 的 ChatPromptTemplate 本质是 Write 策略的模板化实现，但它缺少 Select/Compress/Isolate 的联动
