# ConversationSummaryMemory

> 一句话：当对话达到一定长度时，用一个额外的 LLM 调用将旧消息压缩成摘要，后续推理基于摘要而非完整历史——用一次额外 LLM 调用的代价换 token 消耗的恒定控制。

## 核心机制

### 压缩流程

```
对话进行到第 15 轮
  → 触发压缩阈值（如超过 8000 token）
  → 将前 10 轮消息发给 LLM："请将以下对话压缩为一段摘要"
  → LLM 输出摘要："用户在咨询北京三日游行程，已确认偏好自然风光，预算 5000 元..."
  → 消息列表替换为：[摘要] + 最近 5 轮完整消息
  → 后续推理基于摘要而非完整历史
```

### 优劣对比

| 维度 | Buffer | Summary |
|------|--------|---------|
| Token 消耗 | 线性增长 | 恒定（摘要 + 最近 N 轮） |
| 信息完整性 | 100% | 摘要质量决定 |
| 额外成本 | 无 | 每次压缩需一次 LLM 调用 |
| 延迟 | 推理延迟随 token 增长 | 压缩时有一瞬间延迟 |
| 精度 | 精确原文 | 可能有损失（"北京"→"中国"） |

### 精度损失的典型场景

- **地名模糊化**："张三在海淀区工作" → 摘要："张三在北京工作"（丢失区级信息）
- **数值四舍五入**："预算是 4720 元" → 摘要："预算约 5000 元"
- **决策理由丢失**："我选了 A 方案因为便宜" → 摘要："用户选了 A 方案"（丢了"为什么"）
- **时间精度丢失**："上周三下午三点" → 摘要："最近"

## 面试要点

1. **"用计算换记忆"**——Summary 的本质是用一次额外 LLM 调用节省后续所有调用的 token
2. **摘要质量 = 记忆质量**——如果摘要错了，后续所有推理都在错误基础上进行
3. **压缩触发的时机**——不是每轮都压缩，而是 token 超过阈值时触发，控制额外 LLM 调用频率
4. **结构化摘要提示词**——面试中可以主动说：摘要提示词要要求 LLM 保留人名、地名、数值、决策理由
5. **混合模式首选**——近期全量 + 远期摘要 + 超远期丢弃，是生产系统的标准答案
