# 时间旅行与版本树

> 一句话：通过 Checkpoint ID + parent ID 构成的有向无环图（DAG），回到历史任意检查点重新执行或创建分支。数据结构与 Git commit 树完全同构——调试从"看日志猜"升级为"回到现场重跑"。

## 核心机制

### 版本树数据结构

```
checkpoint_1 (parent: null)
  ├── checkpoint_2 (parent: checkpoint_1)
  │     ├── checkpoint_3 (parent: checkpoint_2)   ← 主线
  │     └── checkpoint_3b (parent: checkpoint_2)  ← 分支（回退后走不同路径）
  └── checkpoint_2b (parent: checkpoint_1)        ← 另一条分支
```

每个检查点是一个"commit"，parent 链是"历史"，不同分支是不同执行路径。这个结构与 Git commit 树完全同构。

### 三种核心操作

| 操作 | 行为 | 类比 Git |
|------|------|---------|
| **回溯（Rewind）** | 沿 parent 链回到祖先检查点，从那里重新执行 | `git checkout HEAD~3` |
| **分支（Fork）** | 从历史检查点创建新执行路径，保留原路径 | `git checkout -b new-branch <commit>` |
| **对比（Diff）** | 比较两个检查点的状态差异 | `git diff commit1 commit2` |

### 时间旅行的典型应用

```
场景：Agent 在第5轮做出了一个错误决策

传统调试：
  看日志 → 猜原因 → 改提示词 → 重新跑完整对话
  （费时、不可控、未必能复现）

时间旅行调试：
  rewind_to(checkpoint_4)           // 回到错误决策前
  modify_system_prompt(new_prompt)  // 修改参数
  resume_from(checkpoint_4)         // 从那个点继续执行
  观察结果是否改善
  （快速、精确、可重复实验）
```

**进阶用法**：同一个父检查点创建多个子分支，每个分支用不同的 System Prompt、不同参数、不同工具配置——批量 A/B 测试 Agent 行为。

### LangGraph 中的一等公民

LangGraph 的 `StateGraph` 原生支持时间旅行：
- `graph.get_state(config)` —— 获取任意检查点的完整状态
- `graph.get_state_history(config)` —— 获取状态历史（沿 parent 链回溯）
- `graph.update_state(config, values)` —— 修改检查点状态后重新执行
- 自动为每个 `Node` 执行后保存状态快照

## 面试要点

1. **"与 Git 同构"是面试中的关键词**——它能证明你理解时间旅行不只是"回到过去"的功能，而是一个有严谨数据结构的版本控制系统。每个检查点 = commit，parent ID = 历史链，分支 = 不同执行路径
2. **时间旅行 ≠ 简单的状态回滚**——回滚是丢弃，时间旅行是保留原路径 + 创建新分支。你可以同时拥有多个"未来"，这在 Agent 调试中是革命性的
3. **分支管理的复杂度**——数十个用户 × 多次时间旅行操作 = 复杂的版本树。需要支持分支合并（类比 `git merge`）或至少分支清理策略
4. **为什么用 DAG 而不是树？**——因为不同分支可能汇聚到同一个状态（两个不同路径到达同一个结论），DAG 允许这种多对一关系，树不允许
5. **时间旅行对审计的价值**——金融/医疗场景中，监管机构可能要求"重现 Agent 在某时刻的完整决策过程"。时间旅行直接满足这个需求

## 关联概念

- Checkpoint机制：时间旅行的数据基础——没有 Checkpoint ID + parent ID 就没有版本树
- 状态持久化策略：增量保存让时间旅行的存储成本可控
- Day14 ReAct：时间旅行回到的是 ReAct 循环中的某个 Thought→Action→Observation 节点
- Day19 Multi-Agent：Multi-Agent 的时间旅行需要保存所有 Worker 的协同状态
