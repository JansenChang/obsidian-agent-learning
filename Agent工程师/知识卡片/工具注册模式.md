# 工具注册模式

> 一句话：装饰器、配置文件、动态注册三种工具管理模式，在直观性、灵活性和安全性之间各有取舍。

## 核心机制

### 三种模式对比

| | 装饰器模式 ⭐ | 配置文件模式 | 动态注册 |
|------|-----------|------------|---------|
| **做法** | `@tool` 装饰函数，自动生成 Schema | JSON/YAML 定义所有工具 | 运行时 add/remove |
| **Schema 来源** | 函数签名 + docstring | 配置文件 | 运行时构建 |
| **代码耦合** | 强（函数即工具） | 弱（定义与实现分离） | 极弱 |
| **非技术人员友好** | ❌ 需改代码 | ✅ 只改配置 | ❌ 需要编程 |
| **灵活性** | 中 | 中高 | 极高 |
| **安全风险** | 低 | 低（静态审查） | 高（需 HITL 审批） |
| **代表框架** | LangChain | 企业自研框架 | MCP 协议插件 |

### 装饰器模式（LangChain 风格）

```python
@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气信息。

    Args:
        city: 城市名称，如 "Beijing"
    """
    # ... 调用天气 API
    return f"{city}今天晴，25°C"
```

装饰器自动提取：
- `name` → 函数名 `get_weather`
- `description` → docstring 首行
- `parameters` → 函数参数类型推断

### 配置文件模式

```yaml
tools:
  - name: get_weather
    description: 查询指定城市的天气
    handler: tools.weather.get_weather
    parameters:
      type: object
      properties:
        city:
          type: string
          description: 城市名
      required: [city]
    timeout: 5s
    max_retries: 3
```

启动时解析 → 按 `handler` 路径加载实现 → 注册到 Tool Registry。

### 动态注册 + Human in the Loop

```
Agent 发现需要新工具
  → 动态生成工具 Schema
  → 提交审批节点（HITL）
  → 用户确认 → 注册到 Tool Registry
  → 工具可用
```

没有审批的裸动态注册可能导致：
- 未审查的工具引入安全漏洞
- 工具名冲突
- LLM 滥用临时工具

## 面试要点

1. **三种模式对比是基础题**——能清晰说出各自的优缺点和适用场景
2. **主动提动态注册 + HITL**——面试中从装饰器主动延伸到动态注册的风险和审批机制，给面试官留深刻印象
3. **装饰器如何生成 Schema**——具体说清楚 name/description/parameters 的来源
4. **配置文件的核心优势**——工具描述可被非技术人员编辑优化，这是装饰器做不到的
5. **动态注册不是银弹**——未经审批的动态工具可能被 LLM 滥用，安全风险需明确
