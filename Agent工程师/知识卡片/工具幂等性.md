# 工具幂等性

> 一句话：相同输入重复调用的结果一致性——读操作天然幂等，写操作需要去重设计以防止重复副作用。

## 核心机制

### 什么是幂等性

```
f(x) = f(f(x))  即：调用一次和调用多次，结果相同
```

在 Agent 场景中：同一个工具用相同参数被调用多次，产生的副作用与调用一次完全相同。

### 读 vs 写

| 操作类型 | 天然幂等？ | 示例 | 风险 |
|---------|----------|------|------|
| **读操作** | ✅ 天然幂等 | 查天气、搜文档、读数据库 | 无——返回相同结果即可 |
| **写操作** | ❌ 需设计 | 发邮件、创建订单、写数据库 | 重复调用产生重复副作用 |

典型问题场景：Agent 在 ReAct 循环第 3 轮和第 5 轮用相同参数调了 `send_email` → 用户收到两封一模一样的邮件。

### 幂等性设计方式

**方式 1：幂等键（Idempotency Key）**

```python
def create_order(user_id, product_id, idempotency_key):
    if order_exists(idempotency_key):
        return existing_order  # 已创建，直接返回
    return create_new_order(user_id, product_id)
```

服务端记录已处理的幂等键，重复请求直接返回已有结果。

**方式 2：Tool Description 约束**

在工具描述的 `description` 中明确告诉 LLM：

```
⚠️ 此工具不可重复调用：如果已用相同参数调用过，不要再次调用。
调用前请检查对话历史中是否已有相同操作。
```

但这依赖 LLM 的理解能力，不够可靠。工具端去重是最稳妥的方案。

**方式 3：适配层统一去重**

在适配层维护一个"本次会话已执行操作"的缓存：

```python
class IdempotentAdapter:
    def __init__(self):
        self.executed_calls = set()  # {(tool_name, args_hash), ...}

    def execute(self, tool_call: ToolCall):
        key = (tool_call.name, hash(tool_call.arguments))
        if key in self.executed_calls:
            return {"cached": True, "result": self.cache[key]}
        result = actual_execute(tool_call)
        self.executed_calls.add(key)
        return result
```

### 不能盲目幂等

某些场景需要允许重复调用：
- 用户明确说"再发一次"
- 分布式系统中的至少一次语义
- 幂等键有时效性（如 24h 后失效）

## 面试要点

1. **读天然幂等、写需设计**——这是最核心的区分，面试基础题
2. **幂等键是最佳实践**——Stripe 等 API 都用 `Idempotency-Key` 头，说出这个证明你有实战经验
3. **两级防御**——工具端去重（可靠）+ description 约束（辅助），面试中展示分层思维
4. **重试场景的关键**——指数退避 + 幂等性 = 安全重试，缺一不可
5. **不是绝对值**——说出什么场景需要允许重复，展示你理解 tradeoff

## 关联概念

- 指数退避：重试可能导致重复调用，幂等性保证重试安全
- 并行工具调用：并行批次中同一工具被多次调度时防止重复执行
- 适配器模式：幂等性检查可统一放在适配层，而非散落在每个工具实现中
