# 指数退避（Exponential Backoff）

> 一句话：失败重试间隔按指数增长（1s → 2s → 4s → 8s → ...）的容错策略，避免在故障未恢复时立即重试造成请求风暴。

## 核心机制

### 为什么不能立即重试

工具调用失败的原因通常在短时间内不会消失：
- **API 限流（429）**：对方服务器正在恢复，需要几秒冷却
- **网络抖动**：瞬时故障，但立即重试命中同一故障概率高
- **服务暂时不可用（503）**：后端需要时间恢复

立即重试 → 大概率再失败 → 浪费请求配额，甚至加剧对方压力。

### 退避公式

```
wait_time = base_delay × 2^(retry_count)
```

| 重试次数 | 等待时间 | 累计等待 |
|---------|---------|---------|
| 第 1 次 | 1s | 1s |
| 第 2 次 | 2s | 3s |
| 第 3 次 | 4s | 7s |
| 第 4 次 | 8s | 15s |
| 第 5 次 | 16s | 31s |

通常设置 `max_retries = 3~5`，超过后标记最终失败。

### 加入 Jitter（抖动）

多个并发请求同时失败时，纯粹的指数退避会导致它们"同步"重试——即**惊群效应**。加入随机抖动：

```
wait_time = base_delay × 2^(retry_count) + random(0, 1000ms)
```

错开各请求的重试时刻，避免同时冲击。

### 在 Agent 中的应用

```
工具调用失败
  → 判断错误类型（429/5xx → 可重试, 400/4xx → 不可重试）
  → 指数退避等待
  → 重试
  → 达到上限 → 标记失败，错误信息回传给 LLM
```

不可重试的错误（如参数错误 400）应立即失败，不应浪费重试次数。

## 面试要点

1. **面试必考关键词**——说出"Exponential Backoff"证明你有真实工程经验，不是只调过 API
2. **与固定间隔的对比**——固定间隔（如每次等 3s）在故障持续时不解决问题，指数退避自适应用更长等待
3. **Jitter 的作用**——防止惊群效应，多实例部署场景必须加
4. **可重试 vs 不可重试**——429/5xx 可重试，400 参数错误不应重试，展示错误分类能力
5. **配合超时使用**——指数退避 + 单次超时 + 最大重试次数 = Agent 容错黄金三角

## 关联概念

- 并行工具调用：批次中单个工具失败后的独立重试策略
- 工具幂等性：重试可能导致重复执行，写操作需要幂等性保护
- 适配器模式：退避逻辑应统一写在适配层，而非散落在各工具实现中
