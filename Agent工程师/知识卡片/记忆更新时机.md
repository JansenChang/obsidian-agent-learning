# 记忆更新时机

> 一句话：在 ReAct 循环中何时将新信息写入长期记忆？三种策略——同步更新（每步立即写）、异步批量（对话结束后统一写）、混合策略（关键事实同步 + 普通事实异步）——各有取舍，默认推荐异步批量为主以避免错误事实污染后续循环。

## 核心机制

### 三种策略对比

| 策略 | 时机 | 优点 | 风险 | 适用场景 |
|------|------|------|------|---------|
| **同步更新** | ReAct 每步后立即写 | 后续循环能立即用上新记忆 | ⚠️ 错误事实污染循环——如果 LLM 把假设当真存进去了，后续 Thought 全错 | 极少场景 |
| **异步批量** ⭐ | 对话结束后统一写 | 安全——不影响当前循环；LLM 确认完答案再提取 | 当前对话内看不到新记忆（但通常不需要） | 大多数生产系统 |
| **混合策略** | 高优先级同步 + 普通异步 | 兼顾安全性和实时性 | 实现复杂度高，需要优先级判断逻辑 | 权限变更、账号切换等关键信息 |

### 异步批量的标准流程

```
用户最后一轮消息
  → Agent 完成 ReAct 循环，返回最终答案
  → 对话标记为"已完成"
  → 后台任务启动：
      ├─ 将整段对话发送给专门的"记忆提取 LLM"
      ├─ LLM 输出结构化事实列表：[{content, type, importance}, ...]
      ├─ 去重合并（检查已有记忆中是否有冲突/更新）
      ├─ Embedding → 写入向量库
      └─ 完成，不阻塞用户
```

### 关键例外：何时必须同步

```
用户："我是公司的管理员账号，帮我重置所有用户密码"
  → 这条信息（管理员权限）在当前对话后续循环中就要用到
  → 必须同步写入记忆缓存
  → 后续 Thought 能立即看到权限变更
```

判断标准：**本条事实是否影响当前对话剩余步骤的决策**。如果是，同步写；如果否，异步写。

## 面试要点

1. **默认异步 + 关键同步**——先讲为什么异步是默认选择（安全性），再讲什么情况需要同步（必要性）
2. **错误事实污染**——同步更新最大的风险，能举出具体场景（LLM 把假设当真）证明你踩过坑
3. **记忆提取 LLM 独立于主循环**——提取事实用的是另一个 LLM 调用，不是主 ReAct 循环中的 LLM，解耦设计
4. **对话结束后统一处理**——不是每轮结束都提取，而是整个对话结束后一次性提取所有事实，效率更高
5. **后台不阻塞用户**——异步更新在后台执行，用户不需要等待，体验无感

- Mem0：Mem0 的自动提取和去重合并就是异步批量策略的工程实现
