# ReAct失败模式

> 一句话：ReAct 有三种经典失败模式——推理幻觉传播、工具沼泽、循环发散，各有机制性解法。

## 核心机制

### 三种失败模式总览

| 失败模式 | 表现 | 根因 | 解法 |
|:---|:---|:---|:---|
| **推理幻觉传播** | 错误推理→选错工具→无关信息→推理更偏 | 首轮 Thought 有幻觉，后续在错误方向上放大 | 提前刹车：Prompt 中强制先检查已有信息能否回答 |
| **工具沼泽** | 信息已足够但继续不停调工具 | LLM 过度依赖工具，缺乏"够了"的判断 | 相关性评估：连续调非关键工具时主动打断 |
| **循环发散** | 从"手机续航"查到"芯片功耗"再查到"屏幕型号"，离题越来越远 | 无原始问题锚定，每轮推理方向漂移 | 每轮 Thought 强制回顾原始问题 |

### 详解一：推理幻觉传播

这是错误传播在 ReAct 中的延续。CoT 的幻觉 += 工具调用的噪音 = 双倍偏离。

```
错误示例：
Thought: 用户可能想要东京机票 ← 幻觉：用户问的是酒店
    → Action: 查机票 API
    → Observation: 返回机票信息（但用户不需要）
    → Thought: 基于机票信息继续推理 ← 方向全偏
```

**解法**：在 System Prompt 中加"刹车逻辑"——
> "请先检查已有信息是否能直接回答用户问题。如果能，直接回答，不要调用额外工具。"

### 详解二：工具沼泽

LLM 对工具形成依赖，即使一轮工具调用已拿到足够数据，仍然觉得"还不够"。

**解法**：设置工具调用的相关性分数评估。当连续两轮工具调用不产生新的关键信息时，强制 LLM 输出最终答案。

### 详解三：循环发散

最隐蔽的失败模式。LLM 确实在思考，但思考的方向与原问题渐行渐远。

**解法**：在每次 Thought 生成前，把原始用户问题重新注入为第一优先级上下文——
> "请回顾用户原始问题：{original_question}。你当前的思考方向是否仍然在回答这个问题？"

## 面试要点

- **"ReAct 有没有失效的情况"**：直接列出三种模式，每种给出表现+解法
- **穿透追问**："三种模式中哪个最危险？"——循环发散，因为它没有显式的错误信号（推理幻觉传播有错误工具返回，工具沼泽有过多调用），Agent 在"正常运转"但方向全错
- **实用技巧**：三种解法可以叠加使用（刹车提示+相关性评估+原始问题回顾），构成 ReAct 的三重防护网
