# Compress 策略

> 一句话：对 Select 标记淘汰的消息进行信息浓缩，用摘要压缩（5:1）、事实提取（20:1）、分层压缩三种手段逐级递进，确保被丢弃的消息中的关键信息不丢失。

## 核心机制

### 三级压缩递进

```
第一级：摘要压缩（轻量）
  └─ 对淘汰消息调用摘要 LLM 生成摘要
  └─ 压缩比 5:1，适用于客套话/确认/重复内容
  └─ 摘要替代原始消息写入上下文
  └─ 检查 token 用量 → 仍超 → 进入第二级

第二级：事实提取（深度）
  └─ 从淘汰消息中只提取结构化事实
  └─ 压缩比 20:1，适用于大量工具返回数据
  └─ 格式：[时间] 实体 → 事实
  └─ 检查 token 用量 → 仍超 → 进入第三级

第三级：分层压缩（终极）
  └─ 对已压缩的内容再次压缩（摘要的摘要）
  └─ 压缩比动态调整，直到满足窗口预算
  └─ 实现最复杂，逐层维护，信息损失最大
```

### 三种压缩手段对比

| 手段 | 压缩比 | 适用场景 | 风险 |
|------|:-----:|---------|------|
| **摘要压缩** | 5:1 | 客套话、确认信息、重复轮次 | 摘要 LLM 可能总结错误或遗漏细节 |
| **事实提取** | 20:1 | 大量工具返回数据、长 API 响应 | 丢失叙述过程，可能丢失推理链路 |
| **分层压缩** | 动态 | 极端窗口压力、多轮压缩仍超限 | 实现复杂，多级压缩错误累积 |

### 触发条件

```
token 用量 > 70% 窗口上限
    → Select 标记淘汰消息
    → 首轮：摘要压缩
    → 仍 > 70%：事实提取
    → 仍 > 70%：分层压缩（提高压缩比，再压）
```

## 面试要点

1. **为什么不是直接丢掉？**——淘汰的消息可能包含关键事实（用户地址、订单号、决策结果），直接丢弃 = 信息丢失，先压缩再丢弃 = 保留关键信息的同时释放窗口
2. **压缩比不是越高越好**——摘要 5:1 有信息损失但可控，事实提取 20:1 损失叙述过程，分层压缩可能丢失推理链路。面试时要能谈 trade-off
3. **压缩 LLM 的选择**——压缩摘要建议使用廉价快速模型（如 GPT-4o-mini 或本地小模型），因为压缩是高频高吞吐操作
4. **与 Day17 SummaryMemory 的区别**——ConversationSummaryMemory 在会话结束做一次总结用于长期存储；Compress 在会话进行中实时触发，是动态窗口管理
5. **错误放大效应**——如果摘要 LLM 对关键事实总结错了，后续所有推理都基于错误信息。这是 Compress 策略最危险的风险点

## 关联概念

- Select策略：Compress 只处理被 Select 标记淘汰的内容
- ConversationSummaryMemory：Day17 的总结记忆是总结→存储，Compress 是压缩→保留在窗口
- Context Engineering四大策略：Compress 在流水线中的位置是 Select 之后、Isolate 之前
- Context Window Pressure：token 逼近上限时的压缩触发信号
