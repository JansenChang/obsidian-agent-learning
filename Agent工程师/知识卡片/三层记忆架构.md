# 三层记忆架构

> 一句话：Agent 的记忆按时间尺度分为三层——工作记忆（秒-分钟级，ReAct 中间变量）、短期记忆（分钟-小时级，当前会话历史）、长期记忆（天-年级，跨会话持久化用户信息），三层不是上下级而是时间刻度的差异。

## 核心机制

### 三层模型

```
工作记忆 (秒-分钟)
  └─ ReAct 循环中的 Thought、工具返回结果、临时状态
  └─ 载体：程序内存变量 / Redis 短期缓存
  └─ 任务完成后释放，无需显式存储

短期记忆 (分钟-小时)
  └─ 当前会话完整对话历史
  └─ 载体：上下文窗口（128K-200K token）
  └─ Day14 ReAct 的 messages 列表即短期记忆载体

长期记忆 (天-年)
  └─ 跨会话持久化：用户偏好、背景、历史决策
  └─ 载体：向量数据库 + Embedding
  └─ 按需检索 → 注入 System Prompt 或 Thought 前
```

### 灰度地带：工作记忆的持久化

大多数场景下工作记忆 = 程序变量，回答后就丢弃。但如果任务跨多轮——"你先帮我规划，我晚上再告诉你决定"——工作记忆需要序列化到 Redis，设较短有效期（如 24h）。这是工作记忆和短期记忆之间的灰色地带。

### 时间维度的面试表达

| 层级 | 时间尺度 | 载体 | 生命周期 |
|------|---------|------|---------|
| 工作记忆 | 秒-分钟 | 程序内存 | 任务结束释放 |
| 短期记忆 | 分钟-小时 | 上下文窗口 | 会话结束释放 |
| 长期记忆 | 天-年 | 向量数据库 | 持久化/按遗忘策略淘汰 |

## 面试要点

1. **时间维度 > 框架名词**——能讲清楚"三层是时间刻度的差异"比背 LangChain 类名有说服力得多
2. **跨轮任务的状态管理**——工作记忆何时需要持久化到 Redis？这是区分初级和高级工程师的问题
3. **三层独立性**——三层可以独立选择技术方案：工作记忆用 Redis，短期记忆用消息列表，长期记忆用向量库
4. **与 RAG 的关系**——长期记忆本质是"用户专属 RAG"，同一套技术栈（Embedding + 向量检索）
5. **反向思考**——什么场景不需要三层？一次性问答 Agent 只需要工作记忆+短期记忆
