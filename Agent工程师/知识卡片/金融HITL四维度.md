# 金融 HITL 四维度

> 一句话：将通用 HITL 框架落地到金融行业时，必须额外考虑的四个维度——审批分级（降低人工瓶颈）、超时策略（应对市场时效）、审计记录（合规证据）、权限模型（职责分离）——缺一不可。

## 核心机制

### 四维度全景

```
                    金融 HITL 系统设计
                          │
        ┌────────┬────────┼────────┬────────┐
        ▼        ▼        ▼        ▼        ▼
    审批分级   超时策略   审计记录   权限模型
   （谁审批） （多久等） （留证据） （谁能批）
```

### 维度一：审批分级

| 级别 | 触发条件 | 审批者 | 示例 |
|:---:|---------|--------|------|
| **L0 免审** | 查询操作、无副作用 | 无 | 查询账户余额、查看交易历史 |
| **L1 一级审批** | 小额交易 / 低风险写操作 | 值班风控人员 | 转账 <1万元 |
| **L2 二级审批** | 大额交易 / 高风险操作 | 风控经理 | 转账 ≥1万元 |
| **L3 额度制** | 极高频交易 | 总额度内自动 + 超额审批 | 高频量化交易（日额度50万） |

> 核心设计原则：不是所有操作都需要最高级别审批。审批分级降低人工瓶颈——让 90% 的操作自动执行，只有真正的关键操作才排队等人。

### 维度二：超时策略

| 操作类型 | 超时行为 | 原因 |
|---------|---------|------|
| **买入** | 超时自动取消 | 市场价格实时变化，过期订单无意义 |
| **卖出** | 超时升级（转交更高权限审批者） | 卖出的紧迫性可能高于买入 |
| **查询** | 超时自动通过 | 无副作用，不需要阻塞 |
| **修改限额** | 超时自动拒绝 + 记录 | 权限变更的安全性优先级最高 |

### 维度三：审计记录

每笔审批事件必须记录（不可篡改）：

```
审计条目 {
  who: "张三（风控主管）",         // 谁审批
  when: "2026-07-24T10:30:15Z",    // 什么时间
  what: "批准转账 50,000 元",       // 审批的操作
  result: "approved",               // 审批结果
  context: {                        // 审批时的完整上下文
    user_id: "U12345",
    transaction_detail: {...},
    risk_score: 0.72,
    checkpoint_id: "ckpt_042"
  }
}
```

> 这不是调试日志，是法律合规的证据。金融行业的审计要求意味着 HITL 系统的日志是法律文件，不是可选的工程实践。

### 维度四：权限模型

```
职责分离（Separation of Duties）：
  ┌─────────────┐     ┌─────────────┐
  │ 权限配置者   │     │ 审批执行者   │
  │ （管理员）   │     │ （风控人员） │
  └──────┬──────┘     └──────┬──────┘
         │                   │
         ▼                   ▼
    配置谁可以审批        只能在自己权限内审批
    （不能自己审批自己     （不能自己改自己的
     配置的操作）           审批权限）
```

权限设计的核心原则：
- **细粒度**：张三只能审批 A 类交易，李四只能审批 B 类交易
- **职责分离**：审批者不可配置权限，权限配置者不可执行审批
- **权限变更需审批**：修改任何人的审批权限本身需要另一个人的审批

## 面试要点

1. **为什么金融 HITL 是四个维度而不是三个或五个？**——审批分级（降低瓶颈）、超时策略（应对时效）、审计记录（合规底线）、权限模型（职责分离）。这四个维度是金融监管的最小完备集。面试官如果追问是否有第五个维度，可以说"通知机制（审批请求的多渠道推送：短信+App+邮件）可以作为第五维度"
2. **超时不等于拒绝——超时策略是操作类型相关的**。买入超时取消、卖出超时升级、查询超时通过。这个分类讨论体现你做过真正的系统设计
3. **职责分离是金融机构的硬性合规要求**——SOX（萨班斯法案）、PCI-DSS 都要求审批权与配置权分离。在 Agent HITL 中同理，能配置审批规则的人和能审批的人必须是不同角色
4. **审计记录的"不可篡改"如何实现？**——追加写入（Append-Only）+ 哈希链（类似区块链的链式校验）。面试时提"Write-Once-Read-Many (WORM) 存储"是加分项
5. **审批分级在面试中容易漏掉的维度**——不要只说按金额分级，还要说按操作类型（读/写）、按数据敏感度（PII/非PII）、按频率（逐笔/额度制）三维交叉

## 关联概念

- Human-in-the-Loop：四维度是通用 HITL 在金融行业的落地模板
- Checkpoint机制：每条审计记录都关联 Checkpoint ID，确保可追溯到审批时的完整 Agent 状态
- Day14 ReAct：金融 Agent 的每次交易决策都在 ReAct 循环中，HITL 审批是循环中的暂停点
- Day17 Memory：审批时展示用户历史交易记忆，辅助风控人员决策
