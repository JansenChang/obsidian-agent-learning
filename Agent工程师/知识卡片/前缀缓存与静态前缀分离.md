# 前缀缓存与静态前缀分离

> 一句话：利用 KV Cache 的前缀匹配特性，将 System Prompt 中不变的部分（角色/任务/约束）作为静态前缀只编码一次、多调用复用；动态内容（用户消息/对话历史/知识注入）作为差异化后缀每轮增量编码，大幅降低 Agent 多轮调用的推理成本。

## 核心机制

### KV Cache 前缀命中原理

```
请求1: [System Prompt 800t] [用户A消息 200t] [历史...]
       └── 编码KV ────┘ └── 编码KV ──────────┘

请求2: [System Prompt 800t] [用户B消息 200t] [历史...]
       └── 命中缓存 ✓ ──┘ └── 重新编码 ────────┘
       
请求3: [System Prompt 800t] [用户A第二轮 150t] [历史...]
       └── 命中缓存 ✓ ──┘ └── 重新编码 ────────────┘
```

KV Cache 按前缀连续匹配：只要前缀 token 序列相同，就可以直接复用已缓存的 Key-Value 对，节省自回归生成中的重复计算。

### 静态前缀分离策略

```
┌──────────────────────────────────────┐
│  静态前缀（所有用户共享，缓存一次）     │
│  ├─ 角色定义：~200 token              │
│  ├─ 任务描述：~300 token              │
│  └─ 约束规则：~300 token              │
│  合计：~800 token → 只编码一次        │
├──────────────────────────────────────┤
│  差异化后缀（每请求重新编码）           │
│  ├─ 用户配置/记忆注入：~200 token      │
│  ├─ 对话历史：持续增长                 │
│  └─ 当前用户消息：每次不同             │
└──────────────────────────────────────┘
```

### 性能对比

| 场景 | 无缓存 | 前缀缓存 | 静态前缀分离 |
|------|:------:|:-------:|:----------:|
| 6次调用 × 2000 token | ~12,000 token 编码 | ~7,000 token | ~5,200 token |
| 缓存命中率 | 0% | ~40% | ~57% |
| TTFT (首个Token时间) | 基准 | -30% | -45% |

## 面试要点

1. **Agent 场景为什么特别适合前缀缓存？**——Agent 循环中 System Prompt 完全不变，每轮只是追加 Thought/Action/Observation，前缀连续性极好，缓存命中率天然高
2. **"静态前缀分离"的工程含义**——不是把 System Prompt 拆成两个文件，而是把 System Prompt 中"所有请求不变的"放最前面，"每次变化的"放后面。顺序是关键
3. **为什么差异化配置不能放最前面？**——因为前缀缓存要求"从第一个 token 开始连续匹配"，如果第 1 个 token 就不同（用户A vs 用户B 的配置），缓存完全失效
4. **与 PagedAttention 的协同**——vLLM 的 PagedAttention 管理 KV Cache 内存块，前缀缓存的命中块可以跨请求共享，实现物理内存复用
5. **缓存失效场景**——System Prompt 修改（模型升级/策略变更）→ 所有缓存失效；System Prompt 中插入动态内容（如把用户ID写在第1个 token）→ 缓存完全失效

## 关联概念

- KV Cache（键值缓存）：前缀缓存的底层机制
- PagedAttention（分页注意力）：vLLM 中 KV Cache 的物理内存管理
- Write策略：决定了静态前缀和动态后缀的边界在哪里
- 系统提示词分层设计：前3层静态=前缀缓存资产，第4层动态=差异化后缀
- Context Engineering四大策略：Write 策略中的前缀缓存优化是整个策略链的性能基础
