# Agent评估三层次

> 一句话：Agent 评估不能只看最终答案对错，必须从组件、轨迹、端到端三个层次分别衡量可靠性、智能性和有用性。

## 核心机制

### 三个层次的定义

| 层次 | 评估什么 | 回答的问题 | 方法 |
|:---|:---|:---|:---|
| **组件评估** | 每个工具独立测试 | 工具是否可靠？ | 单元测试工具返回的准确性、性能、稳定性 |
| **轨迹评估** | 完整推理-行动链是否合理 | Agent 是否聪明？ | 用另一个 LLM 当评判官，逐轮检查 Thought 的逻辑性 |
| **端到端评估** | 最终答案正确性 | Agent 是否有用？ | 标准准确率/满意度指标 |

### 为什么只做端到端评估不够

```
场景：Agent 在某个 Thought 中做了错误推理
       ——"今天有台风所以航班会延误"（实际上没有台风）
       ——但 Action 碰巧正确，调了航班查询工具
       ——最终答案准确

端到端评估：✅ 正确 → 得分高
轨迹评估：   ❌ 推理有逻辑漏洞 → 暴露深层问题
```

两个 Agent——一个推理全错但碰巧结果对，一个推理全对结果也对——在端到端评估中得分完全相同。但前者**不可信赖**，后者**可复现**。对于生产系统，不可信赖比不准确更致命。

### 生产级评估体系

好的评估体系一定是三层组合：
1. **组件评估** → 确保地基（工具层）稳固
2. **轨迹评估** → 确保 Agent 的决策过程可解释、可复现
3. **端到端评估** → 确保最终用户价值

只做第三层，等于只看结果不看过程，无法区分"蒙对的"和"真的会"。

## 面试要点

- **"你怎么评估你的 Agent 好不好"**：不要只说准确率。展示你知道需要分层评估，能说出三层各自回答什么问题
- **"轨迹评估怎么做"**：用 LLM-as-Judge，给评判 LLM 完整的 Thought-Action-Observation 链，让它判断每一步推理是否有逻辑漏洞
- **穿透追问**："推理全错但结果对的 Agent，你怎么打分？"——端到端给高分，轨迹给低分。解释为什么两者必须分开评估
