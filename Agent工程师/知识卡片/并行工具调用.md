# 并行工具调用

> 一句话：利用线程池同时执行 LLM 返回的多个无依赖工具调用，将总耗时从"所有工具耗时之和"压缩到"最慢工具的耗时"，显著降低 Agent 响应延迟。

## 核心机制

### 执行流程

```
LLM 返回 [tool_1, tool_2, tool_3]
  │
  ├── ThreadPool ──┬── execute(tool_1) ──→ result_1
  │                ├── execute(tool_2) ──→ result_2
  │                └── execute(tool_3) ──→ result_3
  │
  ├── 等待所有完成（总耗时 ≈ max(t1, t2, t3)）
  ├── 按 tool_call_id 排序合并
  └── 一次性发回 LLM
```

### 串行 vs 并行延迟对比

假设 3 个工具各需 1s 执行，每轮 LLM 调用 2s：

| 模式 | 3 工具耗时 | 3 轮 LLM 调用 | 总延迟 |
|------|----------|-------------|--------|
| 串行（3 轮） | 3×1s=3s | 3×2s=6s | ~9s |
| 并行（1 轮） | max(1s)=1s | 1×2s=2s | ~3s |

ReAct 循环 5 轮、每轮并行 3 个工具 → 可省 8-10s 用户等待时间。

### 部分失败处理三方案

| 方案 | 做法 | 适用场景 |
|------|------|----------|
| **全或无** | 任意一个失败，整个批次作废，LLM 重来 | 工具间强依赖，部分成功无意义 |
| **部分成功** ⭐ | 成功结果正常回传，失败标记错误信息，LLM 自决 | 最常用——工具间独立，LLM 可选择性重试 |
| **自动重试** | 失败工具独立重试（退避策略），最终汇总 | 工程最完善，实现最复杂 |

### 关键约束

- **无依赖**：工具间不能有数据依赖（tool_2 需要 tool_1 的结果），否则必须串行
- **id 对应**：结果必须按 `tool_call_id` 精确对应，DeepSeek 对此要求极严格
- **顺序保持**：多 tool 消息发回 LLM 时，tool_call_id 顺序必须与原始 tool_calls 一一致

## 面试要点

1. **延迟优化是核心价值**——并行调用不是锦上添花，是体验优化的必要手段
2. **部分成功是最佳实践**——面试中对比三种方案并说明选择理由
3. **DeepSeek 的一一对应要求**——并行调用在 DeepSeek 上更需要小心 id 匹配
4. **OpenAI 和 Anthropic 原生支持**——两者都可在单次响应中返回多个工具调用
5. **无依赖是前提**——能并行的是独立工具，有依赖关系的必须串行

## 关联概念

- 指数退避：并行批次中单个工具失败后的重试策略
- 工具幂等性：并行重试场景下防止重复副作用
- Day14 ReAct：并行调用减少 ReAct 循环轮数，加速 Agent 响应
