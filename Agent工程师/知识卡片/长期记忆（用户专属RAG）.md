# 长期记忆（用户专属RAG）

> 一句话：长期记忆本质上是面向用户个人的 RAG 系统——将用户事实、偏好、历史决策 Embedding 后存入向量数据库，对话需要时相似度检索后注入上下文。技术栈与 RAG 完全相同（Embedding → 向量库 → Rerank），唯一区别是存储的是用户事实而非外部文档。

## 核心机制

### 架构管线

```
对话结束
  → 后台 LLM 提取事实（结构化列表：内容 + 类型 + 重要性）
  → 去重合并（新事实覆盖旧事实的冲突部分）
  → Embedding 编码
  → 向量数据库存储（带元数据：用户 ID、时间戳、事实类型、重要性评分）

新对话 / 新话题
  → Embedding 当前上下文
  → 向量库相似度检索 + 用户 ID 过滤
  → Top-K 召回 + Rerank 精排
  → 注入 System Prompt 或 Thought 前
```

### 与 Day8-13 RAG 的关系

| 维度 | 外部 RAG | 长期记忆 |
|------|---------|---------|
| 数据来源 | 企业文档、知识库 | 用户对话历史 |
| Chunking | 固定大小文本块 | 结构化事实条目 |
| Embedding | 相同 | 相同 |
| 向量数据库 | 相同 | 相同 |
| 检索 | 语义相似度 | 语义相似度 + 用户 ID 过滤 |
| Rerank | 相同 | 相同（相关性 + 重要性加权） |

**结论：在 RAG 上积累的工程经验，改两行配置就能搭出长期记忆系统。**

### 检索结果的排序与截断

一次检索可能返回几十条相关记忆，不能全塞进 System Prompt。排序公式：

```
score = α × relevance（语义相似度）+ β × importance（事实的重要性评分）
取 Top-K
```

K 值不是固定的——对话越复杂需要越多记忆。闲聊 K=3~5，复杂决策 K=10~15。

## 面试要点

1. **"长期记忆 = 用户专属 RAG"**——这一句话证明你理解了两套系统的本质联系，而不是孤立地学概念
2. **全链路理解**——提取→去重→Embedding→存储→检索→注入，能画出完整数据流
3. **检索排序的双因子**——不只靠相似度，还要加权重要性评分，体现工程细节
4. **K 值动态调整**——不是写死的常量，而是根据对话复杂度的自适应策略
5. **多租户隔离**——每条记忆打用户 ID 标签，检索时强制过滤，这是生产系统的安全底线
