## 一、LLM 基础与原理

### Q: 什么是大语言模型 (LLM)，它是如何工作的？
**来源:** ai-engineering-interview-questions, Awesome-LLM, LLMInterviewQuestions
**答案要点:** LLM 是基于 Transformer 架构的大规模神经网络，通过自监督学习从海量文本中学习语言模式。采用 Causal Decoder-only 架构进行自回归文本生成，核心是通过预测下一个 token 来完成各类自然语言任务。

### Q: 预测式 AI (Predictive/Discriminative AI) 和生成式 AI (Generative AI) 有什么区别？
**来源:** LLMInterviewQuestions
**答案要点:** 预测式 AI 学习从输入到标签的映射（分类、回归），生成式 AI 学习数据的底层分布以生成新样本。LLM 属于生成式 AI。

### Q: LLM 开发各阶段是什么？（预训练、SFT、RLHF/DPO）
**来源:** Awesome-LLM, ai-engineering-interview-questions, LLM-Interview-Hub
**答案要点:** (1) 预训练 (Pre-training)：在海量语料上进行自监督学习（Next Token Prediction）；(2) 监督微调 (SFT)：用高质量标注数据调优指令遵循能力；(3) 偏好对齐 (Alignment)：通过 RLHF/DPO 等方法对齐人类偏好，提升安全性和有用性。

### Q: LLM 与统计语言模型（N-gram/HMM）的主要区别是什么？
**来源:** Devinterview-io/llms
**答案要点:** LLM 使用 Transformer 架构的 Self-Attention 捕获全局依赖，支持亿级参数和大规模上下文窗口；传统模型基于 N-gram/HMM，受限于固定短窗口的 Markov 假设，无法处理长距离依赖。LLM 产生上下文感知的动态 Embedding，传统模型产生静态 Embedding（Word2Vec/GloVe）。

### Q: LLM 训练和推理有什么区别？
**来源:** LLM-Interview-Hub
**答案要点:** 训练是前向+反向传播的过程，更新模型权重以最小化损失函数，计算量大但可以并行。推理只有前向传播，使用已训练好的权重预测下一个 token，自回归生成，KV Cache 优化避免重复计算。

### Q: LLM 推理时到底在做什么？（自回归生成）
**来源:** AI_Engineer_Interview_Prep
**答案要点:** LLM 接收 token 序列作为输入，经过多层 Transformer Block（Self-Attention + FFN），对词汇表中每个 token 产生一个概率分布，从中采样下一个 token。采样后的 token 被追加到序列中，整个过程重复。模型没有全局"计划"，逐 token 生成。

### Q: Scaling Law 是什么？
**来源:** LLM-Interview-Hub, llmgenai
**答案要点:** 模型性能（Loss）与计算量 C、参数量 N、数据量 D 之间遵循幂律关系：C ≈ 6ND。随着这三个因素中任何一个的增加，性能可预测地提升，这指导了大模型的资源分配策略。

### Q: 大模型开发各阶段分别包括什么？
**来源:** Awesome-LLM
**答案要点:** 预训练阶段（Pre-training）、监督微调阶段（SFT/Instruction Tuning）、偏好对齐阶段（RLHF/DPO）、评估和部署阶段。

### Q: LLM 如何处理上下文和长距离依赖关系？
**来源:** Devinterview-io/llms
**答案要点:** 通过 Scaled Dot-Product Attention 机制建立全局依赖，RoPE (Rotary Positional Embeddings) 编码相对位置，GQA (Grouped-Query Attention) 减少 KV Cache 内存。2026 年标准还包括 FlashAttention-3、Ring Attention、Mamba-2 等 State Space Models (SSM) 实现 O(n) 复杂度。

### Q: Self-Supervised Learning 在 LLM 预训练中的意义是什么？
**来源:** LLM-Interview-Hub
**答案要点:** 自监督学习让模型无需人工标注即可从海量文本中学习。以 Causal Language Modeling (CLM) 为目标函数，模型通过预测下一个 token 自动学习语言的语法、语义和知识。

## 二、Transformer 与 Attention

### Q: 解释 Transformer 架构及其关键组件
**来源:** ai-engineering-interview-questions, Devinterview-io/llms, Awesome-LLM
**答案要点:** 2026 年标准为 Causal Decoder-only 架构。关键组件包括：(1) Self-Attention 机制（Scaled Dot-Product Attention）；(2) RMSNorm 替代 LayerNorm；(3) SwiGLU 激活函数替代 ReLU；(4) RoPE 位置编码替代 Sinusoidal；(5) Grouped-Query Attention (GQA) 优化 KV Cache；(6) 残差连接 (Residual Connections)。

### Q: Self-Attention 是怎么计算的？逐步解释
**来源:** LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** 对每个 token 生成 Q (Query)、K (Key)、V (Value) 向量。计算 QK^T 得到注意力分数矩阵，除以 √d_k 进行缩放（防止梯度消失），经过 Softmax 得到注意力权重，最后与 V 加权求和。公式：Attention(Q,K,V) = softmax(QK^T/√d_k)V。

### Q: Self-Attention 的计算复杂度是多少？
**来源:** LLM-Interview-Hub
**答案要点:** 时间复杂度 O(n² · d)，空间复杂度 O(n² + n · d)，其中 n 是序列长度，d 是 embedding 维度。这是 Transformer 的主要瓶颈。

### Q: MHA (Multi-Head Attention)、MQA (Multi-Query Attention)、GQA (Grouped-Query Attention)、MLA (Multi-head Latent Attention) 的区别？
**来源:** Awesome-LLM, ai-engineering-interview-questions
**答案要点:** MHA：每个 head 有独立的 Q、K、V；MQA：所有 head 共享 K、V；GQA：若干 head 共享一组 K、V（折中方案，如 Llama 系列）；MLA：DeepSeek 提出，通过低秩压缩进一步减少 KV Cache。GQA 是 2026 年生产环境主流。

### Q: Position Encoding 有哪几种？RoPE 为什么优于 Sinusoidal？
**来源:** Awesome-LLM, ai-engineering-interview-questions, LLM-Interview-Hub
**答案要点:** (1) 绝对位置编码（Sinusoidal）：直接用三角函数编码每个位置；(2) 可学习位置编码：将位置作为可训练参数；(3) RoPE (Rotary Position Embeddings)：通过旋转矩阵编码相对位置信息，支持更好的上下文窗口外推 (LongRoPE/YaRN)。RoPE 是 2026 年标准。

### Q: Encoder-only、Decoder-only、Encoder-Decoder 三种架构的区别和适用场景？
**来源:** Awesome-LLM, ai-engineering-interview-questions
**答案要点:** Encoder-only (BERT)：双向注意力，适合 NLU 任务（分类、NER）；Decoder-only (GPT/Llama)：因果掩码自回归生成，主导生成式 AI；Encoder-Decoder (T5/BART)：适合翻译、摘要等序列到序列任务。

### Q: 为什么用 Transformer 而不是 LSTM/RNN？
**来源:** LLMInterviewQuestions, LLM-Interview-Hub
**答案要点:** (1) Transformer 可并行计算，LSTM 必须串行；(2) Self-Attention 直接建模全局依赖，LSTM 存在长程遗忘问题；(3) Self-Attention O(1) 路径长度连接任意两个 token，RNN 需要 O(n)。

### Q: Transformer 如何处理梯度消失问题？
**来源:** LLM-Interview-Hub
**答案要点:** (1) 残差连接 (Residual/Skip Connections) 提供梯度直通路径；(2) Layer Norm / RMSNorm 稳定激活分布；(3) 注意力分数的 √d_k 缩放防止 Softmax 进入饱和区。

### Q: Softmax 函数在 Transformer 中哪里使用？有什么作用？
**来源:** LLM-Interview-Hub
**答案要点:** 在 Self-Attention 中将注意力分数转换为概率分布。也用于最终输出层将 Logits 转换为词汇表概率。

### Q: Cross Attention 与 Self-Attention 有什么区别？
**来源:** LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** Self-Attention 中 Q、K、V 来自同一序列；Cross Attention 中 Q 来自当前序列（Decoder），K/V 来自另一个序列（Encoder 输出）。用于 Encoder-Decoder 架构中的信息桥接。

### Q: Causal Masking 是什么？为什么需要？
**来源:** ai-engineering-interview-questions
**答案要点:** Causal Masking（因果掩码）确保 token i 只能关注 token j ≤ i，即不能看到"未来"的 token。上三角部分置为 -∞，经 Softmax 后权重为 0。这是自回归生成的必要条件。

### Q: Flash Attention 是什么？解决了什么问题？
**来源:** ai-engineering-interview-questions, LLM-Interview-Hub
**答案要点:** Flash Attention 是 IO-aware 的注意力算法，通过分块计算和重计算策略，将显存访问从 O(n²) 降低到 O(n)，在 GPU HBM 和 SRAM 之间优化数据搬运。FlashAttention-3（2026）进一步利用异步 TMAX/TMIN 操作提升吞吐。

### Q: Cross-Entropy Loss 在 Transformer 训练中如何应用？
**来源:** LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** 比较模型预测的概率分布与真实的下一个 token（One-hot 分布），计算负对数似然：L = -Σ log P(x_i | x_{<i})。

## 三、Tokenization 与 Embedding

### Q: Tokenization 是什么？为什么 LLM 不使用字符或整词？
**来源:** AI_Engineer_Interview_Prep, ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** Tokenization 是将原始文本转换为模型处理的整数 ID 的过程。字符级：序列太长（O(n²) 注意力成本爆炸）；词级：词汇表过大，无法处理未登录词 (OOV)；子词级（BPE/WordPiece/SentencePiece）：平衡词汇表大小和序列长度，现代 LLM 标准。

### Q: BPE (Byte Pair Encoding) 是怎么工作的？
**来源:** AI_Engineer_Interview_Prep, ai-engineering-interview-questions, Awesome-LLM
**答案要点:** 从字符级词汇开始，迭代合并训练数据中出现频率最高的相邻 token 对。重复 N 次（如 32K-100K 次），最终词汇表捕捉最常见的子词模式。

### Q: Embedding 是什么？Transformer 内部的 Embedding 和 Embedding API 的 Embedding 有何不同？
**来源:** ai-engineering-interview-questions, AI_Engineer_Interview_Prep
**答案要点:** Embedding 是高维空间中的稠密向量表示。Transformer 内部的 Embedding 将 token ID 映射为输入向量，经过多层变换后含上下文信息，但为"预测下一个 token"优化；Embedding API（如 text-embedding-3）是专门为语义相似度训练的对比学习模型。

### Q: Token Embedding 在 Transformer 中如何生成？
**来源:** LLM-Interview-Hub
**答案要点:** 通过 Embedding Layer（可学习的查找表），将每个 token ID 映射为一个 d_model 维度的稠密向量。这个向量会被加上位置编码后输入 Transformer Block。

### Q: Context Window 为什么不能无限大？
**来源:** AI_Engineer_Interview_Prep, ai-engineering-interview-questions
**答案要点:** (1) 注意力计算 O(n²) —— 翻倍窗口长度 = 4 倍计算量；(2) KV Cache 随序列长度线性增长；(3) 训练/推理分布不匹配导致性能下降（Lost in the Middle 现象）；(4) 位置编码外推能力有限。

### Q: "Lost in the Middle" 问题是什么？如何解决？
**来源:** AI_Engineer_Interview_Prep, ai-engineering-interview-questions
**答案要点:** LLM 对长上下文中"开头"和"结尾"的信息关注度更高，中间位置的信息容易被忽略。缓解方法：(1) 将关键信息放在开头和结尾；(2) 使用 Reranker 筛选最相关的 Chunk；(3) 在 Prompt 中重复关键指令；(4) 使用递归摘要减少上下文长度。

### Q: 如何处理 OOV (Out-of-Vocabulary) 问题？
**来源:** LLM-Interview-Hub
**答案要点:** 子词分词（BPE/SentencePiece）天然解决 OOV —— 任何未知词都可以分解为已知子词单元的组合。极端情况下回退到字节级表示。

## 四、训练与微调

### Q: 什么是 Fine-tuning？在什么情况下需要 Fine-tune？
**来源:** LLMInterviewQuestions, AI_Engineer_Interview_Prep, Awesome-LLM
**答案要点:** Fine-tuning 是使用特定领域/任务数据进一步训练预训练模型的过程。需要 Fine-tune 的场景：(1) Prompt Engineering 效果不够好；(2) 需要模型学习特定的输出格式/风格；(3) 领域术语和知识模型不掌握；(4) 有 500+ 高质量标注样本。

### Q: Supervised Fine-Tuning (SFT) 和 Alignment (RLHF/DPO) 的区别？
**来源:** ai-engineering-interview-questions, LLM-Interview-Hub
**答案要点:** SFT 使用输入-输出对教模型遵循指令；Alignment 通过偏好数据（哪个回答更好）优化模型的安全性、有用性和行为风格。SFT 教能力，Alignment 教偏好。

### Q: Full Fine-tuning vs PEFT (Parameter-Efficient Fine-Tuning) 的区别？
**来源:** LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** Full Fine-tuning 更新全部参数（训练成本高、需要大显存）；PEFT 只更新少量参数（<1%），包括 LoRA（低秩适配）、Prefix Tuning、Adapter 等方法，大幅降低训练成本。

### Q: LoRA 是什么？它是如何工作的？
**来源:** Awesome-LLM, LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** LoRA (Low-Rank Adaptation) 通过在预训练权重旁添加低秩分解矩阵来模拟权重更新：ΔW = BA，其中 B∈R^(d×r)，A∈R^(r×k)，rank r << d。只有 A 和 B 被训练，大幅减少可训练参数。W_updated = W_pretrained + (α/r)BA。

### Q: QLoRA 和 LoRA 的区别？
**来源:** LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** QLoRA = LoRA + 4-bit 量化 + Double Quantization + Paged Optimizers。可在消费级 GPU（如单张 RTX 4090）上微调 65B 模型，显存需求降低约 4 倍。

### Q: RLHF (Reinforcement Learning from Human Feedback) 流程是什么？
**来源:** LLMInterviewQuestions, ai-engineering-interview-questions
**答案要点:** (1) 收集人类偏好数据（对同一 Prompt 的不同回答排序）；(2) 训练 Reward Model 学习人类偏好；(3) 使用 PPO 算法微调 LLM，使其输出最大化 Reward Model 评分。RLHF 也面临 Reward Hacking 问题。

### Q: DPO (Direct Preference Optimization) 和 RLHF 的区别？
**来源:** ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** DPO 无需显式训练 Reward Model，也不需要 RL 循环。它直接通过偏好对数据优化 LLM，将 RLHF 的两步过程简化为一步，训练更稳定，计算成本更低。2026 年 DPO 和 KTO 正逐渐取代传统 RLHF。

### Q: RLAIF 是什么？与 RLHF 有何不同？
**来源:** ai-engineering-interview-questions
**答案要点:** RLAIF (RL from AI Feedback) 使用另一个 AI 模型（而非人类）提供反馈信号。成本更低，可扩展性更强，但质量依赖 AI 反馈模型的准确性。

### Q: 灾难性遗忘 (Catastrophic Forgetting) 是什么？如何防止？
**来源:** Awesome-LLM, LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** 在领域微调后模型丧失通用能力。缓解方法：(1) 使用 PEFT/LoRA 只更新少量参数；(2) 混合通用数据和领域数据进行训练；(3) 使用 Elastic Weight Consolidation (EWC)；(4) 渐进式学习/课程学习。

### Q: Instruction Tuning 是什么？为什么对 Chat 模型重要？
**来源:** LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** Instruction Tuning 是使用（指令→回答）格式的数据微调模型，使模型学会遵循人类指令而非仅完成文本。这是将 Base Model 转化为 Chat Model 的关键步骤。

### Q: 如何准备 Fine-tuning 数据集？
**来源:** LLMInterviewQuestions, ai-engineering-interview-questions
**答案要点:** (1) 确保数据质量高于数量（500-1000 个高质量样本 > 10000 个噪声样本）；(2) 覆盖目标任务的多样性场景；(3) 数据格式与模型预期格式一致；(4) 包含反例（不期望的输出）；(5) 用合成数据生成补充稀缺类别。

### Q: 指令微调数据集需要遵循什么样的格式？
**来源:** Awesome-LLM
**答案要点:** 不同训练框架有不同格式要求。常见格式包括：System Prompt → User → Assistant 的多轮对话格式（ChatML 风格），或 {instruction, input, output} 的模板格式。需遵循对应框架的数据集规范。

### Q: 强化学习（如 GRPO）的数据集需要什么样的格式？奖励函数如何设计？
**来源:** Awesome-LLM
**答案要点:** GRPO 需要问题+多个生成回答的格式，通过规则奖励函数（如格式奖励、准确性奖励、一致性奖励）进行评分比较。奖励函数设计需平衡多个维度，避免单一指标导致模型走捷径。

### Q: Gradient Accumulation 是什么？如何帮助大模型训练？
**来源:** LLM-Interview-Hub
**答案要点:** 在多个 mini-batch 上累积梯度后再更新权重，有效模拟更大的 batch size 而无须对应显存。使小显存设备也能以较大的有效 batch size 训练。

### Q: PPO、DPO、GRPO 的区别和联系？
**来源:** Awesome-LLM
**答案要点:** PPO 是经典强化学习算法，需要 Reward Model + Value Model，训练复杂；DPO 直接通过偏好对优化，无需显式 Reward Model；GRPO (Group Relative Policy Optimization) 是 DeepSeek 提出的方法，通过组内比较消除对 Value Model 的依赖，更适合大模型对齐训练。

### Q: 增量预训练 (Continual Pretraining) 是什么？
**来源:** Awesome-LLM, ai-engineering-interview-questions
**答案要点:** 在已预训练的模型基础上使用新领域数据继续进行预训练，让模型学习新领域知识（如法律、医疗），同时保持原有能力。与 Fine-tuning 不同，增量预训练使用更大规模的无监督数据。

### Q: 多 LoRA Adapter 如何合并？
**来源:** ai-engineering-interview-questions
**答案要点:** 可以将多个 LoRA 权重按权重求和 W = W_base + Σ(α_i · B_iA_i) 合并。也可使用 SLERP (Spherical Linear Interpolation) 或 TIES-Merging 进行更精细的模型合并。

### Q: 如何估算 Fine-tuning 的资源需求？
**来源:** LLMInterviewQuestions
**答案要点:** 全量微调：参数量的 4-6 倍显存（模型权重 + 梯度 + 优化器状态 + 激活）。使用 LoRA 可降低至 1-2 倍。考虑 batch size、序列长度、GPU 数量等因素进行估算。

## 五、推理优化与部署

### Q: KV Cache 是什么？如何加速推理？
**来源:** LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** KV Cache 缓存已生成 token 的 Key 和 Value 向量，避免每次生成新 token 时重新计算所有历史 token 的 K/V。将每次推理的复杂度从 O(n² · d) 降为 O(n · d)。KV Cache 大小 = 2 × batch_size × num_layers × seq_len × num_kv_heads × head_dim × dtype_size。

### Q: 量化 (Quantization) 为什么不会显著降低模型精度？
**来源:** LLMInterviewQuestions, ai-engineering-interview-questions
**答案要点:** LLM 参数分布通常集中在较窄的数值范围内，低精度（INT8/INT4/FP8）仍能保留足够的表达能力。此外，模型训练时的过参数化使得存在大量冗余，量化相当于某种正则化。常用方法包括 GPTQ、AWQ、GGUF、bitsandbytes。

### Q: Flash Attention 与 Paged Attention 的区别？
**来源:** ai-engineering-interview-questions
**答案要点:** Flash Attention 优化单次 Attention 计算的 IO 效率；Paged Attention (vLLM) 将 KV Cache 分成固定大小的 "Page" 进行管理，类似操作系统虚拟内存，解决了 KV Cache 内存碎片化和过度预留问题。

### Q: Continuous Batching 和 Static Batching 的区别？
**来源:** LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** Static Batching 等所有请求完成后才开始新批次；Continuous Batching（也叫 In-flight Batching）在新生成的 token 级别动态添加/移除请求，显著提升 GPU 利用率和吞吐量。vLLM 和 SGLang 都支持。

### Q: Speculative Decoding 是什么？
**来源:** LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** 使用小型 Draft Model 快速生成 N 个候选 token，再由大模型一次性验证。通过的 token 直接作为输出，失败时回退。可实现 2-3 倍推理加速，不改变输出分布。

### Q: 如何优化 LLM 推理延迟和成本？
**来源:** ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** (1) 模型量化（INT4/INT8/FP8）；(2) Flash Attention/Paged Attention；(3) Continuous Batching；(4) Speculative Decoding；(5) KV Cache 压缩/量化；(6) Prompt Caching；(7) 语义缓存；(8) Model Routing（简单 Query→小模型，复杂→大模型）；(9) Token Streaming。

### Q: Prefill vs Decode 阶段有什么区别？
**来源:** ai-engineering-interview-questions
**答案要点:** Prefill 阶段处理输入 Prompt 的所有 token，并行计算（Compute-bound）；Decode 阶段逐步生成输出 token，自回归串行（Memory-bound）。TTFT (Time to First Token) 主要由 Prefill 决定，生成速度主要由 Decode 决定。

### Q: FP8/FP16/BF16 训练和推理的区别及优势？
**来源:** LLMInterviewQuestions, ai-engineering-interview-questions
**答案要点:** FP16 半精度训练需要 Loss Scaling 防止梯度下溢；BF16 指数位更多，范围更大，无需 Loss Scaling，更稳定；FP8 进一步压缩，需要 Transformer Engine 硬件支持（H100/B200），训练吞吐翻倍。

### Q: GGUF 格式是什么？它的优势？
**来源:** ai-engineering-interview-questions
**答案要点:** GGUF 是 llama.cpp 使用的高效推理格式，支持多种量化级别（Q4_0 到 Q8_0），专为 CPU 推理和边缘设备优化。一个文件包含模型权重、tokenizer 和元数据。

### Q: vLLM 如何工作？和 SGLang 的区别？
**来源:** ai-engineering-interview-questions
**答案要点:** vLLM 使用 Paged Attention 高效管理 KV Cache，Continuous Batching 实现高吞吐推理。SGLang 额外提供 Radix Attention（前缀缓存）和结构化生成语言（SGLang DSL），在某些场景下延迟更低。

### Q: Model Parallelism vs Data Parallelism vs Tensor Parallelism vs Pipeline Parallelism？
**来源:** ai-engineering-interview-questions, Awesome-LLM
**答案要点:** Data Parallelism：每张 GPU 持有完整模型副本，处理不同数据；Model Parallelism/Tensor Parallelism：模型层内参数切分到多 GPU（如 Self-Attention 的 head 维度）；Pipeline Parallelism：模型层间切分，不同 GPU 处理不同层。现代训练常结合三者（3D Parallelism）。

### Q: 混合精度训练 (Mixed Precision Training) 是什么？
**来源:** Awesome-LLM, LLM-Interview-Hub
**答案要点:** 前向和反向传播使用 FP16/BF16 加速计算，权重更新使用 FP32 保持精度。需要动态 Loss Scaling 防止 FP16 梯度下溢（BF16 不需要）。2026 年 H100/B200 还支持 FP8 训练。

### Q: 如何估算大模型训练的显存和时间？
**来源:** Awesome-LLM
**答案要点:** 显存 ≈ 模型参数 × (参数精度 + 梯度 + 优化器状态) + 激活内存。例如 7B 模型 FP16 训练需约：7B × (2+2+8) bytes = 84GB + 激活内存。时间 ≈ (总 token 数 × 6 × 参数量) / (GPU 数量 × 每 GPU TFLOPS × 利用率)。

### Q: 在线推理 (Online) vs 离线推理 (Offline) 的部署区别？
**来源:** LLM-Interview-Hub
**答案要点:** Online 推理需要低延迟、支持 Streaming，侧重 TTFT 和 TPOT (Time Per Output Token)；Offline 推理侧重吞吐量，可使用更大的 Batch Size。Online 需要 Continuous Batching，Offline 可使用 Static Batching。

### Q: Token Streaming 如何工作？
**来源:** ai-engineering-interview-questions
**答案要点:** 服务器端使用 SSE (Server-Sent Events) 或 WebSocket 将每个生成的 token 逐步推送到客户端，配合 KV Cache 实现逐 token 输出。降低用户感知延迟，无需等待完整响应。

### Q: Semantic Caching 是什么？与普通 Cache 的区别？
**来源:** ai-engineering-interview-questions
**答案要点:** 传统缓存基于完全匹配（Exact Match）；语义缓存基于语义相似度匹配——如果新 Query 与缓存 Query 语义足够相近，直接返回缓存结果。使用 Embedding 模型判断相似度，大幅提升缓存命中率。

### Q: Cold Start 问题如何解决？
**来源:** ai-engineering-interview-questions
**答案要点:** 模型加载和初始化需要时间。解决方法：(1) 预热 (Keep-warm) 请求；(2) 模型预加载/常驻内存；(3) 使用模型缓存池；(4) Serverless 场景使用更快的量化模型；(5) 懒加载 + 预取策略。

## 六、Prompt Engineering

### Q: Prompt Engineering 是什么？为什么重要？
**来源:** LLMInterviewQuestions, ai-engineering-interview-questions
**答案要点:** Prompt Engineering 是设计和优化 LLM 输入文本以获取期望输出的技术。它是 AI 工程师最基础也最常用的工具——成本最低、迭代最快、无需重训模型。

### Q: Zero-shot、One-shot、Few-shot Prompting 的区别？
**来源:** ai-engineering-interview-questions, LLM-Interview-Hub
**答案要点:** Zero-shot：不给示例，直接描述任务；One-shot：给一个示例；Few-shot：给多个示例（通常 3-8 个）。示例帮助模型理解任务格式和期望输出风格。

### Q: Chain-of-Thought (CoT) Prompting 是什么？何时使用？为什么有效？
**来源:** ai-engineering-interview-questions, LLM-Interview-Hub, LLMInterviewQuestions
**答案要点:** CoT 要求模型在给出最终答案前展示中间推理步骤。适用于数学、逻辑推理等复杂任务。效果源于它引导模型沿着推理轨迹生成 token，中间步骤增加了最终答案的准确率。

### Q: Self-Consistency 和 CoT 的关系？
**来源:** LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** Self-Consistency 对同一问题采样多条 CoT 推理路径，取多数投票结果。实质是对推理路径进行集成，比单条 CoT 更鲁棒。

### Q: ReAct Prompting 是什么？
**来源:** AI_Engineer_Interview_Prep, ai-engineering-interview-questions, LLM-Interview-Hub
**答案要点:** ReAct (Reasoning + Acting) 将推理和行动交替进行——模型输出思考过程，然后调用工具，观察结果，再思考下一步。是 AI Agent 的核心范式。

### Q: Tree of Thought (ToT) Prompting 是什么？
**来源:** ai-engineering-interview-questions
**答案要点:** ToT 将推理过程建模为树状搜索，生成多个推理步骤候选，评估后选择最有希望的路径继续探索。比 CoT 更强大但计算成本更高。

### Q: System Prompt 和 User Prompt 的区别？
**来源:** LLM-Interview-Hub, ai-engineering-interview-questions
**答案要点:** System Prompt 设定模型的行为准则、角色、输出格式等全局约束；User Prompt 是最终用户的实际输入。System Prompt 通常具有更高优先级。

### Q: Prompt Injection 是什么？如何防御？
**来源:** AI_Engineer_Interview_Prep, ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** 恶意用户输入劫持 LLM 行为，使其忽略系统指令或执行非预期操作。防御策略：(1) 输入分类器检测注入；(2) 分隔符策略（XML/特殊 token 区隔指令和用户输入）；(3) 输出验证；(4) 最小权限原则（LLM 不应有直接高危操作权限）；(5) 多层防御纵深。

### Q: 如何优化 Prompt 的成本和延迟？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 精简 Prompt 长度；(2) Prompt Caching 复用缓存；(3) 选择更小的模型处理简单任务；(4) 使用 Token Streaming 降低感知延迟；(5) 语义缓存复用相似查询结果。

### Q: "Lost in the Middle" 在 Prompt 中的表现和解决方案？
**来源:** ai-engineering-interview-questions, AI_Engineer_Interview_Prep
**答案要点:** 长 Prompt 中间的指令容易被忽略。解决：将关键指令放在 Prompt 开头和结尾；使用分层 Prompt 结构；Rerank 相关 Context 而非全部放入。

### Q: Prompt Chaining 是什么？如何设计？
**来源:** ai-engineering-interview-questions
**答案要点:** 将复杂任务分解为多个 Prompt 调用链，每步的输出作为下一步输入。好处：(1) 每步任务简单可控；(2) 中间结果可验证；(3) 某步失败可独立重试。常见模式：Extract → Analyze → Generate → Validate。

### Q: In-Context Learning (ICL) 是什么？与 Few-shot Prompting 的关系？
**来源:** LLMInterviewQuestions, LLM-Interview-Hub
**答案要点:** ICL 是 LLM 从 Prompt 中的示例"学习"任务模式的能力——模型不更新权重，仅通过示例理解任务。Few-shot Prompting 就是 ICL 最直接的实现形式。

### Q: 如何设计 Prompt Template 用于生产环境？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 使用变量占位符支持动态内容；(2) 清晰分隔系统指令、用户输入、上下文；(3) 版本控制管理 Prompt 变更；(4) A/B 测试验证效果；(5) 包含 fallback 逻辑。

## 七、RAG (检索增强生成)

### Q: RAG 是什么？它的核心架构是什么？
**来源:** All repos
**答案要点:** RAG = Retrieval + Generation。离线流程：文档 → 分块 (Chunking) → Embedding → 向量数据库。在线流程：用户查询 → 查询变换/重写 → Embedding → 向量检索 → Reranking → 上下文组装 → LLM 生成 → 带引用的回答。

### Q: RAG vs Fine-tuning：何时用哪个？
**来源:** LLMInterviewQuestions, ai-engineering-interview-questions, LLM-Interview-Hub
**答案要点:** RAG 适合：(1) 需要最新/实时知识；(2) 知识频繁更新；(3) 需要可溯源的引用；(4) 数据量巨大。Fine-tuning 适合：(1) 需要改变模型行为风格；(2) 领域术语/格式 Pattern；(3) RAG+Prompt 已碰触质量天花板。

### Q: Chunking 策略有哪些？如何选择 Chunk Size？
**来源:** LLMInterviewQuestions, ai-engineering-interview-questions
**答案要点:** (1) Fixed-size：按固定 token 数切分；(2) Recursive：按段落→句子→词递归切分；(3) Semantic：基于语义边界（如按标题/章节）；(4) Parent-Child：大块保存上下文，小块用于检索。Chunk Size 通常 256-2048 tokens，Overlap 10-20%。原则：太小缺少上下文，太大检索精度下降。

### Q: Semantic Chunking vs Fixed-size Chunking 区别？
**来源:** ai-engineering-interview-questions
**答案要点:** Fixed-size 按固定 token 数分割，简单但可能切断语义单元；Semantic Chunking 基于 Embedding 相似度变化判断语义边界，在语义转折处分隔，检索质量更高但成本更高。

### Q: Hybrid Search (混合搜索) 是什么？为什么比纯向量搜索更好？
**来源:** LLMInterviewQuestions, ai-engineering-interview-questions
**答案要点:** Hybrid Search = Vector Search（语义相似） + BM25/Keyword Search（关键词精确匹配）。纯向量搜索对缩写、代码、专有名词等可能效果差；BM25 擅长精确匹配。混合搜索结合两者优势。

### Q: Reranker 是什么？如何改善 RAG 检索质量？
**来源:** Awesome-LLM, ai-engineering-interview-questions
**答案要点:** Reranker 对向量检索返回的 Top-K 结果进行二次精排。Cohere Rerank、Cross-Encoder 等模型将 Query-Document 对作为输入进行精细相关性评分。通常提升 Precision 10-30%。

### Q: Cross-Encoder vs Bi-Encoder 用于 Reranking 的区别？
**来源:** AI_Engineer_Interview_Prep
**答案要点:** Bi-Encoder 分别编码 Query 和 Document（适用于大规模检索的速度要求）；Cross-Encoder 联合编码 Query-Document 对（精度更高但速度慢，仅适用于 Top-K 重排序）。

### Q: Agentic RAG 和 Naive RAG 的区别？
**来源:** ai-engineering-interview-questions
**答案要点:** Naive RAG 是线性的检索→生成流程。Agentic RAG 由 Agent 控制流程：自主决策何时检索、检索什么、是否需要改写查询、是否需要多步检索、是否需要使用不同检索源。更灵活但更复杂。

### Q: 什么是 Self-RAG？GraphRAG？
**来源:** ai-engineering-interview-questions
**答案要点:** Self-RAG：模型自己决定是否需要检索、检索结果是否相关、生成答案是否得到检索支持；GraphRAG：使用知识图谱增强 RAG，从实体和关系中提取结构化知识，适用于需要多跳推理和实体关系的场景。

### Q: HyDE 是什么？在 RAG 中如何应用？
**来源:** ai-engineering-interview-questions
**答案要点:** HyDE (Hypothetical Document Embeddings)：让 LLM 先生成假设性答案，再用假设答案的 Embedding（而非原始 Query）进行检索。假设答案的向量空间与文档更接近，提升检索准确度。

### Q: RAG 系统如何评估？（Faithfulness、Relevance、Context Precision/Recall）
**来源:** ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** 关键指标：(1) Faithfulness：生成内容是否忠实于检索的上下文；(2) Answer Relevance：答案是否相关；(3) Context Precision：检索到的相关文档占比；(4) Context Recall：所有相关文档中被检索到的占比。RAGAS、DeepEval 等框架提供标准化评估。

### Q: Parent-Child Chunking 是什么？
**来源:** ai-engineering-interview-questions
**答案要点:** 将文档分为大块（Parent，如 2048 tokens，保留完整上下文）和小块（Child，如 512 tokens，用于 Embedding 检索）。检索返回 Child Chunk，但传给 LLM 的是对应的 Parent Chunk，保证上下文完整。

### Q: Query Transformation 在 RAG 中有哪些技术？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) HyDE：生成假设答案再检索；(2) Query Decomposition：复杂问题拆为子问题；(3) Step-Back Prompting：先问更抽象的问题获取背景；(4) Multi-Query：生成多个查询变体分别检索后合并。

### Q: RAG 系统的常见 Failure Mode 和调试方法？
**来源:** ai-engineering-interview-questions, rag-interview-system
**答案要点:** (1) 检索失败（相关文档未召回）；(2) 上下文窗口溢出；(3) 检索到矛盾信息；(4) Embedding 不匹配/漂移；(5) Reranker 失效；(6) 索引过期。调试流程：Trace 查询 → 检查检索结果 → 检查生成质量 → 针对性修复。

### Q: 如何处理 RAG 中的表和图？
**来源:** LLMInterviewQuestions
**答案要点:** (1) 表格：使用专门的 Table Parser 提取为结构化格式（Markdown/JSON）；(2) 图表/图像：使用多模态模型（如 GPT-4o Vision）生成文字描述；(3) 大表格：分块或提取摘要后嵌入。

### Q: 如何处理 RAG 中的多跳问题（Multi-hop Questions）？
**来源:** ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** (1) Query Decomposition：拆分子问题，逐跳检索和推理；(2) Iterative Retrieval：每轮检索结果指导下一轮查询；(3) Knowledge Graph RAG：利用实体关系图进行多跳推理；(4) Agentic RAG：Agent 自主规划多步检索策略。

### Q: RAG 中如何实现 Citation 和 Source Attribution？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 检索时携带 source metadata（文件名、页码）；(2) Prompt 中要求模型标注引用（如 [Source 1]）；(3) 后处理验证生成的引用是否确实存在于检索文档中；(4) 使用 NLI (Natural Language Inference) 模型验证 Claim 是否被 Source 支撑。

### Q: RAG 系统从 10K 扩展到 1M 文档，什么先出问题？
**来源:** AI_Engineer_Interview_Prep
**答案要点:** (1) 检索延迟显著增加（ANN 索引需要更多内存/分片）；(2) 相同 Top-K 下 Recall 下降（候选池增大）；(3) Metadata 过滤变慢；(4) 索引更新周期变长。需要引入分片 (Sharding)、HNSW 参数调优、多阶段检索。

### Q: Long Context Models (128K+) 是否能替代 RAG？
**来源:** AI_Engineer_Interview_Prep
**答案要点:** 不能完全替代：(1) 128K 上下文仍有 O(n²) 成本问题；(2) Lost in the Middle 效应；(3) 成本随 Context 线性增长；(4) 知识更新需要重新填充 Context。RAG 仍是更高效的选择，两者实际形成互补。

### Q: RAG 系统中的 Embedding 成本优化？
**来源:** AI_Engineer_Interview_Prep
**答案要点:** (1) 增量索引（只重嵌新增/变更文档）；(2) Embedding 缓存；(3) 使用更小的 Embedding 模型；(4) Embedding 量化（int8/binary）；(5) 降低 Embedding 维度（如 text-embedding-3 支持 Matryoshka 维度缩减）。

### Q: Recursive Document Summarization RAG 是什么？
**来源:** rag-interview-system
**答案要点:** 对长文档分层摘要：将文档分块→每块生成摘要→摘要合并再摘要→直到得到适合上下文的长度。保留层级结构，支持不同粒度的检索需求。

## 八、AI Agent

### Q: AI Agent 是什么？与简单的 LLM 调用有何不同？
**来源:** ai-engineering-interview-questions, Awesome-LLM
**答案要点:** Basic LLM: Input → Model → Output（单次）；Agent: Input → Think → Act → Observe → Think → … → Output（迭代循环）。Agent 有自主决策能力：决定使用什么工具、何时停止、如何处理错误。

### Q: ReAct Agent 架构是什么？
**来源:** Awesome-LLM, ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** ReAct = Reasoning + Acting。Agent 交替输出 Thought（思考下一步做什么）、Action（调用工具）、Observation（工具返回结果），循环直到任务完成。是当前多数 Agent 框架的核心范式。

### Q: Plan-and-Execute Agent 模式是什么？
**来源:** ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** 先制定完整执行计划，再逐步执行各步骤。相比 ReAct，优点是有全局视野、减少中间决策错误；缺点是计划可能不够灵活，遇到意外难以调整。

### Q: Function Calling / Tool Use 是如何实现的？
**来源:** Awesome-LLM, ai-engineering-interview-questions
**答案要点:** (1) 定义 Tool Schema（函数名、描述、参数 JSON Schema）；(2) 将 Schema 传入 LLM API；(3) LLM 返回 Tool Call JSON（函数名+参数）；(4) 应用代码执行函数；(5) 将执行结果返回给 LLM。LLM 本身不执行任何代码。

### Q: MCP (Model Context Protocol) 是什么？
**来源:** ai-engineering-interview-questions
**答案要点:** Anthropic 提出的开放标准，定义 LLM 应用如何通过统一协议连接外部工具和数据源。好处：(1) 一次编写，到处使用；(2) 标准化安全模型；(3) 运行时工具发现；(4) 生态快速增长。

### Q: Single-Agent vs Multi-Agent 系统的区别？
**来源:** ai-engineering-interview-questions
**答案要点:** Single-Agent：一个 LLM + 多个工具，处理全部任务；Multi-Agent：多个专业化 Agent 协作。Multi-Agent 适用场景：(1) 上下文窗口压力大；(2) 需要冲突的系统提示（如辩论 Agent）；(3) 不同 Agent 需要不同工具访问权限。

### Q: Agent Memory 有哪些类型？如何实现？
**来源:** ai-engineering-interview-questions, AI_Engineer_Interview_Prep
**答案要点:** (1) Short-term：当前对话上下文/滑动窗口；(2) Long-term：跨会话持久化记忆（向量数据库 + 检索）；(3) Episodic：具体事件和经验记忆；(4) Semantic：概念和知识。实现：Buffer → Summary → Vector Store → 混合架构。

### Q: Agent Loop 是什么？如何决定何时停止？
**来源:** ai-engineering-interview-questions
**答案要点:** Agent Loop = Think → Act → Observe 的迭代。停止条件：(1) 最大迭代次数；(2) Agent 输出 "Final Answer" / 特定停止 Token；(3) 连续无进展检测；(4) 成本/Token 预算耗尽；(5) 置信度达到阈值。

### Q: 如何防止 Agent 陷入无限循环？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 硬限制最大步骤数；(2) 检测重复输出（相同 Action 连续 N 次）；(3) 监控 Token 消耗和耗时；(4) 使用看门狗 (Watchdog) 超时机制；(5) 给 Agent 显式的循环检测 Prompt 指令。

### Q: Human-in-the-Loop 模式什么时候需要？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 高风险操作（删除数据/资金交易）；(2) 低置信度决策；(3) 合规要求；(4) 需要人工判断的复杂场景（医疗诊断、法律建议）。实现：在关键 Action 前插入 Approval 步骤。

### Q: Agent 的 Guardrails 如何实现？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 输入层：Prompt Injection 检测、输入分类器；(2) 决策层：约束可用 Tool 列表、限制 Action 参数范围；(3) 输出/执行层：Tool 参数验证、结果合规检查、输出格式验证；(4) 架构层：最小权限、沙箱执行、审计日志。

### Q: Context Engineering 是什么？
**来源:** Awesome-LLM, ai-engineering-interview-questions
**答案要点:** 系统化管理 LLM 的上下文窗口——包括 System Prompt、Few-shot 示例、历史对话、检索文档、工具输出等组件的编排和优化。Context Compaction 是其中的关键技术，在上下文溢出时压缩或裁剪信息。

### Q: Agent 失败时如何实现 Error Recovery？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 工具调用失败：重试机制 + 指数退避 + 备选工具；(2) 输出格式错误：重新 Prompt + 格式验证；(3) 推理方向错误：回溯 (Backtracking) + 反思 (Reflection)；(4) 超时：提前终止 + 部分结果返回。

### Q: Code-Generating Agent vs Tool-Calling Agent 的区别？
**来源:** ai-engineering-interview-questions
**答案要点:** Tool-Calling Agent：LLM 选择并调用预定义的 API/函数；Code-Generating Agent：LLM 生成代码，然后在沙箱中执行（如 OpenAI Code Interpreter）。前者更可控但灵活性低；后者更强大但安全性要求更高。

### Q: Harness Engineering 是什么？
**来源:** ai-engineering-interview-questions
**答案要点:** 设计和构建 Agent 的"控制层"——管理 Agent Loop、Tool 编排、错误处理、状态管理、上下文压缩等基础设施。是连接 LLM 能力和产品需求的关键工程层。

### Q: Context Compaction 如何工作？
**来源:** ai-engineering-interview-questions
**答案要点:** 当对话/上下文超出窗口限制时：(1) 摘要压缩历史对话；(2) 滑动窗口保留最近 N 条消息；(3) 选择性保留关键信息（Tool 调用结果、重要决策）；(4) 分层存储（近期用完整文本、远期用摘要）。

## 九、评估与测试

### Q: LLM 输出如何评估？常用指标有哪些？
**来源:** ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** (1) 传统指标：BLEU (机器翻译)、ROUGE (摘要)、BERTScore (语义相似度)；(2) LLM-based 指标：G-Eval、LLM-as-a-Judge；(3) 任务特定指标：Accuracy、F1、Exact Match（QA）；(4) 人工评估：A/B 比较、Likert 评分。

### Q: LLM-as-a-Judge 评估的优缺点是什么？
**来源:** ai-engineering-interview-questions
**答案要点:** 优势：可评估开放式生成质量（流畅性、有帮助性、安全性），比传统 N-gram 指标更贴近人类判断。局限：(1) LLM Judge 自身也有 Bias；(2) 位置偏差（倾向于前/后位置答案）；(3) 成本较高；(4) 对评估 Prompt 设计敏感。

### Q: RAGAS 评估框架的核心指标？
**来源:** AI_Engineer_Interview_Prep, ai-engineering-interview-questions
**答案要点:** (1) Faithfulness：生成答案与检索上下文的事实一致性；(2) Answer Relevancy：答案与问题的相关度；(3) Context Precision：检索结果中相关文档的比例；(4) Context Recall：所有相关文档中被召回的比例；(5) Context Relevancy：检索到的上下文总体相关度。

### Q: Red Teaming 是什么？如何 Red Team 一个 LLM 应用？
**来源:** ai-engineering-interview-questions
**答案要点:** 系统性尝试让 AI 系统产生有害、偏见或不安全的输出，以发现漏洞。方法：(1) 设计对抗性 Prompt；(2) 测试边界 Case；(3) 多语言/多文化测试；(4) 越狱 (Jailbreaking) 测试；(5) 跨模态攻击测试。

### Q: MMLU, HumanEval, GSM8K 等 Benchmark 如何解读？
**来源:** ai-engineering-interview-questions
**答案要点:** MMLU：多领域知识理解（57 学科）；HumanEval：代码生成能力；GSM8K：数学推理能力。注意：(1) Benchmark 可能有数据污染；(2) 不代表实际生产表现；(3) 需结合自建领域 Eval Set 综合评估。

### Q: G-Eval 是什么？和传统指标的区别？
**来源:** ai-engineering-interview-questions
**答案要点:** G-Eval 使用 LLM（如 GPT-4）作为评估器，通过 Chain-of-Thought 生成评估步骤，再给出评分。比 BLEU/ROUGE 更擅长评估生成质量，但成本较高且需要精心设计评估 Prompt。

### Q: Offline Evaluation vs Online Evaluation 的区别？
**来源:** ai-engineering-interview-questions
**答案要点:** Offline Eval：使用固定的 Golden Dataset 离线评估（快速、可重现、但可能不代表真实分布）。Online Eval：在生产环境中实时监控（如用户满意度、任务完成率、A/B 测试），反映真实效果但成本高、反馈慢。

### Q: 如何构建 Eval Suite (Golden Dataset)？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 覆盖所有典型用例和边界 Case；(2) 包含正例和反例；(3) 持续从生产反馈中更新（Data Flywheel）；(4) 标注需要明确的标准和多人校验；(5) 避免与训练数据重叠。

### Q: AI Agent 如何评估？
**来源:** ai-engineering-interview-questions
**答案要点:** 多维评估：(1) 任务完成率；(2) 平均步骤数/耗时；(3) 工具调用准确率；(4) 错误恢复率；(5) Token 效率；(6) 最终输出质量。需设计端到端场景测试而非单一组件评估。

### Q: Hallucination 如何检测和测量？
**来源:** ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** (1) NLI 模型验证 Claim 是否被 Source 支撑；(2) 要求模型标注不确定性；(3) Self-Check：生成→提取 Claim→逐一验证；(4) LLM-as-a-Judge 评估忠实度；(5) 人工抽检。

### Q: Chain of Verification (CoVe) 是什么？
**来源:** LLMInterviewQuestions
**答案要点:** 生成答案后：(1) 提取可验证的事实陈述；(2) 对每个陈述生成验证问题；(3) 独立回答验证问题；(4) 对比原始陈述和验证结果，修正不一致的部分。

## 十、生产部署与 MLOps

### Q: LLMOps 和传统 MLOps 的区别？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) LLM 通常通过 API 调用（非自训练模型）；(2) 评估更多依赖 LLM-as-a-Judge 而非传统指标；(3) Prompt 版本管理成为核心（类似代码）；(4) 成本管理是核心工程挑战（按 Token 计费）；(5) 幻觉和安全性是独特问题。

### Q: Prompt Caching 如何工作？
**来源:** ai-engineering-interview-questions
**答案要点:** 缓存长 System Prompt 和重复前缀的计算结果（KV Cache），避免每次请求都重新计算。对于对话应用（重复使用 System Prompt + 历史消息），可大幅降低 Prefill 延迟和成本（最高节省 90%）。

### Q: AI Gateway 是什么？
**来源:** ai-engineering-interview-questions
**答案要点:** 统一的 LLM API 入口层，负责：(1) 多 Provider 路由和自动故障切换；(2) 认证和速率限制；(3) 成本追踪和 Token 统计；(4) 请求日志和审计；(5) Prompt 安全检测。

### Q: LLM 路由 (Semantic Routing) 如何实现？
**来源:** ai-engineering-interview-questions
**答案要点:** 基于查询复杂度、成本预算、延迟要求动态选择模型。实现：(1) 分类器判断查询类型（简单/复杂）；(2) 成本最优路由（预算内优先大模型）；(3) 级联路由（小模型先试，低置信度升级到大模型）。

### Q: A/B 测试在 LLM 应用中如何实施？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) Prompt A/B：不同 Prompt 配方对比；(2) Model A/B：不同模型（GPT-4 vs Claude）；(3) Architecture A/B：不同系统架构（如纯 LLM vs RAG）。需要定义清晰的 Success Metrics、合理分流、统计显著性检验。

### Q: 如何实现 LLM 输出格式的可靠性（Structured Output）？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 使用 OpenAI Structured Outputs / JSON Mode；(2) 在 Prompt 中明确格式要求；(3) 后处理解析 + 验证（Pydantic/Zod）；(4) Retry 机制处理解析失败；(5) 使用 constrained generation 确保合法输出。

### Q: 如何处理生产环境中 LLM 的 Rate Limiting？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 客户端实现重试+指数退避；(2) 排队机制（先进先出/优先级）；(3) 多 Provider 负载均衡；(4) 缓存减少 API 调用；(5) 降级策略（高峰期切换到备用模型）。

### Q: AI 系统如何实现 Graceful Degradation？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 主模型不可用时 Fallback 到备用模型；(2) 复杂功能不可用时保留基础功能；(3) 静态缓存答案作为最后防线；(4) 错误时给用户明确反馈而非静默失败。

### Q: 生产环境 Prompt 如何版本管理？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) Git 版本控制 Prompt 模板文件；(2) 每次变更附带 A/B 测试结果；(3) 关联 Chaangelog 记录变更原因；(4) 支持一键回滚到历史版本；(5) 与 Eval Suite 绑定，变更自动跑回归测试。

### Q: 如何实现 LLM 申请求的 Logging 和 Tracing？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 记录每步 LLM 调用（输入/输出/延迟/Token 消耗）；(2) Agent 场景记录完整 Trace（Thought-Action-Observation 链）；(3) 使用 LangSmith/Langfuse/Arize 等平台；(4) 关联用户反馈形成数据闭环。

### Q: LLM 推理性能关键 SLA 指标？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) TTFT (Time to First Token)：Prefill + 首 Token Decode 延迟；(2) TPOT (Time Per Output Token)：生成阶段每 Token 耗时；(3) Throughput：每秒处理 Token 数/请求数；(4) P50/P95/P99 延迟；(5) Availability 可用率。

### Q: 如何估算生产环境中 AI Feature 的成本？
**来源:** ai-engineering-interview-questions
**答案要点:** 成本 = 请求量 × (输入 Token × 输入单价 + 输出 Token × 输出单价) + Embedding/检索成本 + 基础设施成本。Token 数估算：用户量 × 日均请求 × 平均 Token/请求。RAG 场景还需考虑 Embedding 和向量数据库成本。

## 十一、安全与伦理

### Q: AI Alignment 是什么？为什么重要？
**来源:** ai-engineering-interview-questions
**答案要点:** Alignment 是确保 AI 系统的行为与人类价值观、意图和偏好一致的过程。包括：(1) Helpfulness：有用但不越界；(2) Harmlessness：不产生有害输出；(3) Honesty：准确表达不确定性。技术上主要通过 RLHF/DPO 实现。

### Q: Hallucination 如何各层级控制？
**来源:** LLMInterviewQuestions, Awesome-LLM, ai-engineering-interview-questions
**答案要点:** (1) Prompt 层：指令明确（"如不确定请说不知道"）、CoT、引用要求；(2) 数据层：RAG 提供事实支撑；(3) 模型层：Fine-tuning/RLAIF 惩罚幻觉；(4) 输出层：事实核查、NLI 验证、置信度评估；(5) 系统层：Human-in-the-loop。

### Q: 大模型在实际业务中难以落地的主要原因？
**来源:** Awesome-LLM
**答案要点:** (1) 幻觉问题导致不可靠；(2) 成本控制困难；(3) 数据安全和隐私合规；(4) 延迟不满足实时要求；(5) 缺乏有效评估体系；(6) 领域知识不足；(7) 结果不可解释。

### Q: GDPR/CCPA 对 AI 应用的影响？如何合规？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 用户数据最小化：只收集必要数据；(2) Right to be Forgotten：从训练数据和索引中删除用户数据；(3) 数据处理的合法基础：明确告知用户；(4) 跨境数据传输限制；(5) 自动化决策的解释权。

### Q: EU AI Act 对 AI 工程的影响？
**来源:** ai-engineering-interview-questions
**答案要点:** 将 AI 系统分为不可接受风险、高风险、有限风险和最小风险四级。高风险系统需要：(1) 风险管理体系；(2) 数据治理；(3) 技术文档；(4) 透明度；(5) 人工监督。违反可能面临全球收入 7% 的罚款。

### Q: Bias 在 AI 系统中如何检测和缓解？
**来源:** ai-engineering-interview-questions
**答案要点:** 检测：(1) 分群体评估性能差异；(2) 代理特征检测（间接反映受保护属性的特征）；(3) Intersectional 分析（交叉群体）。缓解：(1) 训练数据平衡；(2) Counterfactual 数据增强；(3) RLHF 偏好引导；(4) 后处理校准。

### Q: 什么是 Data Poisoning？如何防御？
**来源:** ai-engineering-interview-questions
**答案要点:** 攻击者在训练数据中注入恶意样本影响模型行为。防御：(1) 数据来源验证；(2) 异常检测过滤异常样本；(3) 差分隐私训练；(4) 模型行为持续监控。

### Q: 模型卡 (Model Card) 是什么？包含哪些内容？
**来源:** ai-engineering-interview-questions
**答案要点:** 标准化模型文档，包含：(1) 模型用途和限制；(2) 训练数据和评估结果；(3) 已知 Bias 和风险；(4) 性能在不同子群体上的分布；(5) 伦理考量。类似 AI 模型的"营养标签"。

## 十二、向量数据库与检索

### Q: Vector Database 和传统数据库的区别？
**来源:** ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** 传统数据库：基于精确匹配（SQL/B-tree索引）；向量数据库：基于相似度搜索（ANN 近似最近邻），存储和查询高维 Embedding 向量。核心操作是：给定查询向量，找 Top-K 最相似向量。

### Q: ANN (Approximate Nearest Neighbor) 搜索算法有哪些？
**来源:** LLMInterviewQuestions, ai-engineering-interview-questions
**答案要点:** (1) HNSW (Hierarchical Navigable Small World)：多层级图结构，查询快但内存占用高；(2) IVF (Inverted File Index)：聚类+倒排，适合大规模场景；(3) PQ (Product Quantization)：对向量进行压缩编码；(4) LSH (Locality-Sensitive Hashing)：哈希碰撞保留相似性。

### Q: Cosine Similarity vs Dot Product vs Euclidean Distance 如何选择？
**来源:** ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** Cosine：适合比较方向（归一化后语义相似度）；Dot Product：当向量已归一化时等价于 Cosine，计算更快；Euclidean：适合比较绝对位置/大小。LLM Embedding 常用 Cosine Similarity。

### Q: FAISS, Chroma, Pinecone, Milvus, Weaviate 如何选择？
**来源:** AI_Engineer_Interview_Prep
**答案要点:** FAISS：原型/研究阶段，单机内存；Chroma：小中型应用，嵌入式；Pinecone：全托管，零运维；Milvus：自托管企业级，十亿级向量；Weaviate：多模态+GraphQL 风格的复杂过滤。选择标准：规模、是否需要托管、过滤复杂度、预算。

### Q: Embedding 维度如何影响性能和成本？
**来源:** ai-engineering-interview-questions
**答案要点:** 更高维度 → 更高表达能力但更多存储和更慢搜索。1024-1536 维是性价比甜点。text-embedding-3 支持 Matryoshka 降维（如 3072→256），几乎不损失质量。

### Q: 如何处理 Embedding Drift（模型更新导致向量空间变化）？
**来源:** AI_Engineer_Interview_Prep, ai-engineering-interview-questions
**答案要点:** 切换 Embedding 模型时需要重新 Embedding 全部文档（新旧向量空间不可比）。策略：(1) 影子部署，逐步迁移；(2) 保留旧向量直到全量更新完成；(3) 双写+逐步切换流量。

### Q: 如何 Fine-tune Embedding 模型？
**来源:** LLMInterviewQuestions, ai-engineering-interview-questions
**答案要点:** (1) 收集领域 (Query, Positive Doc, Negative Doc) 三元组；(2) 使用对比学习 Loss (InfoNCE/Triplet Loss)；(3) 使用 Sentence Transformers 框架；(4) 在自有 Benchmark 上评估改进。

### Q: Multi-Tenant 隔离在向量数据库中如何实现？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) Metadata 过滤（每向量标注 tenant_id）；(2) 独立 Collection/Index 隔离；(3) 独立数据库实例（最强隔离）；(4) 多级分区 (Partition Key)。Metadata 过滤最常用但需注意过滤延迟。

### Q: Multi-modal Embedding 如何生成？
**来源:** ai-engineering-interview-questions, LLMInterviewQuestions
**答案要点:** CLIP 类模型将文本和图像映射到同一向量空间，使用 Contrastive Loss 训练 (文本-图像对)。2026 年标准包括多模态大模型 (GPT-4o/Claude 3.5) 的统一 Embedding。

## 十三、系统设计

### Q: 如何设计 ChatGPT？（端到端：训练到推理）
**来源:** ai-engineering-interview-questions, AI_Engineer_Interview_Prep
**答案要点:** (1) 预训练：多语言语料 + Causal LM 目标；(2) SFT：高质量指令对话数据；(3) RLHF/DPO：偏好对齐；(4) 推理服务：vLLM/SGLang + Paged Attention + Continuous Batching；(5) 安全层：输入/输出内容过滤；(6) 多区域部署+负载均衡。

### Q: 如何设计一个 RAG 系统（Chat with Your Documents）？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 离线 Pipeline：文档解析→Chunking→Embedding→向量数据库；(2) 在线 Pipeline：查询变换→检索→Rerank→Context 组装→LLM 生成；(3) 额外考量：访问控制/多租户、缓存策略、引用追踪、增量索引更新、评估监控。

### Q: 如何设计一个 Multi-Agent Customer Support 系统？
**来源:** ai-engineering-interview-questions, AI_Engineer_Interview_Prep
**答案要点:** (1) Triage Agent：分类用户意图（退款/技术支持/账单等）；(2) Specialist Agent 群：每个领域独立的 Agent；(3) Escalation Logic：置信度低→Human Agent；(4) Shared Context：用户画像+历史工单；(5) Supervisor Agent：协调+汇总。

### Q: 如何设计一个 LLM Inference Platform（vLLM-as-a-Service）？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 推理引擎：vLLM/SGLang；(2) API 网关：认证、速率限制、路由；(3) 模型管理：模型仓库+版本管理+Lazy Loading；(4) 调度器：基于负载的自动扩缩容；(5) 缓存层：Prompt Cache + Semantic Cache + KV Cache offload；(6) 监控：TTFT/TPOT/GPU 利用率/Token 消耗。

### Q: 如何设计一个 Deep Research Agent？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 查询理解：分解复杂问题为子问题；(2) 搜索编排：并行搜索+迭代深入；(3) 信息提取：解析搜索结果，提取关键事实；(4) 交叉验证：多个来源验证一致性；(5) 综合写作：结构化报告生成（含引用）；(6) 反思迭代：自我审查+改进。

### Q: 如何设计 Memory for Personal AI Assistant？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 短期记忆：滑动窗口缓存最近对话；(2) 长期记忆：Vector DB 存储关键信息+定期检索；(3) Episodic Memory：事件摘要（时间线）；(4) 语义记忆：用户画像+偏好+关系图；(5) 记忆管理：遗忘机制+优先级+冲突解决。

### Q: 如何设计一个 AI Coding Agent？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 代码库理解：Repo-level 索引（AST + Embedding）；(2) 工具集：文件读写、Shell 执行、Git 操作、LSP；(3) Agent Loop：理解需求→搜索代码→分析→生成→测试→修复；(4) 安全：沙箱执行、权限最小化、人工审批关键操作；(5) Context 管理：只加载相关文件（非全仓库）。

### Q: 设计时如何权衡 Latency vs Quality？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) Model Routing：简单任务小模型（低延迟），复杂任务大模型（高质量）；(2) 级联策略：快速模型先产出，大模型异步精炼；(3) 缓存策略：常见查询直接返回缓存；(4) 流式输出：降低感知延迟。

### Q: 如何设计 Multi-Tenant AI Chatbot 平台？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) Tenant 隔离：独立 Knowledge Base + Embedding 空间（Metadata 过滤）；(2) 定制化：每 Tenant 独立 System Prompt + 品牌配置；(3) 计费：按 Token/请求/活跃用户量维度；(4) 速率限制：Per-tenant Quota；(5) 安全：Prompt Injection 防御 + 数据隔离。

### Q: Multi-Agent vs Supervisor vs Orchestration 架构何时使用？
**来源:** AI_Engineer_Interview_Prep
**答案要点:** Simple Chain → 无需框架；RAG → LlamaIndex；复杂多步骤工作流（含循环） → LangGraph；多角色协作 → CrewAI；需要快速原型 + 多集成 → LangChain。

## 十四、多模态 AI

### Q: Vision-Language Model (VLM) 如何工作？
**来源:** ai-engineering-interview-questions
**答案要点:** 将图像编码为 Patch Embedding（类似 ViT），与文本 Tokens 在统一序列中处理。2026 年标准模型（GPT-4o/Claude 3.5）使用 Native Omni-modality：文本、图像、音频经同一 Transformer 处理。

### Q: Multi-modal RAG 和 Text-only RAG 的区别？
**来源:** ai-engineering-interview-questions
**答案要点:** Multi-modal RAG 需处理图像、表格、图表等多模态文档。挑战：(1) 不同模态的 Chunking 策略；(2) Multi-modal Embedding 对齐；(3) 跨模态检索（用文本查询图片）；(4) 检索结果的跨模态融合。

### Q: Diffusion Models 如何工作？（Stable Diffusion / DALL-E）
**来源:** ai-engineering-interview-questions
**答案要点:** 前向过程逐步向图像加噪→纯噪声；反向过程训练 U-Net/DiT 逐步去噪恢复图像。文本条件通过 Cross-Attention 注入。Stable Diffusion 在 Latent Space 中进行以降低计算成本。

### Q: 如何评估 Multi-modal AI 系统？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 图像描述准确度（CIDEr/CLIPScore）；(2) VQA (Visual Question Answering) 准确率；(3) 幻觉检测（描述中不存在于图像的内容）；(4) 人类评估（自然度、帮助性）。

## 十五、技能实现题 (Coding & Practical)

### Q: 实现一个基础 RAG Pipeline
**来源:** ai-engineering-interview-questions
**答案要点:** 使用 Embedding Model + FAISS/Chroma 向量存储 + LLM API。流程：文档→Chunk→Embed→Index→Query→Retrieve→Augment→Generate→Return。

### Q: 实现一个 Simple AI Agent with Tool Use
**来源:** ai-engineering-interview-questions, AI_Engineer_Interview_Prep
**答案要点:** ReAct Loop: LLM 输出 Thought/Action → 执行 Tool → 返回 Observation → 循环。实现 Tool Schema 定义、Tool 调度器、结果组装。

### Q: 实现 Semantic Search
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 文档 Embedding 预计算；(2) 查询 Embedding 实时计算；(3) Cosine Similarity 相似度排序；(4) 返回 Top-K。

### Q: 实现多种 Chunking 策略
**来源:** ai-engineering-interview-questions
**答案要点:** Fixed-size: `text[i:i+size]`；Recursive: 段落→句子递归分割；Semantic: 基于 Embedding 相似度突变检测边界。

### Q: 实现 LLM API 调用的 Retry + Exponential Backoff
**来源:** ai-engineering-interview-questions
**答案要点:** 遇到 Rate Limit/Server Error → `sleep(2^n)` → 重试 → 最大重试次数 → Fallback 模型。

### Q: 实现 Semantic Caching for LLM
**来源:** ai-engineering-interview-questions
**答案要点:** 将查询 Embedding → Vector DB 查相似查询 → 相似度阈值判断 → 命中则返回缓存，否则调用 LLM 并缓存结果。

### Q: 实现 Conversation Memory 系统
**来源:** ai-engineering-interview-questions
**答案要点:** Buffer: 保留最近 N 条消息；Summary: 定期摘要历史→重置 Buffer；Vector: 关键信息存入 Vector DB 按需检索。

### Q: 实现一个 Token Counting 和 Context Window 管理模块
**来源:** ai-engineering-interview-questions
**答案要点:** 使用 tiktoken 计算 Token 数；检测是否接近限制；触发 Truncation/Compaction/Summarization；警告日志。

### Q: 实现 Prompt Injection 检测
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 检测分隔符重写 (Ignore previous instructions)；(2) 检测指令模仿 (You are now DAN...)；(3) Embedding 相似度检测（异常 Prompt 与正常 Prompt 差异大）；(4) 输出对比（有无注入的差异）。

## 十六、行为面试 & 项目经验

### Q: 数据获取：训练数据集和测试数据集如何获取？
**来源:** Awesome-LLM
**答案要点:** 训练集：公开数据集 + 自有数据爬取/购买 + 人工标注 + 合成数据生成。测试集需与训练集严格隔离（时间/来源分离），覆盖线上真实分布。

### Q: 项目中出现过拟合怎么办？
**来源:** Awesome-LLM
**答案要点:** 诊断：Train Loss 持续下降但 Val Loss 上升。解决：(1) 减少训练 Epochs/Steps；(2) 增加 Dropout/Weight Decay；(3) 数据增强；(4) Early Stopping；(5) 使用 LoRA 的更低 Rank。

### Q: 项目中如何增强模型的泛化能力？
**来源:** Awesome-LLM
**答案要点:** (1) 多样化训练数据（多来源/多风格）；(2) 数据增强（同义改写/回译）；(3) 正则化技术；(4) 对抗训练；(5) 多任务学习；(6) 课程学习（从易到难）。

### Q: 如何构建评测体系和设计评测指标？
**来源:** Awesome-LLM
**答案要点:** 多维度评测矩阵：(1) 准确性（Exact Match/F1/LLM Judge）；(2) 安全性；(3) 延迟/成本；(4) 用户满意度。分层：Unit Test (Prompt 级别) → Integration Test (Pipeline 级别) → E2E Test (用户场景级别)。

### Q: AI 工程和传统 ML 工程的区别？
**来源:** ai-engineering-interview-questions
**答案要点:** AI 工程更侧重于：(1) 使用 API/Foundation Model 而非从零训练；(2) Prompt Engineering 是核心技能；(3) 系统集成优先于模型开发；(4) 幻觉/安全是独特挑战；(5) 评估方法论不同（LLM-as-Judge）。

### Q: 如何决定一个问题是该用 AI 还是传统软件方案？
**来源:** ai-engineering-interview-questions
**答案要点:** AI 适合：(1) 非确定性任务（语义理解/开放式文本生成）；(2) 模式识别但规则难写；(3) 需要领域知识泛化。传统软件适合：(1) 确定性规则明确；(2) 精确计算；(3) 低延迟/低成本要求。

### Q: 如何衡量 AI Feature 的 ROI？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 成本节省（自动化替代人工）；(2) 收入增长（转化率/用户留存提升）；(3) 用户体验改善（NPS/满意度）；(4) 直接成本（Token/API/Infra）计算盈亏平衡点。

### Q: 如何向非技术 Stakeholder 解释 LLM 的局限性？
**来源:** AI_Engineer_Interview_Prep
**答案要点:** "LLM 不是数据库——它不存储事实，而是模式。它可能会以非常自信的语气输出错误信息（幻觉）。100% 准确率是不可能的，因为模型本质上是概率性的。我们的工程工作就是在模型之上建立验证层来提升可靠性。"

### Q: Demo 效果好但生产失败的原因？（Demo vs Production Gap）
**来源:** AI_Engineer_Interview_Prep
**答案要点:** (1) Demo 用精心挑选的 Case，生产有长尾分布；(2) Demo 忽略错误处理，生产需要 Graceful Degradation；(3) Demo 不考虑并发/延迟/成本，生产有硬约束；(4) Demo 数据是静态的，生产数据持续变化。

### Q: 如何处理生产环境中的 Hallucination？
**来源:** ai-engineering-interview-questions
**答案要点:** (1) 立即 Log 案例并分析根因（检索失败/生成失败/Prompt 问题）；(2) 加入 Eval Set 防止回归；(3) 实施针对性修复；(4) 短期方案：该 Case 类型启用更严格验证；(5) 长期：改进 RAG/Fine-tune/Guardrails。
