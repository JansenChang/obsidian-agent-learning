# Day2 早间版 源文稿

## Part 1 (~5950字)

你有没有想过，当你在聊天框里输入"帮我写一封请假邮件"，按下回车之后，模型内部到底发生了什么？它怎么从一个字一个字的输入，变成一整段通顺的回复的？

今天我们就把这最后一步彻底讲清楚。之前我们聊过 Transformer 怎么用 Attention 算出每个词和其他词的关系，聊过 FlashAttention 怎么让这些计算跑得更快。但最关键的一步还没讲——模型算出了每个候选词的分数，接下来该怎么选？

这就是 Decoding 策略管的事。

我们先从最根本的问题开始。模型每一轮推理结束后，会得到一个词表大小的概率分布。什么是概率分布呢？就是词表里大概五万个词，每个词都有一个概率值，表示"下一个词是它的可能性有多大"。比如你刚说了"我今天的午餐是"，模型可能认为"拉面"的概率是百分之三十五，"汉堡"是百分之二十五，"三明治"是百分之十五，"沙拉"是百分之十，剩下的四万九千九百多个词分享最后那百分之十五的概率。

注意，这五万个词里不只有正常的词，还有各种奇怪的东西。有数学符号，有日文汉字，有特殊标记，甚至有一些训练数据中出现的生僻字符。模型不会区分"这是正常词还是怪词"，它一视同仁地给每个词分了一个概率。有的概率大到百分之三十几，有的小到百万分之一。

好了，现在问题来了。我们拿到了这个分布，到底选哪个？

最简单的做法就是：每次都选概率最高的那个。这叫贪婪解码，英文叫 Greedy Decoding。它就像一个有强迫症的人走路，永远只走最宽的大马路，从来不拐进小巷子。你说"我今天的午餐是"，它永远选"拉面"；你说"帮我写封邮件"，它永远从"尊敬的"开始写。

贪婪解码的优点很明显：简单、快、确定。同样的输入永远得到同样的输出。但缺点也很致命：单调、重复、没有惊喜。你会得到一堆平平无奇的回复。更糟的是，它容易陷入重复循环。模型选了"很开心"，因为"很开心"概率最高，下一个又选"很开心"，再下一个还选"很开心"，于是输出变成了"我今天很开心，真的很开心，非常开心，特别开心……"。

这就是贪婪解码的核心问题——每一步都选最好的，但整体看不一定最好。

于是就有了温度采样的概念。温度，英文叫 Temperature，写作 T。这个参数不是深度学习发明的，它来自统计物理学中的玻尔兹曼分布。在物理里，温度控制着粒子处于高能态的概率。温度越高，粒子越活跃，越有可能跳到高能态。温度越低，粒子越稳定，集体往低能态挤。

在语言模型里，温度干的是同一件事。它控制着概率分布的陡峭程度。温度低的时候，概率分布变得很尖很陡，高的更高，低的更低。温度高的时候，概率分布被压平了，高的和低的差距变小了。

具体来说，温度等于零点一的时候，概率分布几乎变成了一个独热向量——概率最高的词被极度放大，其他词几乎为零。这跟贪婪解码差不多，只是偶尔不那么固执。温度等于零点七的时候，分布保留了一定的区分度，同时给低概率词留了一些机会。温度等于一的时候，就是原始的 softmax 输出，不做任何改变。温度等于二的时候，分布几乎被压平了，所有候选词的概率趋近于相等。

你可以把温度想象成一个旋钮。向左拧到底，模型变得刻板，永远选最安全的词。向右拧到底，模型变得疯狂，什么胡话都敢说。中间的位置，零点七左右，是大部分对话模型使用的默认值，因为它给了模型适度的创造力，又不会让它跑题。

现在用我们刚才的午餐例子来走一遍。输入"我今天的午餐是"，模型的概率分布显示"拉面"百分之三十五，"汉堡"百分之二十五，"三明治"百分之十五，"沙拉"百分之十。温度等于零点二的时候，概率分布变成"拉面"百分之七十一，"汉堡"百分之十八，"三明治"百分之七，几乎是锁死选"拉面"了。温度等于零点八的时候，概率分布相对自然，"拉面"还是最高，但"饺子""披萨"也有了一点点机会。温度等于一点五的时候，概率分布被拉得很平，原来百分之三十五的"拉面"降到只有百分之二十二，而原来百分之四的"披萨"涨到了百分之八。这意味着模型有可能在你说"我今天的午餐是"之后，接一个"披萨"出来——不是因为它更合理，而是因为温度高了，它更愿意冒险。

但是，光调温度还不够。别忘了词表里有五万个词，其中一大堆是几乎不可能出现的怪词。温度调高之后，这些怪词的概率也会跟着涨。本来概率百万分之一的"饕餮"，温度一点五之后可能涨到千分之一，十万次生成里就会出现一次。这在正式场景中是完全不能接受的。

所以我们需要一个工具，把那些明显不合理的词提前排除掉。这就是 Top-K 采样的作用。

Top-K，顾名思义，就是只保留概率最高的前 K 个词。K 就是数量。K 等于四十，就是只取前四十个词，剩下的四万九千九百六十个全部归零，然后在保留的四十个里重新算概率。K 等于一百，就是保留前一百个。

这样做的效果很直接——不管温度怎么调，那些尾部的怪词从一开始就被踢出候选名单了。你永远不会看到模型在一个正常的对话里输出一个数学符号，因为它在 Top-K 那一步就被砍掉了。

但是 Top-K 有一个隐藏的问题。K 是固定的，但概率分布的形态是动态的。有些时候，分布非常集中，前面三四个词就占了百分之九十的概率。这时候 K 等于四十意味着剩下的三十六个候选都是概率极低的噪声词，保留它们等于保留了三十六个陷阱。有些时候，分布非常分散，前面二十个词每个都只有百分之三到五的概率。这时候 K 等于四十又会把后面一些本来合理的候选砍掉了。

一句话说就是——分布集中的时候 K 太大了，分布分散的时候 K 又太小了。

为了解决这个问题，二零一九年，Holtzman 和他的团队提出了 Top-P 采样，也叫核采样。它的做法不是固定保留多少个词，而是从概率最高的开始累加，一直加到累积概率达到 P 为止。

P 就是 Probability，概率。P 等于零点九，就是从"拉面"开始加，加上"汉堡"，加上"三明治"，一直加到累积概率大于等于百分之九十为止。这时候候选词的数量不是固定的——分布集中的时候可能只保留两三个，分布分散的时候可能保留十五二十个。

还是用午餐的例子。P 等于零点九，概率从高到低累加。"拉面"百分之三十五，累积三十五。"汉堡"百分之二十五，累积六十。"三明治"百分之十五，累积七十五。"沙拉"百分之十，累积八十五。"饺子"百分之八，累积九十三，超过九十了，停。保留了五个词。如果是 Top-K 等于四十，分布这么集中的时候，另外三十五个全是噪声。Top-P 就聪明多了，只留五个核心候选，干净利落。

实际生产中，Top-K 和 Top-P 通常是一起用的。先做 Top-K，一刀砍掉尾部的极端噪声词。再做 Top-P，动态剪裁候选集。两者互补，K 管下限，P 管上限。

执行顺序是这样的：先 Top-K 砍掉尾部噪声，再 Top-P 动态剪裁候选，然后 Temperature 调节分布形态，最后根据新的概率分布采样选一个词。这四步合在一起，就是当前几乎所有主流语言模型在实际推理时使用的解码流程。

除了上面这三种采样策略，还有一种完全不同的思路，叫束搜索，英文 Beam Search。

前面讲的 Greedy、Temperature、Top-K、Top-P，它们每一步都是独立的，只看当前步选什么。Beam Search 不是这样，它同时维护多条候选路径，每一步都从所有候选路径中扩展，然后整体排序，只保留概率最高的前 N 条。

这解决了贪婪解码的一个根本问题：局部最优不等于全局最优。每一步选概率最高的词，但可能这条路后面的路越走越窄。Beam Search 让模型同时走 N 条路，最后选整体概率最高的那一条。

但 Beam Search 也有明显的代价：慢。保留 N 条路径，计算量就是 N 倍。而且生成的文本越长，越容易陷入重复。所以对话模型几乎不用它，它主要用在机器翻译、语音识别、文本摘要这种场景，因为那些场景需要全局最优的序列，而不是逐词最优。

现在我们来看各大模型实际用的是什么配置。ChatGPT 的默认配置是温度零点七，Top-P 零点九，Top-K 四十。Claude 的默认温度偏低，大约零点五到零点七，反映了 Anthropic 倾向于安全可控的设计理念。DeepSeek 的默认配置跟 ChatGPT 类似，温度零点七，Top-P 零点九，但多了 Thinking Mode 下的特殊解码逻辑。Gemini 的默认温度在零点四到零点六之间，更偏向事实性任务。代码专用模型通常用更低的温度，零点二左右，同时保留 Top-P 零点九，因为代码需要确定性，不能自由发挥。

ChatGPT 还有两个独有的参数，叫 frequency_penalty 和 presence_penalty。它们不是标准的解码策略，是 OpenAI 在采样之后加的一道后处理。frequency_penalty 根据一个词在已生成文本中出现的频率来衰减它的概率，出现越多扣分越多。presence_penalty 更简单，只要这个词出现过就扣分，不管出现了几次。这两个参数的作用是抑制重复。

那问题来了——如果把 DeepSeek 的温度从零点七调到零点二，它的代码生成能力会变强吗？答案是不会。Decoding 参数能影响的只是输出的风格和多样性，不是模型的能力上限。Claude 的代码生成比 DeepSeek 好，是因为 Claude 在训练阶段见过的代码数据更多、质量更高，在强化学习阶段"写出正确代码"被奖励得更多。这不是调几个参数能弥补的。模型能力是训练决定的，解码策略只是把模型训练时学会的东西更好地表达出来。

好了，现在我们来做一个完整的串讲，把之前学过的知识跟今天的 Decoding 策略连接起来。

你输入"帮我找一下第一性原理"之后，数据是这样流动的。第一步，Tokenization——模型把每个字转换成数字编号，我们明天会专门讲这一步。第二步，Embedding——查表把每个 token 转换成向量。第三步，RoPE 旋转位置编码——告诉模型每个字在句子中的位置顺序。第四步，Transformer 层，这是核心。在每一层里，Attention 先计算每个字和其他字的关系，生成上下文相关的表示。然后 FFN 前馈网络从训练时学会的知识中提取语义——"第一性原理"这个词的含义就是在这里被检索出来的，FFN 的权重里存着这个词大致等于"回归事物本质"这个映射。第五步，经过所有 Transformer 层之后，最后的隐藏状态被送入 LM Head，一个线性层，把它映射到词表维度，每个词得到一个分数。第六步，Softmax 把这些分数转成概率。第七步，Decoding 策略开始工作——先 Top-K 砍掉尾部噪声，再 Top-P 动态剪裁候选，接着 Temperature 调节分布形状，最后采样选出一个词。第八步，这个词被输出，同时它的 Key 和 Value 被存入 KV Cache，方便后续的 token 复用前面已经算好的注意力信息。

你看，从 RoPE 到 Attention 到 FFN 到 Decoding，每个环节各司其职。RoPE 负责位置感知，Attention 负责上下文融合，FFN 负责知识提取，Decoding 负责策略选择。任何一个环节出了问题，最终输出都会受影响。但 Decoding 能弥补的范围是最小的——它只能在上游已经算好的概率分布上做选择，不能改变这个分布本身。

有一个很多人问的问题，就是解码策略跟贝叶斯模型有没有关系。直接回答是：技术上没有直接关系。贝叶斯模型的核心是"先验概率乘以似然函数得到后验概率"，是一个不断更新的概率框架。而语言模型的解码策略，是从已经一次性算好的概率分布中选一个词，没有更新过程。两种完全不同的技术路线。不过思想上有一些有趣的对应关系。最大后验估计 MAP 对应贪婪解码，因为它们都取概率最高的那个点。后验采样对应 Top-P 加温度的采样方式，因为都是从完整分布中抽取。贝叶斯模型平均对应束搜索，因为都是维护多条路径然后综合。所以如果你被问到，可以说它们在思想层面有一些类比关系，但技术实现上完全不同。

还有一个实用的话题——怎么把这些策略用到你自己的项目里。假设你在做一个 AI 客服，用户的问题比较固定，比如查订单、改地址、问退换货。这时候你应该用偏低的温度，零点三左右，因为客服不能自由发挥。Top-P 保持在零点九，K 可以设到五十左右。如果用户反馈回答太死板，把温度往上调一点，从零点三提到零点五。如果用户反馈出现跑题，就把温度降回去。这个调试过程，本质上跟调一个物理旋钮是一样的——每次转一点点，听听效果，再决定往哪个方向转。

如果你在做一个代码生成工具，温度应该更低，零点二左右。因为代码是精确的，高温度会编造不存在的变量名和函数。如果你在做一个创意写作助手，比如写文案或者写歌词，温度可以调到零点八，同时把 Top-P 放开到零点九五，让模型有更大的选择空间。如果你在做翻译，用接近零的温度甚至直接用贪婪解码，因为翻译的核心是忠于原文。

还有一个细节经常被忽略——K 和 P 的默认值不是拍脑袋定的，是经过大量实验验证的。K 等于四十到五十，是因为大部分场景下，概率分布的有效头部大致在这个范围内。P 等于零点九，是因为百分之九十的概率已经能覆盖几乎所有合理候选，同时不会引入太多噪声。如果你的场景比较特殊，比如你需要模型只从非常有限的词里面选，比如选择题的四个选项，那就应该把 K 设成四，P 设成零点九九，确保模型不会到选项之外找词。

## Part 2 (~3070字)

总结一下今天学到的所有策略，我用一个统一的类比帮你串起来。想象你是一家餐厅的老板，每天要决定菜单上写什么菜。词表里的五万个词就是所有你能想到的菜名。

贪婪解码就像你每天只卖卖得最好的那道菜。简单、安全，但客人很快就会吃腻。

温度就是这个"冒险"的程度。温度低的时候你只卖最稳的招牌菜。温度高的时候你会试试新菜式，有时候客人很惊喜，有时候会翻车。

Top-K 是你规定只看销量排行榜前四十的菜，后面的菜不管多新颖都没资格上菜单。这样不会出现奇怪的菜，但也可能错过了潜力股。

Top-P 是你从销量最高的菜开始看，一直看到累积占了百分之九十销量的那些菜为止。如果前三道菜就占了百分之九十，后面的就不用看了。如果前二十道菜才占百分之九十，那就看二十道。动态调整，比 Top-K 灵活。

束搜索是你同时让三个厨师各试一条不同的菜单路线，最后选整体卖得最好的那条。

实际餐厅里这些策略是组合用的。先看销量前五十的菜，再从中选累积百分之九十的那些，再根据今天的冒险程度做最终决定。

现在回到模型本身。解码参数在你的项目中到底该怎么调？我给你几条可操作的规则。

第一条规则。安全默认值：温度零点七，Top-P 零点九，Top-K 四十。这是 ChatGPT 的默认配置，覆盖绝大多数场景。你不知道怎么设的时候，先用这个。

第二条规则。发现输出太死板太重复，往上调温度。从零点七调到零点八，零点九。每次调零点一，听听效果。如果调到一点二还是不行，说明问题不在温度，可能是模型本身的能力瓶颈。

第三条规则。发现输出跑题、出现怪词，往下调温度，或者收窄 Top-P。从零点九调到零点八五。如果还出现怪词，加 Top-K 四十甚至二十。K 越小越安全。

第四条规则。代码和数学场景用低温，零点二到零点三。创意场景用高温，零点八到零点九五。翻译和摘要用极低温，零点一或者直接贪婪解码。

第五条规则。如果你的输出要的只是一个选择题的答案，比如 A B C D 四个选项，把 Top-K 设成四，P 设成零点九九，确保模型不会跑到选项之外去找词。

面试中考频最高的几个问题我帮你总结一下。第一个问题，贪婪解码有什么问题。答案是缺乏多样性、容易重复、陷入局部最优。第二个问题，温度和 Top-P 分别控制什么。温度控制概率分布的陡峭程度，Top-P 控制候选词的动态截断。第三个问题，Top-K 和 Top-P 的区别。K 是固定数量，P 是动态累积概率。分布集中的时候 K 太大引入噪声，分布分散的时候 K 太小砍掉合理候选。P 自动适配，更灵活。第四个问题，实际生产中怎么组合。先 K 后 P 再温度最后采样。第五个问题，Decoding 参数能弥补模型能力的差距吗。不能，参数决定的是表达方式，模型能力是训练决定的。

还有一个经常被问到的变体题。面试官会问你，如果用户说模型总是在重复同一个句式，你会怎么调。这时候你应该考虑的不是温度，而是频率惩罚参数 frequency_penalty 或者出现惩罚 presence_penalty。如果你的框架支持这两个参数，把频率惩罚设到零点一或零点二，模型就会自动避免重复使用已经出现过的词。如果不支持这两个参数，那就升高温度，同时加大 Top-P，让模型有更多候选词可选，自然就不容易重复了。

如果用户说模型输出非常平淡没有亮点，但又不能太天马行空。这时候你只调温度是不够的，因为温度会把所有概率拉平，包括那些不合理的选择。更好的办法是保持温度不变，把 Top-P 从零点九放宽到零点九五，让候选集扩大一些。这样模型有更多选择，但失控的风险比直接升温要小得多。

如果用户说模型总是忘记上下文里提到的关键信息，比如你在一段长对话中说了你的名字，但模型后面还是用错误的称呼。这时候你要检查的不是解码参数，而是上下文长度和注意力机制的问题。解码策略再怎么调也救不了注意力分配的问题。

面试官还可能问你，你做过哪些解码参数的实验，结果如何。如果你没有真实经验，可以这样回答：你在对比不同温度对代码生成的影响时发现，温度从零点二升到零点五，代码的正确率从百分之九十五降到了百分之七十八，但生成的代码风格更多样了。这说明低温对代码生成确实有显著的正向作用。这种回答比背诵参数定义有力得多。

还有一个非常实用的知识点，就是如果你在调解码参数的时候没有明确的方向，有一个简单的调试方法。你把温度分别设成零、零点五、零点七、一，对同一个输入各生成一次输出。对比这四次输出，你就能直观感受到温度对模型的影响。温度为零的时候输出最保守，温度为一的时候输出最多样。然后根据你的业务需求，在这条线上找到最合适的点。

温度这个参数有一个很多人不知道的特性。温度为零的时候，模型实际上变成了确定性算法——同样的输入永远产生同样的输出。这在调试和测试阶段非常有用。如果你的应用需要可复现的结果，比如自动化测试或者合规审计，你就应该用零温度。如果你需要每次输出都不一样，比如聊天机器人，你就用非零温度。

Top-K 和 Top-P 的关系，有一个更直观的理解方式。Top-K 是硬限制，Top-P 是软限制。硬限制的意思是，候选数不会超过 K。软限制的意思是，候选数根据分布自动调整，但不会超过 P 允许的范围。所以先加硬限制再加减软限制，是最安全的做法。如果你只加软限制不加硬限制，极端情况下 Top-P 可能保留几百个候选，计算量会不必要地变大。

在实际的系统设计中，解码参数通常不会暴露给终端用户。用户的界面上可能只看到一个"创意程度"的滑动条或者"温度"的输入框。后端会把用户选择的"创意程度"映射到一组解码参数上。比如创意程度百分之二十对应温度零点三和 Top-P 点九，创意程度百分之五十对应温度零点六和 Top-P 零点九，创意程度百分之八十对应温度零点九和 Top-P 零点九五。这样用户不需要理解什么是 Top-P 什么是 Temperature，就能调出合适的输出风格。

还有一个大家容易忽略的细节，就是解码参数在不同模型之间的迁移性。你在 ChatGPT 上调好的温度零点三和 Top-P 零点九的配置，直接用到 DeepSeek 或者 Claude 上，效果是类似的但不完全一样。因为每个模型的 softmax 输出分布本身就不一样。ChatGPT 的分布可能天然更集中，Claude 的可能更分散。所以换模型的时候，即使参数名字一样，建议重新微调一下温度，而不是直接搬过去用。

解码策略这个领域还在快速发展。最新的方向包括对比解码，也就是同时让两个模型——一个小模型和一个大模型——一起推理，用小模型不选什么来指导大模型选什么。还有一个方向叫不和谐解码，通过主动避免模型过于确定的模式来增加多样性。但这些都属于比较前沿的研究，作为 Agent 工程师，你只需要知道通用解码策略就够了——面试官一般不会问这些前沿方向。

最后给你一个可以把所有知识串起来的一句话。当你输入一段文字，模型通过 RoPE 知道每个字的位置，通过 Attention 知道字与字的关系，通过 FFN 提取字背后的含义，通过 LM Head 转换成词表概率，而解码策略就是从这个概率分布中做出最佳选择的艺术。每个环节缺一不可，而解码策略是让模型输出从可用变成好用的最后一公里。
