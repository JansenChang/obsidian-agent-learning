昨天我们深入拆了 Re-ranking 和 RAG 的缓存架构，也讨论了那个经典问题——用户问「君心似我心讲什么」，系统却在全库搜 chunk 拼碎片。这个问题的本质你已经抓住了：检索粒度不匹配，系统不理解用户真实意图。

今天我们在之前三天学的内容上再往上走一层。传统 RAG 告诉你检索一次然后生成答案。Agentic RAG 说别急，你可以搜一次，看看结果，不够好再搜第二次，直到满意为止。

先理解 Agentic RAG 解决了传统 RAG 什么硬伤。传统 RAG 执行单次搜索，找不到就找不到，不会尝试换一种方式。用户查产品价格，系统扫向量库没搜到，直接说没有。但人查价格会翻产品列表、定位到产品页、再找价格栏目。Agentic RAG 允许模型在检索和生成之间反复，像人一样逐步缩小范围。

还有一种更常见的场景。用户问「你们公司支持哪些支付方式」。传统 RAG 搜向量库找到了一篇三年前的文档，里面写了支持支付宝和微信支付。但公司去年已经接入了银联和 Apple Pay，文档没来得及更新。传统 RAG 不会意识到文档可能过时、不会去查最新的产品说明页。Agentic RAG 会搜到旧文档后自问时效性够不够，发现存疑就再去搜最新的产品公告页面，拿到准确信息后再做回答。

我给你一个具体对比来建立直觉。一个 RAG 系统里面存了公司所有员工手册、项目文档和技术方案。用户问「那个区块链项目当时的架构方案在哪」。传统 RAG 把这句话编码成向量去库里搜最像的十个 chunk。因为用户没有说出项目代码名，只说了「那个」，向量检索很可能找到的是所有包含区块链关键词的文档，排名靠前的未必是真正要找的那篇。Agentic RAG 会先自问这句话里缺了什么关键信息。它发现缺了项目名，于是生成一个子问题去搜项目列表或者查最近的项目文档，拿到项目名后再做精确检索。两步走，第一步补信息，第二步精准搜。

从今天开始，你学的不再是怎么把一次检索做好的技巧，而是让系统自己决定怎么做检索的架构。

先把它最核心的运作机制拆开。这个核心机制我称之为判断执行评估循环。用户来一个问题。Agent 先问自己，我需要检索吗。如果需要，选什么策略。走 BM25 关键词精确匹配还是 Embedding 语义搜索还是标题索引还是网络搜索。执行选定的检索策略，拿到一批结果。这时候 Agent 不着急用结果生成答案，而是再做一轮评估：这些结果够不够回答用户的问题。够了就生成答案，不够就改写查询再搜一次。自己评估自己搜到的东西，这是传统 RAG 和 Agentic RAG 最关键的分界线。

循环次数有一个上限，通常设两到三次。达到上限的时候返回已有结果，同时给用户注明局限性。这个循环依赖三个支撑组件。第一个叫判断力，模型要能评估当前信息够不够。第二个叫工具选择，模型要能根据问题类型选不同的检索入口。第三个叫记忆，模型要记住自己已经搜过什么，避免反复搜同一个东西。这个记忆怎么实现。最简单的做法是在上下文里维护一个已检索查询的列表，每次执行新检索之前先查列表，如果发现当前查询已经做过了就直接跳过。更完善的实现是把已检索的内容摘要也存下来，让模型在决定要不要重检时能参考之前已经拿到了什么信息。这样即使两次查询表面不同，但搜到的内容高度重叠，也能避免重复工作。

业内目前有四条主流的技术路线来实现这个循环。

第一条叫 Corrective RAG。做法是检索后让 LLM 给自己的检索结果打分。评估通常看两个维度。一，检索结果的核心实体和问题中的核心实体是否匹配。二，检索结果的主题和问题是不是同一个领域。两个维度都达标了就认为质量合格，否则改写查询重搜一次。改写有三种常用策略。补全缺失关键词，比如用户问「那个东西怎么用」，改写后变成「钉钉审批功能怎么用」。换同义词，比如把「调优」换成「优化」或者「性能提升」。完全换一种问法，比如把陈述句改写成疑问句。最多重试一到两次。

第二条叫 Self-RAG。精度更高但开销也更大。每条候选 chunk 会生成两个辅助标记。第一个标记叫 isRel，isRelevant 的缩写，表示这条 chunk 和问题有没有关系。第二个标记叫 isSup，isSupported 的缩写，表示这条 chunk 能不能支撑答案。假设你搜到了十条候选 chunk。模型逐条判断。第一条 isRel 是 but isSup 否，保留但注明不完全支撑。第二条 isRel 否，直接丢弃。第三条 isRel 是且 isSup 是，高优先级纳入。最后只有通过这两道门槛的 chunk 才放进 LLM 的上下文。Self-RAG 是目前精度最高的方案，医疗法律这类不能出错的场景非常需要它。代价是每条 chunk 都要额外过一遍 LLM，十条候选就意味着十条额外推理，算力和延迟的消耗相当可观。

第三条叫 HyDE，全称假想文档嵌入。这个方案的思路很有意思。用户的问题往往太短，信息量不够。用户说一句「异步怎么调」，提取出来的语义信号非常有限，拿去向量库里搜命中率不高。HyDE 的做法是先让 LLM 根据问题生成一段假想文档，内容按照答案的风格来写。比如「异步编程可以通过减少 await 次数合并协程使用 asyncio dot run 来优化延迟」。然后拿这篇假想文档去向量库检索。假想文档比用户输入的几个字携带更多语义信号，匹配的精确度自然更高。很多短查询场景下命中率能提升百分之二十到三十。

第四条叫 Multi-Hop 多跳检索。这是最贴近我们说的推理能力的一条路线。第一次检索找到的内容里包含一个新实体，这个实体变成下一次检索的起点。来一个具体的走查。假设用户问 2024 年诺贝尔化学奖得主是谁。检索，找到一条结果说获奖者是 David Baker、Demis Hassabis 和 John Jumper。模型的推理模块从中提取出一个实体：Demis Hassabis。然后拿这个实体做第二次检索，问 Demis Hassabis 是哪家公司的创始人。检索结果说 DeepMind。再拿 DeepMind 做第三次检索，问 DeepMind 的 AlphaFold 解决了什么问题。检索结果说蛋白质结构预测。最终答案综合了这三轮检索的信息。每解开一关就拿到下一条线索，这是 Multi-Hop 的核心模式。这个方案对 LLM 自身的推理能力要求比较高，小模型在 Multi-Hop 上的表现往往不理想。

把四种模式放在一起快速对比一下。Corrective RAG 和 Self-RAG 都是围绕质量评估展开的，区别在于评估的对象不同。Corrective RAG 评估整体的检索结果质量和覆盖度，Self-RAG 评估每一条候选 chunk 的质量。前者适用于快速判断要不要重检，后者适用于精细筛选哪些 chunk 进上下文。HyDE 和 Multi-Hop 都是围绕查询本身展开的，区别在于改写策略不同。HyDE 通过扩展查询的文本来提升命中率，Multi-Hop 通过分解查询为多步子查询来逐步逼近答案。一个是横向扩展，一个是纵向深入。

四种方案各有各的适用场景。Corrective RAG 适合输出质量波动较大的系统，核心是加了一层质量门防止生成质量低的内容。Self-RAG 适合精度优先对延迟不敏感的场景，代价是额外算力。HyDE 适合用户查询很短的场景，比如移动端搜索或者语音助手。Multi-Hop 适合多步骤推理的场景，比如研究型问答和故障排查。它们不是互斥的。生产环境里最常见的组合是把 Corrective RAG 的质量评估机制和 HyDE 的查询扩展能力叠加在一起。

说完了 Agentic RAG 的四种模式，接下来说 GraphRAG。

传统 RAG 对跨文档的问题一直很吃力。用户问华为和腾讯在云计算领域有哪些合作。传统 RAG 检索到了华为云的某一段和腾讯云的某一段，但没有任何一篇文章同时提到了双方的合作关系。拼出来的答案大概率两头不靠。GraphRAG 把文档中的实体和关系抽取出来建成知识图谱，然后按图的结构去检索。同样的问题，知识图谱里有这样的路径：实体华为通过合作关系连接到实体腾讯，同时华为和腾讯各自通过主营业务关系连接到实体云计算。检索时沿着实体关系走，就能定位到两者的交叉点。

GraphRAG 的完整流程分成五步。第一步用 LLM 从每篇文档里提取实体。实体包括人物、公司、产品、概念、地点、事件。第二步识别实体之间的关系。比如 A 是 B 的母公司，C 是 D 的竞争对手，E 投资了 F。第三步把实体作为节点、关系作为边存入图数据库，常用的图数据库是 Neo4j 或者 ArangoDB。第四步为每个实体生成向量 Embedding。检索时先识别出用户问题中涉及的实体，在图中定位到对应节点，沿着关系扩散一到三跳，把扩散到的子图内容聚合成一段文本。第五步 LLM 基于聚合后的子图内容生成最终答案。

微软 2024 年开源的 GraphRAG 是目前业界关注度最高的实现。它在基础流程上额外做了两件重要的事。第一件是社区检测。它把相似主题的文档分到同一个组里，组内实体关系更密集的集合称为一个社区。为什么要做社区检测？因为整个知识图谱的连接太多了。实体越多关系越密，检索时从用户问题涉及的实体出发沿关系扩散，可能扩散到完全不相关的话题。社区检测相当于给知识图谱画了一个分区，检索时只在相关社区内搜索，极大降低噪音。第二件是社区摘要。每个社区生成一段摘要，说明这个社区里有哪些核心实体和关系。这些摘要本身也作为索引数据存入。底层存原始实体和关系，上层存社区摘要，形成一套分层索引。检索时分两层走。全局性的问题比如我们产品整体表现如何，搜社区摘要就够了。局部性的问题比如某个功能的 Bug 修复记录，搜具体的实体关系。这种分层设计是 GraphRAG 的精髓——不是所有问题都需要下钻到最细粒度。

GraphRAG 擅长回答三类传统 RAG 很难处理的问题。第一类跨文档关联问题，比如华为和腾讯在哪些领域有合作。传统 RAG 找不到一篇同时覆盖两个实体的文档，但图结构的优势就是天然能跨越文档边界。第二类全局统计问题，比如哪个模块收到的 Bug 报告最多。这需要汇总大量文档中的相同概念。第三类对比分析问题，比如我们的产品在哪些方面收到最多好评。图结构可以聚合所有提到好评的实体和关系然后归纳出主题。

做一组三类问题在传统 RAG 和 GraphRAG 上的对比。问我们产品的主要客户群体是谁。传统 RAG 搜 customer 相关片段，只能找到某篇文档里提到的个别客户名称，拼不出整体画像。GraphRAG 的社区摘要可能直接包含「主要客户集中在金融行业和教育行业」这样的全局信息。问 2024 年第四季度哪个项目延期最多。传统 RAG 找不到哪篇文档同时汇总了所有项目的延期情况。GraphRAG 可以从项目实体和延期关系节点中聚合出答案。问我们的技术方案和竞争对手有什么区别。传统 RAG 搜到自家方案和竞品方案各自独立的文档，但找不到对比内容。GraphRAG 通过竞争关系边直接找到两者的对比信息。

GraphRAG 的代价同样明显。建图成本高，用 GPT-4 处理一百万 token 大约需要一百美元。实体抽取的质量直接影响下游检索，抽错了后面的搜索只会越偏。知识库更新频繁的场景也不适合，因为每次更新都要重新抽取实体。

现在讲工程实践里三个最直接的关键问题。

第一个问题，Agentic RAG 什么时候停止循环。三种策略。最简单的固定次数，限定最多搜两到三次。这个策略的好处是延迟可预测，坏处是对复杂问题可能不够用、对简单问题可能浪费了一次评估开销。第二种是置信度阈值。评估分数超过 0.8 就停。灵活但需要花时间调参数，阈值设太高会导致不必要的重检，设太低会导致结果还不足够就停了。第三种是边际收益法，新结果不带来新信息就停。比如第一次搜到了五条相关文档，第二次搜到的新文档和这五条基本重复，那就没必要再搜了。边际收益法效果最好实现也最复杂，需要比较新结果和已有结果的信息重叠度。生产环境里最常见的是固定次数加置信度阈值的混合方案——最多搜三次，但任何一次评估分数达标了就提前停。评估分数的来源有两种。一种是用 LLM 自己打分，比如通过 Prompt 让 LLM 输出一到十分的可信度评分。另一种是专门训练一个小模型做质量评估打分，这个方案精度更高但需要额外的训练数据和部署成本。

第二个问题，延迟怎么控制。Agentic RAG 的总延迟约等于检索次数乘以检索时间加上评估时间加上改写时间。优化方向有三个。第一次检索走缓存，很多常见问题一次就够了，纯缓存命中零延迟。检索之间共享上下文窗口，已经看过的内容不需要重复传输。如果多跳检索不依赖上一步的结果，可以用并行搜索来降低串行的累积延迟。

第三个问题，GraphRAG 的建图频率怎么定。静态知识库一次性建图就够了。周更新的知识库可以每周做增量重建，只处理新增的文档和变化的部分。实时动态数据基本不适合 GraphRAG，因为建图成本太高，跟不上更新速度。

现在做知识连接。你昨天自己推理出的问题答案是一条很重要的连接。你问「君心似我心讲什么为什么传统 RAG 做不好」，自己走完了先识别意图再选策略搜标题定位文档再取整篇做概览的完整路线。你自己推导出的方案就是 Agentic RAG 的一个具体实例。这件事说明 Agentic RAG 不是凭空出现的理论，是你在工程实践中靠直觉就能想到的优化方向，只是现在有人给它起了个正式名字、整理出了几条标准化的实现路线。

Agentic RAG 的判断检索再判断再检索的循环跟 Day 7 的 ReAct 框架是同一个结构。ReAct 是思考加行动加观察的通用循环，Agentic RAG 把行动具体化为执行一次检索，把观察具体化为评估检索结果的质量。你在 Day 7 学到的 Tool Use 能力，在这里变成了 Agent 用来检索的工具调用。你当时学的把搜索结果作为 Observation 喂回给 LLM 的模式，今天变成了 Corrective RAG 的核心评估环节。所以 Day 7 和 Day 11 之间有一条清晰的递进关系——Tool Use 是基础能力，Agentic RAG 是这种能力在 RAG 领域的具体应用。

Day 10 的查询改写今天变成了 Corrective RAG 的内置组件。查询改写单独看是一个点上的技巧，组合进质量评估循环后变成了系统级的能力。它们的搭配顺序是这样：查询改写负责把模糊问题变成精确查询，Corrective RAG 的质量评估负责判断改写后的查询效果够不够好。如果不够好就再改一次，直到改到可以用为止。

Day 5 的 Prompt 工程在这里也有新的应用。Agentic RAG 的 System Prompt 需要告诉 LLM 三件事。你可以多次检索。每次检索完要自评结果质量。你知道什么时候该停下来。这三个指令写在 System Prompt 里，模型的行为就会从一次性检索变成多轮自适应检索。你 Day 5 学的 System Prompt 设计原则——包括上下文注入、行为约束——在今天变成了 Agentic RAG 的行为控制器。Day 9 的 Hybrid Search 和今天也有关系。Agentic RAG 选择检索策略时可以动态决定走 BM25 还是 Embedding 还是两者都走。Hybrid Search 的互补思想在这里升级为动态调度。系统不再固定走哪条路，而是根据每次查询的特点实时选路。

今天的暗线是 Day 8 到 Day 11 的 RAG 进化路径。Day 8 你学了 Embedding 基础检索。Day 9 加上了 Chunking 策略和混合搜索。Day 10 加上了 Bi-encoder 加 Cross-encoder 的精排架构。Day 11 让 Agent 接管检索策略，从被动搜一次变成了主动多轮搜索。每一步都在增加系统的自主决策权。当你回顾这四天的内容，你会发现一个明确的趋势——系统的决策权在逐步从人类转移到机器手里。Day 8 人类决定用什么模型怎么切文档，Day 9 人类决定 BM25 和 Embedding 的权重配比，Day 10 人类决定粗筛精排的串联顺序，Day 11 系统自己决定要不要搜、搜什么、什么时候停。这种决策权的转移，是 RAG 系统从工具进化为自主系统的标志。另一条暗线是检索粒度的层次。最底层是片段级，对应传统 chunk 检索。中间层是文档级，对应昨天讨论的标题定位加全文概述。最上层是知识图谱级，对应今天的 GraphRAG。越往上走，检索的单位越大包含的上下文越完整。系统设计里有一条通用的规律：增加自主决策权的同时也要增加对应的约束机制。Agent 搜了三轮还不满意，不能无限搜下去，必须有停止条件。

总结今天的内容之前，做一个完整的端到端走查，把今天所有内容串起来。假设你是一个企业知识库 RAG 系统的架构师，每天处理十万次查询，知识库里有十万篇文档、包含公司内部详细的技术方案、产品文档和项目文档。用户问「去年那个支付中台项目，最终用了什么技术方案」。这个查询包含了几个模糊点。去年，是哪一年。支付中台项目，代码名可能不是支付中台四个字。最终用了什么技术方案，这个问题需要从项目文档的结论部分提取。

Agentic RAG 系统开始处理。第一步，查询改写。系统用 HyDE 先生成一篇假想的答案文档，包含技术方案选型相关的内容，拿假想文档去向量库做第一轮粗筛。第二步，检索拿到 Top-30，自评发现结果里包含了好几个支付相关项目，不确定哪个才是用户说的那个。第三步，系统决定搜项目元数据索引，通过关键词支付中台和项目立项时间范围来定位具体项目。这一步走的是 BM25 精确搜索而不是向量语义搜索，因为项目名和年份是精确字段而非语义概念。第四步，定位到项目后，再搜该项目的技术方案文档，用 Cross-encoder 精排出最相关的五段内容。第五步，系统评估这五段内容是否完整覆盖了技术选型的全部信息。如果发现缺少关键信息比如没有提到数据库选型，就针对性再搜一次。最多搜三次。在这个场景下大模型到第二次搜索时已经拿到了足够信息。最终返回答案的同时写回缓存，下次同样的问题零延迟响应。这个链路里，Corrective RAG 的质量评估在每一步之后都在发挥作用。HyDE 在第一轮帮助短查询扩展语义信号。Multi-Hop 的思路体现在从找到项目再到找技术方案文档的两步搜索。四种模式不是单独用的，是组合在一起形成一个完整系统。

总结今天的内容。第一，Agentic RAG 的核心是检索加自评加可能的重检循环，不是一条道走到黑。第二，Corrective RAG 适合输出质量波动大的场景，Self-RAG 适合精度优先场景，HyDE 适合短查询场景，Multi-Hop 适合多步推理场景。第三，GraphRAG 通过实体关系图谱解决了传统 RAG 跨文档关联能力弱的问题，但建图成本高。第四，工程上最核心的权衡是循环次数、延迟和效果三者之间的关系。
