从 Day14 到 Day18，我们搭好了一个完整的单 Agent 系统——它有 ReAct 循环做推理，有 Function Calling 做执行，有记忆系统做持久化，有 Context Engineering 做上下文管理。这是一个能力全面的单兵。但现实世界的问题，往往不是一个单兵能解决的。

想象一个场景。一个用户对 Agent 说——帮我分析一下新能源汽车市场，包括政策趋势、竞品定价、技术路线对比，最后给我一份投资建议。这个问题跨了政策分析、商业定价、技术评估和金融投资四个领域。如果你让一个 Agent 来处理，它的系统提示词要同时涵盖这四个领域的知识，它的工具列表要包括政策数据库查询、竞品价格爬虫、技术论文检索和财务建模工具。上下文窗口塞不下这么多工具 Schema，LLM 在这么多工具之间的选择准确率也会下降。这就是单 Agent 的天花板——能力边界受限于上下文窗口和单一角色的认知负载。

Multi-Agent 的思路是把一个大问题拆成多个子问题，每个子问题由一个专职的 Agent 处理。每个 Agent 只关注自己的领域，有自己的系统提示词、自己的工具集、自己的记忆。它们之间通过消息或共享数据结构来协作。这就是从单兵作战到团队协作的跃迁。今天我们就来拆解五种主流的 Multi-Agent 编排模式，面试中怎么对比它们，以及常见的框架选型。

第一种模式叫 Supervisor，调度模式。这是最直观也最常用的 Multi-Agent 模式。

结构是一个 Supervisor Agent 加上若干个 Worker Agent。Supervisor 自己不执行业务逻辑，只做一件事——分析用户的请求，把它拆解成子任务，指派给合适的 Worker，汇总 Worker 的结果，形成最终回答。

给你一个具体的走查。用户说要一份市场分析报告，包含竞品分析和定价建议。Supervisor 先思考——这个请求需要两个子任务，竞品分析需要调用市场数据工具，定价建议需要用到财务模型工具。Supervisor 决定把竞品分析指派给 Market Analyst Agent，把定价建议指派给 Pricing Agent。两个 Worker 收到各自的任务后，在自己的 ReAct 循环中独立工作。Market Analyst 去查竞品数据、整理对比表格。Pricing Agent 去跑成本模型、计算价格区间。两个 Worker 完成后把结果返回给 Supervisor。Supervisor 检查两份结果是否完整，把它们整合成一份报告，返回给用户。

Supervisor 模式的核心优势是结构清晰。每个 Agent 的职责边界明确，Supervisor 掌握全局调度权。当子任务之间有明确的依赖关系时——比如先做竞品分析才能定价——Supervisor 可以控制执行顺序，先派 Market Analyst，等它返回后再派 Pricing Agent。这就是串行依赖的处理。

Supervisor 模式的代价是单点瓶颈。所有任务都经过 Supervisor 中转，如果 Supervisor 自己做错了任务拆解或者分配错了 Worker，整个流程就偏了。而且 Supervisor 本身也是一个 LLM 推理节点，它分析任务和汇总结果也会消耗 token。

面试官可能会追问——Supervisor 自己要不要也有领域知识？答案是不需要。Supervisor 只需要知道每个 Worker 的能力描述，不需要知道 Worker 领域的具体知识。它的系统提示词大概是——你是一个任务调度器，负责分析用户请求并分配给合适的专业 Agent。你不需要自己回答问题。这种职责分离让 Supervisor 保持轻量。

第二种模式叫 Planner-Executor，规划执行分离。它在 Supervisor 的基础上做了进一步的分工细化。

在 Supervisor 模式中，任务拆解和结果汇总都在 Supervisor 一个 Agent 里完成。Planner-Executor 把这拆成了两个角色。Planner 负责制定完整的执行计划——先做什么，后做什么，每步的输出是什么。Executor 负责按计划逐步执行，每完成一步就汇报结果和状态。

Planner-Executor 适合流程明确、步骤可以预先规划的任务。比如用户说帮我部署一个 Web 应用。Planner 先输出完整计划——第一步检查服务器状态，第二步拉取代码，第三步安装依赖，第四步启动服务，第五步验证可访问性。Executor 拿到计划后逐步执行，每完成一步给 Planner 反馈。如果某一步失败了——比如安装依赖时报了版本冲突——Planner 根据反馈调整后续步骤。

这个模式和 Supervisor 的关键区别在于规划的时机和粒度。Supervisor 是一边执行一边动态决策下一步做什么，属于在线规划。Planner-Executor 是在执行前就做完整的离线规划。离线规划的优点是全局最优——Planner 能看到全貌，不会走着走着偏离方向。缺点是对变化不敏感——如果执行到一半发现情况跟规划时不一样了，Planner 需要重新规划，增加了延迟。

第三种模式叫 Reviewer，评审循环。这是一种质量保障模式。

结构是一个 Generator 和一个 Reviewer。Generator 负责产出内容——代码、文案、分析报告等等。Reviewer 负责评审 Generator 的产出，指出问题，要求修改。Generator 根据修改意见重新产出。这个循环可以重复多次，直到 Reviewer 认为质量达标，或者达到最大循环次数。

Reviewer 模式最经典的应用场景是代码审查。Generator 写了一段代码，Reviewer 检查这段代码有没有安全漏洞、有没有性能问题、是不是符合团队规范、有没有边界条件没处理。Reviewer 返回一个修改清单，Generator 逐条修改。这个循环替代了人工 Code Review 的大部分工作。

Reviewer 模式的关键设计在于 Reviewer 的判断标准。你不能只告诉 Reviewer 你是一个代码审查者，你需要给它非常具体的评审维度——检查安全漏洞，检查 OWASP Top Ten；检查性能，注意 N 加一查询和大对象循环；检查规范，确保函数名符合驼峰命名。评审维度越具体，Reviewer 的判断越一致。

面试中有一个高频追问——如果 Reviewer 和 Generator 意见不一致怎么办？Generator 说我的设计是合理的，Reviewer 说你的设计有问题。这就引入了一个仲裁机制。最简单的仲裁是把争议提交给 Supervisor 或者 Human in the Loop。高级的做法是引入第三个 Agent 当裁判，各自陈述理由，裁判投票。这就是从双人评审升级到了三人仲裁。

第四种模式叫 Debate，辩论模式。这是处理需要多角度分析的问题时最有效的模式。

结构是多个对等的 Agent，各自从一个立场或角度出发给出方案，然后互相质疑对方的方案，最后通过投票或仲裁达成共识。

给你一个具体的例子。用户问——我应该用单体架构还是微服务架构来开发这个新项目？你启动三个 Agent。一个站在单体架构的立场，一个站在微服务架构的立场，第三个是中立仲裁者。单体架构的 Agent 找出了所有支持单体的论据——开发速度快、部署简单、运维成本低。微服务架构的 Agent 找出了支持微服务的论据——独立扩展、技术栈自由、团队解耦。双方各自列出论据后，互相质疑对方的弱点。仲裁者听完双方的辩论后给出综合建议——考虑到你的团队只有五个人，项目处于早期验证阶段，单体架构更适合当前阶段，等项目规模增长后再考虑拆分。

Debate 模式的核心价值不是选出唯一正确答案，而是确保所有角度都被考虑了。在很多复杂决策中，最坏的结局不是选错了方案，而是漏掉了一个关键角度。

辩论模式的协调成本比 Supervisor 高得多。Supervisor 是中心化的，n 个 Worker 加上 Supervisor 只需要 n 加一次 LLM 调用。辩论模式是去中心化的，n 个 Agent 如果两两辩论，调用次数就是 n 的平方级别。所以 Debate 只适合参与方较少但每方都需要深度推理的场景。

第五种模式叫 Swarm，群集模式。这是最去中心化也最灵活的模式。

结构是大量轻量级的 Agent，没有中央调度者。每个 Agent 有自己的一个小目标，通过观察环境和与其他 Agent 的局部交互来调整自己的行为。整体智能从大量简单 Agent 的交互中涌现出来，而不是由某个中心规划产生。

Swarm 模式的灵感来自自然界——蚂蚁群没有队长，每只蚂蚁只遵循几条简单规则，比如发现食物就留下信息素、沿着信息素浓度高的路径走。整个蚂蚁群表现出的觅食效率是任何单只蚂蚁都做不到的。

在 Agent 的世界里，Swarm 模式同样有效。假设你需要从海量数据中找出异常模式。你启动五十个轻量级 Agent，每个 Agent 只负责扫描数据的一个子集。当某个 Agent 发现潜在的异常，它发出一个信号，附近的 Agent 收到信号后也来检查同一个区域。异常越明显，聚集的 Agent 越多。没有中央调度，所有 Agent 各自为战但通过局部信号协同。

Swarm 的优点是弹性好——某个 Agent 挂了不影响整体。也适合大规模并行任务。缺点是行为难以预测和调试——你写了一堆规则给 Agent，但整体涌现出来的行为可能完全出乎意料。这也是为什么生产系统中 Swarm 用得最少——可解释性和可控性是生产系统的硬要求。

好，讲完了五种模式各自是什么，现在我们来做一个系统的对比。面试中如果你被问到——请对比一下 Multi-Agent 的几种编排模式，你应该怎么回答？

从三个维度来比较。第一个维度是协调成本。Supervisor 成本最低，所有任务通过调度者中转。Planner-Executor 稍高，因为 Planner 需要做全局规划。Reviewer 成本中等，取决于评审循环的次数。Debate 成本最高，因为 Agent 之间需要多轮互相质疑。Swarm 的单 Agent 成本最低但整体协调成本不可预测。

第二个维度是容错能力。Swarm 最强——一个 Agent 失败不影响全局。Debate 次强——多个 Agent 给出了冗余的方案，单点失败被多票覆盖。Supervisor 最弱——Supervisor 自己出问题了整个流程就断了。这引出了一个工程上的补救手段——给 Supervisor 加一个备份 Supervisor，主 Supervisor 超时或异常时备用的顶上。这就是 Supervisor 的高可用方案。

第三个维度是适用场景。Supervisor 适合任务可以明确拆解且有依赖关系的场景。Planner-Executor 适合步骤固定的自动化流程。Reviewer 适合对质量有硬要求的产出场景。Debate 适合需要多角度权衡的决策场景。Swarm 适合海量并行探索和异常检测场景。

框架选型也是一个面试热点。目前最主流的三个 Multi-Agent 框架是 AutoGen、CrewAI 和 LangGraph。

AutoGen 是微软的，核心抽象是对话驱动的 Agent 协作。多个 Agent 之间通过消息来通信，Agent 可以自主决定回什么、问什么、要不要调用工具。AutoGen 天然支持 Supervisor 模式和 Debate 模式，但 Planner-Executor 需要手动实现。它的优势是对话可追溯——所有 Agent 之间的交互都被记录下来，方便调试和审计。

CrewAI 的设计理念是角色驱动。你先定义一组 Agent，每个 Agent 有自己的角色、目标和背景故事。然后你定义 Task，指定哪个 Agent 执行哪个 Task，Task 之间的关系是顺序的还是分层的。CrewAI 更贴近 Planner-Executor 和 Supervisor 的混合体。它的优势是上手快，定义角色和任务的方式非常直观，适合快速原型。

LangGraph 不是专门的 Multi-Agent 框架，而是一个通用的图编排引擎。你可以用节点代表 Agent，用边代表 Agent 之间的消息传递，用条件边实现动态路由。LangGraph 的优势是灵活性最高——五种模式都能实现，而且可以混合使用。代价是开发成本高——需要自己设计图结构，没有开箱即用的 Supervisor 或 Debate 模板。

面试中怎么回答选型问题？标准答案是看你的需求。如果你需要快速验证一个 Multi-Agent 的想法，CrewAI 最省事。如果你需要对话可追溯和微软生态集成，选 AutoGen。如果你需要最大的灵活性来定制编排逻辑，或者你的系统本身就在用 LangChain 生态，选 LangGraph。这三种不是互斥的，很多团队是先用 CrewAI 做原型，确认模式可行后用 LangGraph 重写生产版本。

再多讲两个面试中最容易被追问的实战问题。

第一个问题——Multi-Agent 一定会比单 Agent 好吗？答案是不一定，而且很多场景下单 Agent 更好。面试官问这个是想看你有没有批判性思维。

Multi-Agent 引入了额外的协调成本。每增加一个 Agent，就多了一套系统提示词、一套工具列表、一套记忆。整体 token 消耗不是线性增长，而是因为 Agent 之间互相通信而可能是指数级增长。一个经典的翻车案例——团队用 Supervisor 模式搭了八个 Agent 来处理客服请求，结果发现百分之七十的 token 消耗在了 Agent 之间的通信上，只有百分之三十真正用于回答用户问题。后来把八个 Agent 合并成两个，回答质量反而提高了，因为 Supervisor 不再因为要从八个 Worker 的返回结果中提取关键信息而丢失上下文。

所以什么时候才应该上 Multi-Agent？判断标准有三个。第一，单 Agent 的系统提示词已经长到开始明显影响工具选择的准确率了——说明职责太多，需要拆分。第二，不同的子任务需要的工具集完全不重叠——比如一个任务需要 SQL 查询工具，另一个任务需要图像分析工具。拆成两个 Agent 各自只加载自己需要的工具，降低了每个 Agent 的认知负载。第三，子任务之间没有共享上下文——每个子任务可以独立运行而不需要其他子任务的中间结果。如果三个条件都不满足，单 Agent 反而是更好的选择。

第二个问题——Multi-Agent 系统的错误怎么排查？单 Agent 出错了你打印消息列表逐条看就行。Multi-Agent 出错了呢？到底是 Supervisor 分派错了，还是某个 Worker 执行错了，还是 Worker 之间的通信出了误解？

排查 Multi-Agent 的错误需要一套分层日志。第一层是编排日志——Supervisor 做了什么决策，把什么任务派给了哪个 Worker，什么时候派的，什么时候收到返回。第二层是 Worker 日志——每个 Worker 在自己的 ReAct 循环中每一步的 Thought、Action 和 Observation。第三层是通信日志——Agent 之间传递的消息原文是什么，有没有被截断或被修改。当你怀疑是某个 Worker 的问题时，只看那个 Worker 的日志就行。当你怀疑是调度逻辑的问题时，看编排日志。当你怀疑是信息传递的问题时，看通信日志。

还有一个容易被忽略的排查技巧——时间戳对齐。不同 Agent 的日志可能分布在不同的文件或服务中。必须在每条日志上打上统一的时间戳，然后才能按时间排序还原出整个 Multi-Agent 系统的执行时序。这就是分布式追踪在 Agent 系统中的对应物。LangGraph 和 AutoGen 都内置了追踪支持，CrewAI 需要额外配置。

好，最后来建立知识连接。

Day14 我们手写的 ReAct 循环是单 Agent 的核心推理引擎。在 Multi-Agent 系统中，每个 Agent 都在独立运行自己的 ReAct 循环——Supervisor 在自己的循环中拆解任务和汇总结果，Worker 在自己的循环中执行子任务。区别在于 Worker 的循环中多了一个步骤——把结果返回给 Supervisor 而不是直接返给用户。所以 Multi-Agent 不是替代 ReAct，而是在 ReAct 的上层加了一个编排层。

Day17 我们讲的记忆系统在 Multi-Agent 中有一个重要的扩展——共享记忆。不同的 Agent 之间可以通过一个共享的向量数据库来交换信息。比如 Market Analyst 查到的竞品数据写入共享记忆，Pricing Agent 从共享记忆中读取竞品数据来做定价。共享记忆降低了 Agent 之间的通信带宽需求，但它也需要更精细的权限控制——不是所有 Agent 都应该看到所有共享记忆。

Day18 我们讲的 Context Engineering 在 Multi-Agent 中同样适用，而且更复杂。每个 Agent 有自己的上下文窗口，有自己的 Write、Select、Compress、Isolate 策略。Supervisor 的上下文管理比 Worker 更关键，因为 Supervisor 需要同时看到多个 Worker 的输出，它的上下文窗口压力更大。这就是为什么 Supervisor 的系统提示词要尽量精简——把更多的 token 留给 Worker 的返回结果。
