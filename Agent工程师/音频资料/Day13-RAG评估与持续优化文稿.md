在过去的一周里，我们从 Day 8 的 Embedding 基础检索一路讲到 Day 12 的向量数据库选型。Chunking、Hybrid Search、Cross-encoder 精排、Corrective RAG、GraphRAG、向量数据库部署，整个 RAG 系统的每个环节都拆了一遍。但有一个问题一直没有回答：你的 RAG 系统在实际跑起来之后，到底好不好？怎么证明它变好了还是变差了？今天就是 RAG 专题的最后一站——评估与持续优化。

先说一个很多人都会犯的错。很多人觉得评估 RAG 系统就是设计十几个测试问题，人工看几遍回答，觉得差不多就上线了。这样做的最大问题是，那十几个测试问题是你自己精心设计的，你早就知道答案长什么样。但真实用户的问题千奇百怪。有人用缩写，把知识图谱缩写成 KG 或者只写图谱不说全名。有人拼写错了，把 Retriever 拼成 Retriver 或者把 Embedding 拼成 Embbeding。还有人问的问题你的知识库里根本没有任何相关文档，比如问你们公司的产品支持哪些技术指标，但知识库里根本没有产品规格相关的内容。上线之后用户反馈说答案不准，你完全不知道问题到底出在哪一步。是检索阶段没找到对的内容？还是找到了但 LLM 没用好这些内容？这两个问题的修复方向完全不同。检索的问题要去调索引参数、查数据质量。生成的问题要去改 Prompt、换模型。诊断错了方向，修复只会越修越糟。

要系统性地解决这个问题，你得把评估拆成两个完全独立的维度。第一个维度叫检索质量，衡量的是检索阶段有没有把相关文档找回来。第二个维度叫生成质量，衡量的是 LLM 有没有基于找回来的文档生成正确答案。这两个维度必须分开诊断，不能混在一起看。如果检索阶段就没找到相关文档，你花大把时间优化生成阶段的 Prompt 等于白干，因为 LLM 拿到的上下文本身就是错的，再好的 Prompt 也救不回来。反过来如果检索拉回了正确的文档但 LLM 答错了，那就是 Prompt 或者模型本身的问题，跟检索完全没关系。分清问题出在哪一步，是建立评估体系的核心价值，也是面试官最想看到的系统性思维。

先说检索质量的评估怎么做。第一步是建一个测试集。这个测试集至少包含一百到两百条问题。不是随便选几条就行，每条问题都需要有人工标注的正确答案或者最相关的文档 ID。那问题从哪里来？最好的来源是线上真实的用户查询日志。你从日志里抽取出不同类型的查询，各种类型都要覆盖。有的问题是精确查找，比如公司去年的营收是多少，这种问题应该指向一篇特定的文档。有的问题是模糊搜索，比如最近在人工智能领域有什么新突破，这种问题可能有多篇相关文档。还有的问题是跨文档关联的，比如华为和腾讯在哪些领域有竞争关系，这种问题需要从多篇文档中综合信息才能回答。各种类型的查询都要覆盖，测出来的分数才有代表性。

有了测试集之后，你定期跑检索系统，看它返回的结果和人工标注的结果有多吻合。吻合度用两个核心指标来衡量。第一个叫命中率，英文叫 Hit Rate。这个名字很直观，命中率就是最相关的那篇文档在你的搜索结果里出现了没有。只看有没有出现，不关心它排第几。比方说你设置每次返回十条结果，如果最相关的那条在这十条里出现了就算命中，不在就没命中。第二个指标叫平均倒数排名，英文叫 MRR。这个指标更精细，它关心最相关的那篇文档在结果里具体排第几位。排第一名得一分，排第三名得三分之一分，完全没出现就得零分。举个例子，三条查询，最相关的文档分别排第一、排第三、完全没出现。Hit Rate 是百分之六十七，MRR 是零点四四。这两个指标给我们一个重要的启发——只看 Hit Rate 是不够的。如果你的精排模型把最相关的文档从第一位推到第三位，Hit Rate 还是百分之百看不出变化，但 MRR 从一点零掉到了大概零点五左右。所以 MRR 对排序变化更敏感。这两个指标是工业界最常用的检索质量评估指标。面试里你能说出 Hit Rate 和 MRR 就已经展示了足够的工程理解。如果能补一句如果有多级相关性标注还可以用 NDCG，那就是加分。

说完了检索质量的评估，再说生成质量的评估。这一步比检索评估难得多，因为答案质量是主观的。同一个答案，一个人觉得非常好，另一个人可能觉得一般。所以生成质量的评估目前有两种主流做法。

第一种叫人肉标注。找几个标注员，逐条看问题和 LLM 生成的答案，从三个维度打分。第一个维度叫忠实度，也就是答案有没有编造检索结果里没有的信息。这是最重要的一条，因为 RAG 的核心价值就是减少幻觉。如果答案编造了检索结果里不存在的信息，RAG 的核心理念就被违背了。第二个维度叫完整性，意思是答案有没有覆盖问题的所有方面。有时候 LLM 只回答了问题的一半。比如用户问这个功能支持哪些平台，LLM 只说了支持 iOS 和 Android，但你公司的产品其实还支持 Web 端和 Mac 桌面端。第三个维度叫有用性，就是答案对用户有没有实际帮助。有的答案忠实度高也完整，但写得非常啰嗦绕圈子，用户看到一半就不想看了。人工评估最准但也最贵，标一条可能要几十秒，两百条就要好几个小时，成本不低。

第二种做法叫 LLM-as-a-Judge，用另一个更强的 LLM 来打分。你写一个评估 Prompt，把问题、检索结果和 LLM 生成的答案一起送进去，让 LLM 从忠实度、完整性和有用性三个维度打分，每条输出一个一到五分的评分。OpenAI 做过一个实验，GPT-4 打分和人工打分的一致性达到了百分之八十以上。虽然不是百分之百的完美匹配，但用来做日常的质量监控已经完全够用了。成本比人工标注低得多，你只需要写好 Prompt，写个脚本批量跑，最后汇总统计就能得到质量分数的趋势图。

有了评估方法之后，不能只跑一次就完事，得把它变成持续运行的流程。两周跑一次全量评估是比较合理的节奏。每次跑完之后对比前一次的结果。如果检索质量的 Hit Rate 和 MRR 下降了，说明索引或者数据可能有变化。需要检查是不是新写入的数据质量有问题，或者索引参数需要重建了。如果生成质量的分数下降了，说明 Prompt 或者模型版本有问题，需要检查最近有没有人改过 Prompt。整个链路是发现问题、定位根因、修复、验证。这个闭环一旦跑起来，你的 RAG 系统就不会越跑越差了。

评估方法说完了，接下来说线上最常见的四种翻车场景。这些都是花了不少时间踩过坑才积累出来的经验，面试也常考。

第一种翻车叫索引漂移。你的向量数据库在持续写入新数据，文档越来越多，但是索引参数是建索引时设定的。数据量翻了一倍之后，原来的 HNSW 参数就不再是最优配置了。召回率会非常缓慢地下降，每天下降可能只有千分之几。这个下降速度你很可能一两周都察觉不到。等到用户投诉开始增多的时候，召回率可能已经掉了好几个百分点。解决方案就是定期重建索引。重建的频率取决于数据增长的速度，每周或者每两周重建一次是比较合理的节奏。同时持续监控召回率这个指标，如果连续两次评估都发现下降趋势，就主动触发一次重建。

第二种翻车叫查询分布漂移。上线初期的用户查询和半年后的用户查询差异非常大。上线第一个月，用户问的都是产品手册里直接能找到的问题，比如怎么注册、怎么登录、怎么绑定银行卡。半年后用户开始问只有少数资深员工才知道的冷门问题，比如这个系统在极端高并发流量下的限流策略是什么。你的测试集是上线之初建的，测出来的分数一直很好看，但线上的真实体验早就变了。测试集和真实查询的差距就像一个不断扩大的裂缝，时间越长差距越大。解决方案就是定期更新测试集。从线上收集真实的用户查询日志，挑出有代表性的新问题加进测试集。建议每个月做一次测试集更新，保持测试集和线上真实分布的匹配度。

第三种翻车叫数据污染。知识库里混入了低质量、过时或者重复的文档。一条过时的产品信息混在几万条文档里，人工检查根本发现不了。但 LLM 可能刚好检索到了这条过时文档，基于错误信息回答用户的问题。更隐蔽的情况是多篇高度相似的文档同时被检索到。比如说公司官网的产品介绍在三个月内反复修改了五次，每次修改后都新增了一篇文档，五篇文档高度相似但核心信息不同。检索时系统把五篇都拉回来了，它们互相重复没有带来新信息，但占用了上下文宝贵的 token 位置。而且如果 LLM 刚好看到的是三个月前的那篇而不是最新的那篇，答案就是错的。解决方案是在检索阶段加入质量分过滤。给每条文档打一个质量分，检索时只返回分数高于阈值的文档。质量分可以让 LLM 自动标注，也可以基于文档的时效性和来源可靠性计算。

第四种翻车叫评估噪音。你的 LLM-as-a-Judge 评估结果波动很大，这周准确率百分之九十，下周掉到百分之八十五，但实际系统没有任何变化。这时不用太紧张，先检查是不是评估所用的采样导致的波动。LLM 打分本身有随机性，同样的 Prompt 同样的输入，不同次调用的分数可能不一样。解决方案是每次评估使用完全相同的测试集，不在测试集上做任何改动，确保前后严格可比。同时扩大每次评估的样本量，两百条以上的测试集能有效减少随机波动对整体平均分的影响。

把今天的内容串成一句话。RAG 评估分两路走：检索质量看 Hit Rate 和 MRR，生成质量靠人工或者 LLM-as-a-Judge 打分。持续优化靠定期重建索引、更新测试集、过滤低质量文档、固定样本量减少评估噪音。这四件事做到位，你的 RAG 系统就不会越跑越差了。
