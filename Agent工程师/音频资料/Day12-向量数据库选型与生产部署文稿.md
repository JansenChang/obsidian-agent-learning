从 Day 8 到今天，我们一路从 Embedding 检索学到 Agent 自主决定检索策略。Chunking、Hybrid Search、Cross-encoder 精排、Corrective RAG、Multi-Hop、GraphRAG，这些技术全部建立在同一个底层基础上：向量数据库。向量数据库负责做一件事：把你的查询向量和知识库里所有的文档向量做近似最近邻搜索，在毫秒级返回语义上最接近的 Top-K 条结果。不管你的检索策略多先进、精排模型多精准、缓存设计多巧妙，如果向量数据库这一层慢或者不准，整个系统都会跟着崩。今天我们把视角从策略层切换到基础设施层，回答最实际的问题：市面上那么多向量数据库到底怎么选？选完怎么部署？生产环境踩过的坑有哪些？

先回答根本问题。为什么不能用关系数据库来查向量？关系数据库的 B 树索引是为精确匹配和范围查询设计的。你查用户张三，B 树能直接指向张三那条记录。你查价格在一百到两百之间的商品，B 树也能快速返回。但向量检索完全不同。你要查的不是一个精确值，而是语义上最接近的十条文本。每句文本被 Embedding 模型编码成一串浮点数，这些浮点数之间没有排序关系，你不能说向量 A 大于向量 B。你需要在高维空间里找到距离问题向量最近的邻居。没有任何传统数据库索引能直接支持这种查询。你需要专门的索引来做近似最近邻搜索，简称 ANN。没有 ANN，搜一百万条就得逐条算距离，每条几微秒，一百万条累积到几秒钟，完全不可接受。这就是向量数据库存在的理由。

第一种叫 HNSW，全称是分层可导航小世界图。做法是给所有向量建一张多层图结构。顶层节点很少，只有几十个代表向量，分区很粗，用来快速定位大致区域。中间层节点逐渐增多，分区逐渐变细。底层节点最多，连接最密，用来做最终的精确搜索。搜索时从顶层进入，逐层下钻，每一层找到距离最近的节点，以它为入口进入下一层继续搜索，直到最底层拿到精确的 Top-K 结果。HNSW 的召回率通常能达到百分之九十八以上，是所有索引方案中最高的。代价是需要把整张多层图全部加载在内存里。一百万个 768 维的向量，HNSW 索引大约需要 8 到 10 GB 内存。如果数据量达到十亿级别，光索引就可能吃掉几百 GB 内存，服务器成本非常高。所以要不要用 HNSW 取决于你的数据规模和预算。

第二种叫 IVF，全称是倒排文件。思路和给书编目录一样。先把向量空间聚类成几百个组，组数可以自己设参数，通常设一百到一千。每个组选一个中心向量，也就是组内所有向量的平均位置。搜索的时候先计算问题向量到每个组中心向量的距离，找到最近的几个组，参数叫 nprobe。然后只在选中的组内搜索，不搜全库。IVF 的好处是内存占用远低于 HNSW，因为不需要维护复杂的多层图结构，只需要存聚类中心向量和每个组的成员 ID 列表。IVF 的代价是精度略低。原因是只在邻近的几个组里搜，正确答案有可能落在没被选中的组里。调大 nprobe 可以提升精度但会增加延迟，调小 nprobe 延迟更低但可能遗漏正确结果。IVF 适合内存有限但能接受一点精度损失的场景。

第三种叫 DiskANN，微软出品。核心设计是把索引结构存在 SSD 硬盘上而不是内存里。架构上和 HNSW 类似也是基于图结构，但是做了内存和磁盘的优化。它把热数据存在内存里、冷数据存在磁盘上，通过预取机制来减少磁盘访问的延迟。DiskANN 的延迟比纯内存方案高几毫秒，但是可以管理十亿以上的向量。小规模用不上，当向量规模超过一亿时这是唯一可行的方案。代价是查询延迟比 HNSW 高大约两到三倍，但相比完全搜不到正确结果，多等几十毫秒是可以接受的。

三种索引的选型很简单。内存够大追求召回精度用 HNSW。内存有限能接受轻微精度损失用 IVF。规模超过一亿向量用 DiskANN。

说完了索引，我们来看数据库选型。市面上主流的五个向量数据库。

先说 Milvus。它是最成熟的独立向量数据库，由 Zilliz 公司开源维护。云原生架构支持分布式部署，有查询节点、索引节点和数据节点三层独立架构。索引类型最全，HNSW、IVF、DiskANN 全部原生支持不需要额外配置。Python、Java、Go 的 SDK 都齐全，社区非常活跃。当你的场景需要高可用、大规模、多租户隔离时，Milvus 是首选。它支持分片和复制，可以做到零宕机升级。缺点是架构非常复杂，部署和维护成本相当高。一个小规模的 Milvus 集群至少需要 etcd 做元数据管理、MinIO 做对象存储、Pulsar 做消息队列，再加上查询节点和索引节点和数据节点，光配置这些组件就要花不少精力。如果数据量不到一百万条，就是杀鸡用牛刀。

再说 Qdrant。用 Rust 语言写的，单机性能极好。部署非常简单，一个二进制文件就能跑起来，一条 docker 命令就能启动。同样是分布式架构，支持分片和复制。因为 Rust 的实现，Qdrant 的资源消耗比 Milvus 低很多，同样的硬件配置能支撑更大的数据规模。4 核 16 GB 的服务器，Qdrant 可以流畅处理五百万条向量，Milvus 同样的硬件可能只能处理三百万条。Qdrant 还提供了 quantization 量化功能，可以把 Float32 向量压缩成 Float16 或者 Uint8，内存占用直接减半。在精度影响可以接受的场景下，这个功能非常实用。我个人的建议是，在你确定自己最终规模之前优先从 Qdrant 开始。因为它部署成本最低，从下载到启动只需要几分钟。万一规模上去了，Qdrant 也支持水平扩展，不需要换库。

Chroma 是专为 AI 应用设计的轻量级向量数据库。Python 原生 API，安装就是 pip install chromadb，一条命令搞定。创建一个 Collection，往里 add 文档，然后 query 查，不需要做任何配置。看起来很方便，但代价是功能最少。不支持分布式，不支持高级索引参数调优，不支持分片和复制。我见过有团队在生产环境用 Chroma，数据量到十万条就开始频繁内存溢出，查询延迟飙到几秒钟。Chromas 的定位很清晰：快速原型验证、个人项目、教学演示。千万不要用在生产环境。

PGVector 是 PostgreSQL 的一个扩展，不是一个独立的数据库。安装就是 pip install pgvector，然后在你现有的 PostgreSQL 表上加一个向量类型的列，就可以直接用 SQL 来查向量距离。如果你的系统已经在用 PostgreSQL，PGVector 是零额外运维成本的选择。不需要学新的 API，不需要部署新的服务器，数据备份直接用 pg_dump。PGVector 的查询性能低于专用的向量数据库，支持的索引也只有 IVFFlat 和 HNSW 两种。但它有一个独特优势是专用数据库没有的：事务支持和数据一致性。你的业务数据和向量数据在同一个数据库里，增删改查都在同一个事务里，不需要跨系统同步。如果你已经重度依赖 PostgreSQL，PGVector 是第一优先考虑。它可能不是最快的，但它一定是最稳的。

Pinecone 是唯一的纯托管方案。不需要自己部署服务器，注册账号获取 API key 创建 index 就能用。免费 Starter 层可以跑十万条向量，足够做原型验证和早期测试。性能优秀运维成本为零。缺点是付费后按使用量计费，如果查询频率很高成本不低。而且数据托管在 Pinecone 的服务器上，对数据合规有要求的场景要慎重评估。适合不想碰基础设施的团队，也适合快速上线验证的场景。

把五个数据库放在一起对比。有运维能力的大团队需要高可用选 Milvus。想要部署简单同时性能优秀选 Qdrant。个人项目或者原型验证选 Chroma。已经在用 PostgreSQL 的选 PGVector。不想碰服务器愿意付费的选 Pinecone。

选型聊完了说生产部署。这部分是真正的避坑指南。因为选型决定上限，部署决定下限。

第一件事是容量规划。你需要大概知道你的数据会吃多少内存和硬盘。举一个具体的例子帮助你建立直觉。假设你的知识库里有一百万篇文档，每篇切了五到十个 chunk，总共大概五百万个向量。这个规模用 HNSW 索引大概需要 32 GB 内存，用 IVF 只需要 8 到 12 GB。你不需要记住精确的字节数，需要记住的是这个比例。HNSW 比 IVF 吃内存吃得厉害，因为它在内存里维护了一张多层图，但换来的是最佳的搜索精度和速度。IVF 省内存但是精度略低一点。生产环境的经验法则是：在估算值上翻一倍留余量。如果你预计一年后会增长到五百万条向量，就买 64 GB 内存的服务器。多花钱买内存比到时候查询退化成暴力扫描要好得多。内存不够的后果是什么？索引加载不进来，查询退化成逐条算距离，延迟从几十毫秒变成几秒。这是最严重的事故之一，因为用户能直接感知到系统变慢了。

第二件事是索引构建时机。数据规模小的时候可以实时构建索引，插入一条向量立刻就能被搜索到。但是当数据量大到几十万条以上时，实时构建索引会导致写入性能急剧下降。因为 HNSW 需要在每次插入时调整多层图结构，这个操作很昂贵。生产环境的推荐做法是先用 Flat 模式写入。Flat 就是 plain 模式，不做任何索引结构，只存原始向量。这种模式下每秒能写入几千条向量。等数据积累到目标量之后，手动触发一次索引构建。Qdrant 用 optimize 命令，Milvus 用 create_index 接口。构建期间写入操作不阻塞，一直在进行，但是新写入的向量暂时不能被搜索到。构建完成后系统自动切换索引，所有数据都可用。这个模式叫先写后建。理解这个模式很重要，不然你可能会觉得为什么插入后搜不到。

第三件事是备份策略。向量数据库的备份比关系数据库麻烦，因为索引结构高度依赖内存状态，不能简单地复制数据文件。Milvus 官方推荐的做法是定期用 SDK 导出全量向量到 S3 或者阿里云 OSS，在需要恢复时在新集群上重建索引。Qdrant 有内置的快照功能，一条 create_snapshot 命令导出整个集合，恢复也一条命令恢复，非常方便。PGVector 最简单，底层就是标准 PostgreSQL，直接用 pg_dump 就能备份整张表。如果你对备份的易用性要求很高，Qdrant 和 PGVector 在这方面有非常明显的优势。

第四件事是监控指标。三个必须盯着看的指标。第一个是查询延迟的 P99，也就是百分之九十九的查询都在这个延迟以下。如果 P99 超过一百毫秒，说明索引参数没调对或者硬件不够。正常配置下 HNSW 的 P99 应该在二十到五十毫秒之间。第二个是召回率，用一套固定的测试集定期检查 Top-K 的命中率是否稳定。向量数据库在数据持续增长时，如果索引参数不跟着动态调整，召回率会逐渐下降。降到百分之九十以下就要考虑重建索引。第三个是索引构建进度，大规模重建时你要清楚知道什么时候能完成，不然影响上线排期。

第五件事是版本升级策略。向量数据库行业还在快速发展期，版本更新频繁。Milvus 从 1.x 到 2.x API 完全不兼容，很多功能命名也变了。Qdrant 的升级相对平滑但偶尔也有 breaking change。生产环境的铁律是绝对不要追最新版本。新版本发布后至少观察一个月，确认社区没有大规模 bug 报告再考虑升级。而且升级前一定要在测试环境完整验证一遍查询准确率和延迟表现，确认没有性能回退再推到生产。

把今天的内容串成一句话。索引选 HNSW 保底，按规模升级到 DiskANN。数据库选型没有最好的只有最适合你的。部署上容量翻倍规划、离线构建索引、定期备份、版本升级踩好节奏。这四件事做到位，向量数据库这一层就不会成为 RAG 系统的瓶颈。
