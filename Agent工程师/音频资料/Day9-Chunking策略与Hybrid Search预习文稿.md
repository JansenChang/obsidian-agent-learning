昨天我们讲了 RAG 的基本流程。检索增强生成，把闭卷考试变成开卷考试。你还记得那个三段式 Pipeline 吗——索引、检索、生成，三环环环相扣，任一环节出问题，整个系统就崩了。

今天我们深入索引阶段的第一步，也是最容易被低估的一步——Chunking，分块策略。以及检索阶段的进阶话题——Hybrid Search，混合搜索。这两个话题合在一起，回答一个核心问题：怎么让 RAG 搜得更准。

先从一个你绝对遇到过的问题说起。你有没有试过用 AI 搜索工具提问，结果它给你返回了一大段乱七八糟的内容，里面有你要的答案，但被埋在一堆无关的文字里，你得自己翻半天才能找到？这就是 Chunking 没做好的典型症状。Chunking 这个环节很少出现在 RAG 的炫酷 Demo 里，但它是决定 RAG 系统能不能落地的第一道分水岭。

Chunking 到底是什么呢？很简单，就是把一篇长文档切成一小块一小块的段落。为什么要切？因为你不切的话，用户问一句话，系统可能返回整整三页文档，其中跟问题相关的可能只有三句话，占比可能不到百分之十。其余全是噪声。LLM 大语言模型拿到这一大坨东西，它的注意力被稀释了，生成出来的答案质量就下来了。这就好比你想找一个电话号码，别人却给了你一整本电话簿——号码确实在，但你找不到。

但切太小也不行。你想，如果一句话被切成两半——「Jansen 最近在学」在一个块里，「Agent 面试知识」在另一个块里——检索的时候只搜到了前半句，LLM 拿到「Jansen 最近在学」，学什么？不知道。这就是上下文断裂。或者说，一个完整的论述段被切成三个碎片，散落在不同的块里，检索的时候永远只能搜到其中一块，LLM 看到的永远是不完整的论证。

所以 Chunking 的核心是什么？在「语义完整性」和「检索精度」之间找到平衡。太大噪声多，太小信息碎。这个平衡点不是固定的——取决于你的文档类型和查询模式。对于一般的文本知识库，每块五百到一千个 token，大概三四百个中文字，是工程上最常见的起点。

好，那具体怎么切？三种主流策略，我们一个一个来。

第一种，固定长度分块，最简单粗暴的方法。你设一个数，比如每五百字，程序就每五百字切一刀。不管这句话说没说完，不管是不是把一个专有名词劈成了两半，到了数就切。

我给你一个具体的例子。假设原文是这么一句话——「RAG 是检索增强生成，它解决了大语言模型的知识截止日期问题。」固定长度设的是每十个字切一刀。那么第一个块就是「RAG 是检索增强生成，它解决了大语言」，你看，「大语言」三个字被切出来了，和后面的「模型的知识截止日期问题」分到了两个不同的块里。

这种切法快是真的快，实现起来几行代码就搞定，计算量基本为零。在快速原型阶段、做一个 Demo 给老板看的时候，完全可以用。但一旦涉及到真正的检索精度，它就不够用了。你可能会问，那为什么还有人用？因为有些场景下，文档本身就是高度碎片化的——比如聊天记录、社交媒体内容——每条消息天然就是一个小块，固定长度切反而正合适。

第二种，递归分块。这是 LangChain 框架的默认策略，也是目前工程实践中推荐度最高的方式。它的思路是什么呢？不是上来就按字数切，而是先尊重自然语义边界。

怎么做到呢？它定义了一个分隔符优先级。第一优先级是段落分隔符，也就是两个换行。段落是作者自己划分的，代表了最自然的语义边界。先按段落切。切完之后，如果某一段还是太长，超过了块大小上限，就降级到第二优先级——按句号切。句号是句子的边界。句号切完还太长？降到第三优先级——按逗号切。逗号还长？降到第四优先级——按空格或单个字符切。

我给你一个完整的走查。假设有一篇技术文档，chunk_size 设的是五百字。程序先找段落分隔符，发现有五个自然段，每段大约三百字，都在五百字以内。好，就按这五个段落切成五块，完美。另一篇文章不一样，只有一段，但这段有两千字。程序先尝试按段落切——发现只有一个段落，切不了。降级，按句号切。找到二十个句子，前六个句子加起来四百八十字，第七个句子一加就超了。好，前六个句子是第一块，第七到第十二个句子四百六十字是第二块，以此类推。

递归分块的好处在于，它尽量尊重作者原始的语义结构，同时保证每一块都不超过上限。它不完美——比如两个句子分别讨论不同话题，但因为句号是同一优先级，它们可能被强行压在同一个块里——但它是目前自动化方案里性价比最高的。

说到这，你有没有想过一个问题——分块策略能不能再智能一点？不是靠分隔符猜测语义边界，而是让 AI 自己判断「这里该切了」。这就是第三种策略，语义分块，也叫 Semantic Chunking。

它的做法非常优雅。把文档里每一句话逐一做 Embedding，也就是向量化，变成一串数字。然后计算每一对相邻句子的余弦相似度。如果两句话都在讲同一个话题，相似度会很高——比如零点九几。但如果两句话之间话题突然变了，相似度就会骤降——从零点九跌到零点二甚至更低。这个骤降，在数据上就是一个明显的「语义断崖」。断崖在哪里，刀就下在哪里。

用一个具体的例子来帮你建立直觉。假设有一段文字，前三句在讲 RAG 的检索原理，后三句在讲 Jansen 每天通勤时听音频。前三句互相之间的相似度都在零点八五以上，因为它们讲的都是 RAG。后三句互相之间的相似度也在零点八以上，因为它们讲的都是学习习惯。但第三句和第四句之间的相似度只有零点二——因为一句在讲检索，一句在讲通勤。语义断崖出现了，就在第三句和第四句之间切一刀。

语义分块是三种策略里语义最完整的，基本上不会切断逻辑链。但它有一个明显的代价——每一句话都要做一次 Embedding 计算。如果你的知识库有一百万条文档，每条文档平均五十句话，那就是五千万次 Embedding 计算。这个成本，在时间和金钱上都不小。所以实际项目中，百分之八十的场景用递归分块就够了，语义分块留给那些对精度有极致要求的场景——比如医疗文献检索、法律文书检索——这些场景下切错一个语义边界的代价很高。

好的，三种切法讲完了。现在我们来讨论一个非常重要的配套机制——Overlap，块重叠。你想想，不管你用哪种切法，一刀下去，总有信息会刚好落在刀口上。比如一个关键的技术术语出现在第四百九十九字和第五百字之间——刚好在边界上。怎么办？

Overlap 就是最后的保护网。它让相邻的两个块共享一部分内容，就像两块瓦片之间的重叠区域。假设你的块大小是五百字，Overlap 设的是百分之二十，也就是一百字。那么第一块是文档的第一到第五百字，第二块不是从第五百零一字开始，而是从第四百字开始——前一百字跟第一块的尾一百字一模一样。

这样做有什么好处？假设有个关键信息「模型的上下文窗口从一百二十八 K 扩展到了一百万 token」，正好落在第一块的尾部。如果没有 Overlap，这句话在第二块里就只剩下「一百万 token」几个字，前文全丢了。有了 Overlap，第二块的头部完整包含了「从一百二十八 K 扩展到一百万 token」这整个信息，即使第一块被检索时被截断了后文，第二块也能完整命中。

那 Overlap 是不是设越大越好？设到百分之五十甚至更高？不是。重叠太多有两个代价。第一，浪费存储空间——你的向量数据库里存了大量重复内容，索引体积翻倍。第二，检索噪声——用户搜一个问题，可能两三个相邻块都有重叠内容，一起被搜出来塞给 LLM，LLM 看到三遍一样的东西，反而干扰它生成。工程实践中，百分之十到百分之二十是公认的最佳区间。如果你是做代码文档，函数定义之间本来就不需要重叠，设百分之十就够了。如果你是做长篇文章，论述逻辑链比较长，设到百分之二十更安全。

好，Chunking 这个环节我们讲深了。接下来我们进入今天第二个核心话题——Hybrid Search，混合搜索。这个话题直接升级你昨天对 RAG 检索阶段的理解。

回到 Day 8 我们学的 Embedding 检索。它的核心能力是语义匹配。你搜「Python 教程」，它能找到标题叫「编程语言入门指南」的文档，因为两者的向量距离很近。这很强大，对吧？

但 Embedded 检索有一个天生的盲区，一个无论怎么换模型都绕不过去的盲区。你想想，如果用户搜的是一个专有名词、一个产品编号、一个文件名、一个错误码——比如「API-2024-Q3-v2」或者「BR-9527 号工单」或者「Jansen」这个人名——这些东西在 Embedding 模型的训练语料里几乎没出现过。模型根本没见过「Jansen」，它怎么知道「Jansen」是什么意思？它不知道，所以它算出来的向量就是随机的，跟任何文档的相似度都不靠谱。

这就引出了第二种检索范式——关键词检索，学名叫稀疏检索，最经典的代表就是 BM25。BM25 的全称是 Best Match 25，不要被这个名字吓到，它就是 Best Match 第二十五版的缩写。它背后的思想非常朴素，朴素到上世纪七十年代就有人提出来了，但它到今天依然是 RAG 系统里不可替代的一环。

BM25 不看语义，只看关键词。用户搜「Jansen」，它就精确匹配所有包含「Jansen」的文档。用户搜「E10023 错误码」，它就找包含这个编号的文档。精确、直接、零歧义。

BM25 里有两个核心变量你需要理解。第一个叫词频，Term Frequency。就是这个关键词在文档里出现了多少次。直觉上，出现次数越多，文档越相关。但 BM25 做了一个聪明的修正——它设置了饱和效应。一个词出现五次和出现五十次，对最终得分的贡献差不太多。BM25 的逻辑是，关键词只要出现了，就说明文档相关，出现太多是冗余，不需要线性加分。就像你找一本讲 Python 的书，书里「Python」这个词出现一万次还是两万次，区别不大——关键是有没有出现。

第二个变量叫逆文档频率，Inverse Document Frequency。这个更有意思。它衡量的是一个词在整个知识库里有多稀罕。越稀罕的词，检索价值越高。为什么呢？因为「的」「了」「是」「在」这些词，几乎每篇文档都有。你搜「的」，返回整个知识库，等于没搜。而「RAG」「Embedding」「Chunking」这些专业术语，在整个知识库里可能只出现在特定文档中。这些词才是真正有区分力的检索锚点。

用一个例子你就全明白了。假设知识库里有一篇文档介绍 Jansen 的 Agent 项目。用户搜「Jansen 的 Agent 框架有什么最新功能」。BM25 拿到这个查询，先拆词，然后看每个词。发现「的」这个字，整个知识库百分之九十九的文档都有，逆文档频率极低，约等于零，忽略。「Jansen」这个词，整个知识库只有三篇文档有，逆文档频率极高，权重拉到最高。「Agent」这个词，大约百分之二十的文档有，逆文档频率中等，有一定参考价值。「框架」和「最新功能」也是类似中等权重。最终 BM25 会根据这些权重，给每篇文档打一个分，排一个序。因为「Jansen」的权重最高，所有包含「Jansen」的文档都被推到最前面。

好，现在我们的 RAG 系统有了两套检索能力。Embedding 擅长语义理解——你说「编程助手」，它能找到「AI 编程工具」。BM25 擅长精确匹配——你说「Jansen」，它绝不给你「张三」。单用任何一套都有盲区。你说「帮我找一下去年第三季度的 API 文档」——Embedding 能理解「去年第三季度 API 文档」的语义，但可能搜到「前年第四季度」的文档因为语义相近。BM25 能精确匹配「2024-Q3-API」这个关键词，但搜不到文件名是「api_v2_spec」的文档、即使内容完全一样。

怎么办？合起来用。这就是 Hybrid Search，混合搜索的核心哲学——不赌单一检索策略，两路并行，互相补盲。

具体实现上，用户问一个问题，系统同时走两路。一路把问题做 Embedding，去向量数据库里做语义检索，返回语义最相近的 Top-K 个文档。另一路把问题做分词，用 BM25 去倒排索引里做关键词检索，返回关键词匹配度最高的 Top-K 个文档。两路各返回一个排序列表，然后需要一个融合算法，把两个列表合并成最终排序。

最常用的融合算法叫 RRF，Reciprocal Rank Fusion，倒数排名融合。它为什么叫这个名字？因为它的计算方式就是取排名的倒数。每个文档的最终得分，等于它在 BM25 排名中的倒数加上它在 Embedding 排名中的倒数。两路排名都靠前的，两个倒数加起来就大，最终排名就靠前。

这个算法有一个巨大的实用优势——它不需要归一化。BM25 的分数可能是零点几到几十，Embedding 的余弦相似度是负一到一之间，两个分数的量纲完全不同。如果直接加权相加，你需要调一个权重参数，而且这个参数可能随着知识库更新需要重新调。RRF 不关心原始分数是多少，只看排名，彻底绕过了这个问题。现在我们用的所有主流 RAG 框架——LangChain、LlamaIndex、Haystack——都内置了 RRF 融合，开箱即用。

讲到这里，我想跟你分享几个在实际工程中反复踩过的坑。这些坑，面试官不问，但你的 RAG 系统上线第一天就会遇到。

第一个坑，Embedding 模型和 Chunking 策略不匹配。什么意思？你用的嵌入模型，比如 BGE，它的最佳输入长度是五百一十二个 token。但你的 Chunking 把每块切到了一千五百 token。结果呢？嵌入模型在编码这些超长块的时候，后面的内容被截断了，向量里根本就没有后半段的信息。你检索的时候以为搜到了相关内容，实际上那些内容根本没进向量库。怎么避免？选 Chunking 策略前，先去查你的 Embedding 模型的最大输入长度。块的 token 数绝不能超过这个上限。

第二个坑，多语言知识库用同一个 Embedding 模型。你的知识库里中英文混在一起，用的却是只能处理好中文的 BGE 模型。英文文档的向量质量极差，检索的时候这些文档永远排不到前面。解决方案有两个，要么选多语言模型——比如 Jina 或者 OpenAI 的模型，要么按语言分库，中文一个向量库，英文一个向量库，各用各的模型。

第三个坑，也是最容易被忽略的——混合搜索的权重不是固定的。你一开始可能设 BM25 和 Embedding 各占一半权重。但上线后你会发现，用户搜索模式变了，新的文档类型进来了，之前的权重不灵了。混合搜索的权重是需要根据实际查询日志持续调的。如果你发现用户经常搜产品编号、工单号这些精确关键词，就该把 BM25 的权重调高。如果用户喜欢用自然语言问长问题，就该把 Embedding 的权重调高。没有一劳永逸的权重配比。

第四个坑，Chunking 的 Overlap 和 Hybrid Search 的 Top-K 之间有连锁反应。如果 Overlap 设大了，相邻块里有大量重复内容。Hybrid Search 检索的时候，两路都可能返回同一段重复内容的不同块。LLM 看到三遍一样的东西，浪费了宝贵的上下文窗口。所以调 Overlap 的时候，要同时考虑 Top-K 的设置。Overlap 越大，Top-K 应该越小，反之亦然。

这些坑听起来像是细节，但在生产环境里，一个细节没处理好，用户就会觉得「这个 AI 不太行」。而实际上不是你模型的问题，是工程细节没做到位。这也是为什么 Agent 工程师的面试里，越来越重视这些工程实践问题——能写 Demo 的人很多，能把 RAG 系统做到生产级的人很少。

好，我们来做一个知识连接，让今天的内容和前面几天的学习织成一张网。

第一个连接，Chunking 和 Day 3 的 Tokenization 分词。这两个概念看起来完全不同，但底层逻辑是一样的——都是把文本切碎。区别在哪？Tokenization 是把文本切成 token，是模型读文本的最小单元，通常在字符和子词级别。Chunking 是把文档切成段落块，是检索系统定位信息的最小单元，通常在段落级别。两者都是预处理的第一步，两者都说明同一个道理——切分的策略选错了，后续全链路都受影响。一个 Tokenization 切错了，模型读不懂；一个 Chunking 切错了，检索搜不准。

第二个连接，Overlap 和 Day 8 提到的 Lost in the Middle。Lost in the Middle 是说 LLM 对上下文中间位置天然关注度低，中部的信息容易被忽略。Overlap 是从索引层面减少信息丢失，保证即使信息在边界上也不会被丢掉。Lost in the Middle 是 LLM 自身的缺陷，靠模型架构无法根治；Overlap 是检索系统的防御措施，靠工程策略来弥补。两者的关系是，一个告诉你「模型有这个弱点」，一个告诉你「我们可以用这个方法来规避」。

第三个连接，BM25 和 Embedding。这不是替代关系，是互补关系。就像你同时拥有查字典的能力和问老师的能力——遇到一个不认识的字，翻字典最快；遇到一个不理解的概念，问老师最有效。好的 RAG 系统，字典和老师都要有。具体到你的项目里，如果你做的是企业内部知识库，里面充满了产品名、人名、项目编号这些 Embedding 不认识的东西，BM25 就是你必备的兜底方案。如果你的用户喜欢用自然语言问问题，经常换着方式问同一个事情，Embedding 就是你的核心能力。

最后一个连接，把 Chunking 和 Hybrid Search 放在 RAG 三段式里看。索引阶段——Chunking 做得好，块大小合适、有重叠保护，向量库里的数据质量就高。检索阶段——Hybrid Search 做得好，BM25 抓专有名词、Embedding 抓语义理解、RRF 融合两路结果，搜出来的内容就精准。生成阶段——Prompt 里加退出机制和源标注，LLM 就能基于高质量检索结果生成好答案。你会发现，前两个阶段做好了，第三阶段几乎不需要怎么调。

现在我们把今天学的所有东西串起来，做一个完整的端到端走查。假设你是一家公司的 Agent 工程师，老板让你搭一个内部知识库 RAG 系统。知识库里有什么呢？有产品文档、技术方案、会议纪要、API 手册，中英文混在一起的，大概十万篇文档。

你第一步做什么？先看文档类型。产品文档和 API 手册是结构化的，标题清晰、段落分明，用递归分块就行。会议纪要是对话形式的，段落边界不明显，可以先用固定长度分块快速出原型，后面再优化。Chunking 策略不是全局一刀切的——不同类型的文档，用不同的切法。

块大小怎么定？你看了一下用的是 BGE 模型，最大输入是五百一十二 token。所以每块控制在四百到五百 token，留出一些裕度。Overlap 设百分之十五，大概六七十 token 的样子。

第二步，建索引。中英文分开——中文文档用 BGE 做 Embedding，存到一个 Qdrant 集合里。英文文档用 E5 模型，存到另一个集合里。每个集合同时建一个 BM25 倒排索引。

第三步，搜索的时候。用户问「Jansen 上个月写的那个 Agent 框架 v2.3.1 的技术方案在哪」。系统收到问题，先判断语言——中文。走中文集合。两路并行：Embedding 检索去向量库里找语义相近的文档，BM25 去倒排索引里精确匹配「Jansen」「v2.3.1」「Agent 框架」。两路各返回 Top-20。RRF 融合两路排名，取 Top-5。这里为什么是 Top-20 粗筛再精排？因为混合搜索阶段多拉一些候选，给后面的 Re-ranker 留足选择空间。这个我们明天会详细讲。

第四步，生成。把 Top-5 文档块拼进 Prompt，加上系统指令——「基于以下参考资料回答，如果资料中没有相关信息，请说不知道。」加上源标注——每个引用后面标上文档名和章节。LLM 生成答案。用户看到的是有来源、可追溯、不编造的精准回答。

这个走查里，Chunking 策略决定了块的质量，决定了检索能搜到什么。混合搜索决定了搜得全不全、准不准。两者的选择都是在工程阶段做好的，一旦上线，改动的代价就大了。所以面试的时候，面试官问你「RAG 系统怎么设计」，你从这个走查的角度回答，从文档分析开始，到 Chunking 策略选择，到多路检索，到融合排序，到 Prompt 拼接——每一步都有据可依，不是拍脑袋的。

好，来总结一下今天的核心内容。第一，Chunking 是 RAG 索引阶段的第一步，核心权衡是语义完整性对检索精度。五百到一千 token 是常见起点。第二，三种分块策略——固定长度最简单但经常切断语义，递归分块是工程的默认首选，语义分块精度最高但成本也最高。第三，Overlap 是块边界的保护网，典型值是块大小的百分之十到二十。第四，混合搜索的意义在于补盲区——Embedding 搞不定专有名词和精确匹配，BM25 来补；BM25 搞不定语义改写和同义词，Embedding 来补。第五，RRF 做结果融合，简单有效，不依赖分数归一化。

明天我们进入 Day 10——Re-ranking 重排序与 RAG 深度优化。明天要学的 Cross-encoder 是一个非常巧妙的设计。你想想，Embedding 模型做检索的时候，是把问题和文档分别编码成向量，然后算距离。这像什么呢？像你把问题和文档各拍一张照片，然后看两张照片像不像。Cross-encoder 不一样，它把问题和文档一起输入模型，让模型从头到尾读一遍，然后输出一个相似度分数。这就像让一个专家同时看问题和文档，然后告诉你「这篇文档是否真的在回答这个问题」。粗筛用 Embedding 拍快照，精排用 Cross-encoder 请专家，两步走，这就是 RAG 检索的完整范式。明天见。
