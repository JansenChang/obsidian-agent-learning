明天我们要学的内容是 Tokenization，中文叫分词。在你开始正式学习之前，我先用这半个小时给你做一个完整的预习，把整个主题的框架、核心概念、关键问题都提前讲一遍，这样明天正式学习的时候你就能直接深入到细节了。

我们先来定位 Tokenization 在整个推理链条中的位置。还记得我们之前讲过的完整链路吗？你输入一段文字，第一件事就是被 Tokenization 切成一个个 token，然后每个 token 被转换成对应的数字 ID，接着进入 Embedding 层变成向量，再经过 RoPE 旋转位置编码、Attention 计算上下文关系、FFN 提取语义知识，最后 LM Head 映射到词表概率，Decoding 策略从概率分布中选一个词输出。Tokenization 就是这一切的起点。如果这个起点出了问题，后面的所有环节都会被波及。

到底什么是 Tokenization？简单说就是把一段文本切分成模型能处理的最小单元。这个最小单元叫 token。每个 token 对应词表里的一个唯一编号。模型不认识汉字，不认识英文字母，它只认识数字。所以你的输入"我今天吃了拉面"，必须先被转换成一串数字，比如 1234、5678、9012 这样的形式，模型才能处理。这个转换过程就是 Tokenization 在做的事。

举个例子。"我今天吃了拉面"这句话。如果按照中文的自然语义来切，最理想的分词结果是"我"、"今天"、"吃了"、"拉面"，这样四个有意义的 token。但如果分词器设计的不好，可能会把"我今天"切在一起、"吃了拉"切在一起，得到一些没有实际意义的 token。更极端的，如果分词器根本没有中文优化，每个字单独成 token，那么这句话就会被切成"我"、"今"、"天"、"吃"、"了"、"拉"、"面"七个 token。不同的分词方式，直接决定了模型看到的输入长什么样。

这里引出一个核心概念，叫做词表，英文叫 Vocabulary。词表就是模型认识的所有 token 的集合。你可以把它理解成一本字典，字典里每一页对应一个 token，页面上写着这个 token 的编号、向量表示和它在训练语料中出现的统计信息。一个中型模型的词表大小在三万到五万个 token 之间，大型模型的词表可能达到十万甚至二十万个 token。LLaMA 的词表是三万两千个 token，DeepSeek 的词表大约十万个 token，GPT-4 的词表大约十万个 token。词表越大，模型能表达的粒度越细，但也会带来更大的 Embedding 矩阵和 LM Head 层，增加模型体积和计算量。这是一个经典的工程权衡。

第二个核心概念叫 OOV，Out of Vocabulary，也就是词表之外的词。这个词听起来很学术，但它的含义很简单。如果用户输入了一个在模型词表中不存在的字或者词，比如一个新造的汉字，或者一个非常生僻的专业术语，分词器必须要有办法处理这种情况。如果不处理，模型会干脆不认识这个字，直接跳过去或者用占位符代替，导致信息丢失。那么好的分词器是怎么处理 OOV 的呢？答案是通过子词切分。即使一个词不在词表里，它组成的子部分可能在。比如"矽肺病"这个词不在词表里，但"矽""肺""病"三个字可能都在。分词器就会把它切成"矽""肺""病"三个 token，保证每个字都能被识别。

第三个核心概念叫压缩率。这个指标直接关系到你的 API 账单。压缩率是指模型用多少个 token 来表示一段自然语言文本。如果一个模型平均用一个 token 表示一点五个中文汉字，那么压缩率就是一点五。压缩率越高，同样的上下文窗口能放进去的内容就越多。不同模型的中文压缩率差异巨大。GPT-4 的中文压缩率大概是一点二到一点五。LLaMA 的中文压缩率大概是一点五到二点零。DeepSeek 作为中文优化模型，压缩率大概在一比一到一点二之间。这意味着同样一万个汉字的论文，GPT-4 需要大约一万三千个 token，LLaMA 需要大约一万八千个 token，DeepSeek 只需要大约一万一千个 token。如果你的 API 按 token 计费，模型的选择直接决定了你的使用成本。

现在我们正式进入算法的部分。Tokenization 有三种主流算法，BPE、WordPiece 和 Unigram。我会给你完整的讲解。

BPE 是最常用的一种，全称是 Byte Pair Encoding，字节对编码。它的核心思路非常直观。首先，把所有字符都拆散到最细的粒度，也就是每个独立的字节或者 Unicode 字符作为一个初始 token。然后，统计所有相邻字符对的出现频率，找到频率最高的那个字符对，把它们合并成一个新的 token。重复这个过程，直到词表的大小达到你的预设目标。

我直接给你一个例子。假设你的训练语料只有一句话"拉面很好吃拉面很贵"。第一步，拆成单个字符。得到"拉""面""很""好""吃""拉""面""很""贵"。第二步，统计相邻字符对的频率。"拉面"出现了两次，是频率最高的。第三步，把"拉面"合并成一个新 token。现在词表里有"拉面""很""好""吃""贵"五个 token。第四步，重新统计，发现"好吃"和"很贵"各出现一次，频率相同，可以选一个合并。重复直到词表达到目标大小。

通过这种方式，BPE 能自动发现语料中最常见的字词组合，把它们合并成长 token。不常见的组合则保持为短 token 或者单字符。这样词表的分配就自动向高频词汇倾斜了。

BPE 有一个明显的缺点。它是基于频率的，也就是说纯粹的统计驱动，不考虑语义。有些字词虽然经常一起出现，但合并成一个 token 后并没有语义上的好处。比如"的了"在中文中出现的频率很高，但把它们合并成一个 token"的了"没有任何意义。BPE 不会区分"这个合并有没有语义"，它只看频率。

为了克服这个缺点，Google 提出了 WordPiece。WordPiece 的思路跟 BPE 类似，也是从单个字符开始逐步合并。但合并的标准不是频率，而是互信息。互信息衡量的是两个字符放在一起比分开能多提供多少信息量。如果"拉"和"面"的互信息很高，说明它们合在一起比分开提供的信息更多，WordPiece 就会合并它们。如果"的"和"了"的互信息很低，说明合在一起也没什么额外信息，WordPiece 就不会合并它们。

BERT 用的就是 WordPiece。WordPiece 的最终效果通常比 BPE 好，因为它的合并标准更合理。但 WordPiece 也有自己的问题。每次合并都需要计算一对字符的互信息，而语料中字符对的数量是巨大的，计算开销远高于 BPE。所以很多公司在选择时宁愿用 BPE 加一些后处理规则，也不愿意承受 WordPiece 的高计算成本。

第三种算法叫 Unigram。它的思路跟 BPE 和 WordPiece 完全相反。Unigram 是从一个非常大的词表开始，然后慢慢删掉那些最不重要的 token，直到达到目标大小。怎么判断一个 token 重要不重要呢？Unigram 用一个语言模型来计算，如果去掉某个 token 后整个语料的损失增加不大，那这个 token 就是不重要的，可以删掉。

Unigram 有一个非常独特的优势。它天然支持子词正则化。什么意思呢？就是同一个词可以用多种不同的切分方式。比如"拉面"这个词，可以切成"拉面"一个 token，也可以切成"拉"和"面"两个 token。在训练阶段，Unigram 会每次都随机选择一种切分方式，这等于给模型注入了数据增强的效果，让模型看到更多的变异。SentencePiece 这个库使用的就是 Unigram 算法。

现在我们来谈谈中文分词的特殊性。这是一块极其重要但又经常被忽视的内容。

英文的分词太简单了。按空格切就行。最多加一个处理标点符号和特殊字符的规则。但是中文没有空格。每个字都可能独立成词，也可能跟其他字组合成词。而且中文的组词规则非常灵活，"我们"是一个词，"我"是一个词，"们"永远不会单独使用。这种灵活性给中文分词带来了巨大的挑战。

中文分词有三种主流的技术路线。第一种是基于词典的最大匹配法。准备一个非常大的词典，然后从左到右扫描文本，尽可能匹配最长的词。比如"我们今天学习了中文分词"，词典里有"我们""今天""学习""了""中文""分词"。最大匹配法从第一个字"我"开始，在词典里找以"我"开头的词，发现"我们"是词典中最长的匹配，就切出"我们"。然后从"今"开始，发现"今天"是最长的匹配。以此类推。这种方法的速度很快，但完全依赖词典的质量。如果词典里没有"深度分词"这个词，它就没办法把"深度学习"正确地切成"深度学习"，可能会切成"深度"和"学习"。

第二种是基于统计的方法。用大量标注好的语料训练一个统计模型，比如隐马尔可夫模型或者条件随机场。这些模型能从上下文中判断某个字要不要跟前面的字组合。例如"个人"这个词。"这是我个人的手机"里的"个人"应该是一个完整的词。"一个人来了"里的"一个人"应该切成"一"和"个人"。统计模型可以通过上下文做到这种消歧，这是基于词典的方法做不到的。

第三种是基于神经网络的方法。用 Transformer 或者 LSTM 来分词。现在很多主流的分词工具都已经转向了神经网络方法。这些方法的效果是最好的，但需要的计算资源和训练数据也是最大的。

中文分词还有一个特别出名的现象叫 Token 危机。我来给你一组真实的数据。"的"字在中文中的出现频率大约是每十个字就有一个，是所有汉字中频率最高的。它占了一个完整的 token 位置。英文中的"the"也占了一个 token 位置，但"the"在英文中的出现频率远低于"的"在中文中的出现频率。结果是什么？中文的"的"在生成的时候会消耗大量的概率空间，导致模型倾向于在所有地方都使用"的"字，即使其他表达方式更合适。这就是 Token 危机的本质——词表是一种零和博弈，你给了"的"一个位置，就相当于少给了其他字的位置。Token 危机不是一个比喻，它是真实存在的工程问题，直接影响着中文模型的生成质量。

另外一个密切相关的问题是罕见词汇的处理。有些专业术语在大型通用语料中出现的次数很少，比如"矽肺病""心肌梗死""梯度下降"，但这些词汇在实际应用场景中非常重要。BPE 的分词策略会根据频率来分配 token，高频词汇优先被合并成长 token，低频词汇则被拆成更细粒度的子 token。比如"梯度下降"可能在词表中不是作为一个整体存在的，而是被切成了"梯""度""下""降"四个 token。这增加了序列长度，但保证了每个字都能被识别。所以如果你做一个医疗问答系统，你其实应该关心你的模型的 tokenizer 能不能涵盖医学术语。如果不行，模型看"心肌梗死"这个词就像看四个独立的字符，它很难理解这是一个整体概念。

接下来我们来看不同模型之间的 tokenizer 对比。这是你选模型时必须考虑的因素。我先给你一组真实的数据。用同一个中文句子，GPT-4 消耗的 token 数量比 DeepSeek 少百分之二十左右。原因很简单——GPT-4 的词表更大，包含了更多中文字组合。DeepSeek 的词表虽然做了中文优化，但整体词表大小还是小一些，所以长词会被拆得更碎。这意味着，同样是 128K 的上下文窗口，你用 GPT-4 能放进去的中文内容比 DeepSeek 多百分之二十。

再来看 LLaMA。LLaMA 的词表只有三万两千个 token，是所有主流模型中词表最小的之一。为什么 LLaMA 故意把词表做小？因为词表越小，Embedding 矩阵越小，模型的整体体积就越小。Embedding 矩阵的参数量等于词表大小乘以向量维度。三万两千乘以四千零九十六，大约是一亿三千万个参数。十万乘以四千零九十六，大约是四亿一千万个参数。所以光词表这一块，LLaMA 就比 GPT-4 少了将近三亿个参数。代价是什么？LLaMA 的中文压缩率很差。同样的中文内容，LLaMA 消耗的 token 数量比 GPT-4 多百分之三十到五十。如果你在公司里部署 LLaMA 做中文应用，你会发现在同样的上下文窗口下，LLaMA 可能只能装下三分之二的内容。

DeepSeek 的 tokenizer 做了专门的中文优化。DeepSeek 在训练 BPE 时，语料中包含了大量的中文文本，所以中文字组合的覆盖率比 LLaMA 好很多。这也是为什么 DeepSeek 的中文表现比同等规模的 LLaMA 强的一个重要原因。从分词这一步开始，DeepSeek 就已经赢了。Qwen 的 tokenizer 也做了类似的中文优化，而且 Qwen 的词表达到十五万，是中文压缩率最高的模型之一。

GPT-4 的 tokenizer 使用了大约十万个 token 的词表，它的合并策略经过大量优化，中文和英文的压缩率都非常均衡。这也是 GPT-4 在中文任务上表现优秀的一个底层原因。

在英文分词上，这些模型的差异不大。因为英文天然按空格分，Tokenization 的挑战主要在于处理变形，比如 run、runs、running、ran 应该有关联。BPE 和 WordPiece 通常会把 run 和 ing、ning、ran 分别建成不同的 token，让模型自己学习这些变形之间的关系。

在实际项目中怎么选择合适的 tokenizer，这也是一个非常重要的实操问题。如果你是做中文聊天机器人，优先选择中文压缩率高的模型，比如 Qwen 或者 DeepSeek。如果你是做英文代码生成，LLaMA 的小词表反而是优势，因为模型体积更小，推理更快。如果你是做多语言产品，GPT-4 那种大词表更合适，因为它在中英文之间达到了更好的平衡。如果你的应用会输入大量专业术语，比如法律条文或者医学文献，你应该检查目标模型的 tokenizer 对这些术语的覆盖率。有些模型提供了交互式 tokenizer 工具，你可以直接输入一段专业文本，看看它被切成了多少个 token，有哪些术语被拆碎了。

关于 tokenizer，还有一个你一定会遇到的问题，就是特殊 token 的处理。模型在分词时除了处理正常的文字，还需要处理特殊标记。比如序列开始标记 BOS，序列结束标记 EOS，填充标记 PAD，未知标记 UNK。这些特殊 token 也占用词表的位置。有些模型有几十个特殊 token，有些有一百多个。Agent 场景下还有特殊的工具调用标记和思考链标记。DeepSeek 在 V3 版本中加入了专门的 thinking token 来处理 CoT 的边界。这些特殊 token 的增加会扩大词表，但它们在 Agent 场景下是必不可少的。

在实际编码中你还会遇到一个常见的问题，就是 token 数和字符数的换算。很多刚接触大模型开发的人会问，一段文本到底会消耗多少个 token？不同模型的换算关系不同，但有几个经验法则供你参考。在英文中，一个 token 大约等于零点七五个英文单词。在中英文混输中，一个 token 大约等于零点五到零点七个汉字。在纯中文中，一个 token 大约等于零点六到一点五个汉字，取决于模型的压缩率。如果你需要精确计算，最好使用模型自带的 tokenizer 工具，而不是靠估算。

我可以给你一个实际的工具使用技巧。大多数模型都提供了 encode 和 decode 方法。encode 就是把一段文本转换成 token ID 列表，decode 就是把 token ID 列表恢复成文本。你可以在代码里调用模型的 tokenizer 对一段文本做 encode，然后打印 token ID 列表的长度，这就是这段文本消耗的 token 数量。如果你发现一段常用文本消耗的 token 出乎意料地多，说明这个模型的 tokenizer 没有针对你的语言做好优化。

还有一些有趣的 tokenizer 技巧。比如有的开发者会在 prompt 里故意使用能让 tokenizer 输出更少 token 的措辞，从而降低 API 成本。如果某个模型对"因为我们有"这几个字消耗的 token 比较多，换成"由于"可能只消耗一个 token。这就是所谓的 prompt 工程中的 token 优化技巧。虽然我建议你优先关注模型质量而不是 token 数量，但在大规模部署时，这种微小的差异乘以几百万次调用，可能就是一笔可观的费用。

Tokenization 这个领域还在快速发展。有一些新的研究方向，比如字节级 tokenizer，直接从字节层面处理输入，完全绕开分词这个步骤。这种方法的优点是不需要任何语言的先验知识，一个 tokenizer 处理所有语言。缺点是序列长度会显著增加，因为每个字符需要多个字节来表示。另一个方向是多模态 tokenizer，把图像、音频和文字统一成同一个 token 空间。这已经在 GPT-4V 和多模态模型中实现，但它们使用的 tokenizer 通常还是文字 tokenizer 加图像编码器。

最后我帮你总结一下明天学习的重点。第一，BPE 是怎么从字符开始逐步合并成 token 的，合并规则是什么。第二，WordPiece 和 BPE 的核心区别是什么，为什么互信息比频率更好。第三，Unigram 为什么能支持子词正则化，它的删除策略是怎么工作的。第四，中文分词为什么比英文分词难这么多，三种主流的方法各有什么优缺点。第五，怎么判断一个 tokenizer 适不适合你的项目。

还有一个特别重要的实际话题，就是怎么在自己的项目里使用 tokenizer。假设你正在基于 HuggingFace 的 Transformers 库开发一个应用。你加载模型时，tokenizer 会自动跟随模型一起加载。你可以直接调用 tokenizer 的 encode 方法把文本转成 ID，调用 decode 方法把 ID 转回文本。你还可以通过 tokenizer 的 vocab_size 属性查看词表大小。这些操作每秒可以执行几万次，非常轻量，所以你不必担心它成为性能瓶颈。

但有一个隐藏的坑需要你注意。模型的 tokenizer 和模型的权重是紧密耦合的。你不能把 GPT-4 的 tokenizer 用到 LLaMA 的模型上，因为它们对同一个 token ID 的解释完全不一样。GPT-4 的 token ID 12345 可能是"拉面"，LLaMA 的 token ID 12345 可能是一个标点符号。所以 tokenizer 和模型必须配对使用，不能混用。

在实际的开发场景中，你可能会遇到需要自己训练 tokenizer 的情况。比如你的公司内部有一个专业领域的语料，通用模型的 tokenizer 在这个领域的压缩率很差。这时候你可以基于 SentencePiece 或者 Tokenizers 库，在自己领域的语料上训练一个定制化的 tokenizer。训练完成后，你还需要重新训练或者微调模型的 Embedding 层和 LM Head 层，因为词表变了，这两个层的参数量也变了。这是一个大工程，通常情况下没有必要——除非你的场景确实非常特殊，通用 tokenizer 完全无法覆盖。

HuggingFace 的 Tokenizers 库是目前最主流的 tokenizer 训练工具。它支持 BPE、WordPiece 和 Unigram 三种算法，训练速度极快，可以在几分钟内处理几十 GB 的语料。你只需要准备一个纯文本文件作为训练语料，然后指定词表大小和算法类型，剩下的 Tokenizers 会自动帮你完成。

还有一个大家经常忽略的点。模型的 tokenizer 训练用的是跟模型预训练一样的数据集。如果那个数据集里中文占比不高，tokenizer 的中文压缩率就不会好。这也是为什么有些模型虽然号称支持中文，但实际使用效果很差——不是模型本身的问题，而是 tokenizer 的语料分布就没给中文留够空间。

我们来看一个实际的数据。LLaMA 的预训练语料中英文占比大约是百分之六十七，中文只占不到百分之十。LLaMA 的 BPE tokenizer 在合并时自然更倾向于合并英文的常见组合，比如 th、ing、tion 这种。中文的常见字组合因为出现的频率不够高，很难被合并成长 token。所以你用 LLaMA 处理中文看到的基本上都是单字级别的 token，压缩率自然就差。

而 Qwen 的训练语料中中文占比超过了百分之三十。Qwen 的 tokenizer 在合并时就有足够多的中文数据可以用，所以它能合并出更多有意义的双字 token 和三字 token。这就是 Qwen 中文压缩率高的直接原因。
