前三天我们搭好了 Agent 的两条腿——ReAct 循环负责推理和决策，Function Calling 负责执行工具。但一个真正智能的 Agent 还缺一样关键的东西。它缺记忆。

你想象一下这个场景。你第一次跟一个 Agent 聊天，你说我叫张三，我在北京工作，我最喜欢的颜色是蓝色。Agent 用 ReAct 循环帮你查了北京的天气，给你推荐了蓝色的雨伞。你很满意。第二次你第二天打开它，又问同样的问题，它完全不记得你叫张三，也不记得你在北京，更不记得你喜欢蓝色。它像一条金鱼，每次对话都从零开始。你之前花时间告诉它的所有偏好、背景、历史决策，全部归零。这不是技术故障，而是它没有记忆系统。

记忆是让 Agent 从工具变成伙伴的那一层。今天我们就来拆解 Agent 的记忆系统到底怎么设计，面试官会问什么，以及工程上最常踩的坑是什么。

先建立一个三层模型。这个模型是理解一切记忆设计的基础。你把 Agent 的记忆想象成三个不同时间尺度的存储空间。

第一层叫工作记忆。这是 Agent 正在处理的任务状态。比如你问它帮我规划一个去东京的三天行程，它在 ReAct 循环中调了航班工具拿到了机票信息，调了酒店工具拿到了三家备选酒店。这些中间步骤的结果存在工作记忆里——下一步推理需要用到它们，但不是永久保存。任务完成后，工作记忆就释放了。这就像人大脑的前额叶，临时记住一个电话号码拨完就忘。

第二层叫短期记忆。这是一个会话内的完整对话历史。从你发的第一条消息到刚才最后一轮工具调用结果，都存在短期记忆里。Day14 讲的 ReAct 循环中传给 LLM 的消息列表就是短期记忆的载体。短期记忆的大小受限于上下文窗口——GPT 可能是十几万 token，Claude 可能是二十万 token。在这个窗口内，所有历史都在。

第三层叫长期记忆。这是跨会话的、持久化的记忆。昨天你说你在北京工作，今天你打开新的会话，Agent 还能记住这件事。长期记忆通常存在向量数据库或者专用记忆系统里。它的关键设计是将用户信息编码成可检索的形式，在需要的时候取出来注入到对话中。

这三层不是简单的上下级关系，而是时间刻度的差异。工作记忆管秒级到分钟级的任务状态。短期记忆管分钟级到小时级的单次会话。长期记忆管天级到年级的跨会话积累。面试中如果你能把这个时间维度讲清楚，比你背一堆框架名词更有说服力。

好，现在逐层深入。

工作记忆的实现其实不复杂。你在 ReAct 循环中，每一次工具调用的结果、每一个 Thought 的内容，在没有返回给用户之前都是工作记忆。工程上就是程序内存中的变量和中间结果。当最终答案返回给用户，这些变量就可以丢弃。但这里有一个面试中很常见的问题——工作记忆需要多大？什么情况下需要持久化？

大多数场景下工作记忆就是程序变量的生命周期，不需要显式的存储。但如果你的 Agent 执行的任务可能跨多轮——比如用户说你先帮我规划行程，我晚上再告诉你最终决定——这时候工作记忆就不能在回答后丢弃了。你需要把它序列化到某个短暂的持久层，比如 Redis，设一个较短的有效期。这其实就是工作记忆和短期记忆之间的灰色地带。面试官如果问你怎么处理长时间任务的状态管理，你就从这个角度切入。

短期记忆是面试中考查频率最高的部分。它的实现有四种经典模式，每种有各自的取舍。

第一种叫全量缓冲模式，对应 LangChain 里的 ConversationBufferMemory。最简单粗暴——把整个对话历史原封不动地存下来。每一轮都把所有消息传给 LLM。好处是信息完整，LLM 能看到全部上下文。坏处是随着对话增长，token 消耗线性膨胀。一个用户跟客服 Agent 聊了三十分钟、五十轮对话，消息列表可能塞了好几万 token。每次 LLM 调用都全量传入，成本很快就爆了。而且还有个隐蔽的问题——LLM 对长上下文中间部分的信息容易忽略，这就是 Day13 我们讲的 Lost in the Middle。所以全量缓冲只在短对话场景里好用。

第二种叫滑动窗口模式。只保留最近若干轮的对话，超过窗口的旧消息直接丢弃。比如设置窗口大小为十轮，那第十一轮的 Thought 只能看到第二轮到第十一轮的信息，第一轮的内容被丢弃了。好处是 token 消耗恒定，不随对话增长。坏处是早期的关键信息可能被丢弃——比如用户第一轮说了我叫张三，第十二轮 Agent 问请问怎么称呼，丢了就尴尬了。

第三种叫摘要压缩模式。每当对话达到一定长度，不是丢弃旧消息，而是用一个 LLM 调用来把旧消息压缩成一段摘要。摘要保留了关键信息但丢掉了精确的原文。后面的推理基于摘要而不是完整历史。好处是既控制了 token 又不完全丢失信息。坏处是摘要的质量直接决定了记忆的质量。如果摘要把张三在北京工作压缩成了张三在中国，精度损失了。另一个问题是每次压缩需要额外的一次 LLM 调用，增加了延迟和成本。

第四种叫 token 数截断模式。跟滑动窗口类似，但不是按轮数截断，而是按 token 数截断。设置一个 token 上限，消息从最早的一条开始依次删除，直到总 token 数低于上限。这个模式更精细，因为不同轮次的对话长度差异可能很大，按轮数可能要么太紧要么太松，按 token 数刚好卡在模型能处理的范围内。

面试官问你怎么选的时候，标准答案是看场景。客服场景对话短且信息密度低，滑动窗口就够了。教育辅导场景需要记住学生的薄弱点，摘要压缩更合适。代码审查场景对话极长但每轮信息量巨大，token 截断最实用。大多数生产系统是混合模式——最近几轮用全量保留，更早的用摘要，再早的丢弃。核心思想是越近的信息越需要细节，越远的信息只需要要点。

现在来看长期记忆。这是 Agent 从工具变成伙伴最关键的跃迁。没有长期记忆，你每次跟 Agent 对话都是一次全新的自我介绍。有了长期记忆，Agent 能记住你的偏好、你的历史决策、你喜欢什么方案不喜欢什么风格。

长期记忆的工程实现本质上是一个存储加检索的系统。当对话中产生了值得记住的信息——比如用户提到我在北京工作——这个信息被抽取出来，编码成向量，存到向量数据库里。下次用户开始新对话时，或者在当前对话中切换到新话题时，Agent 用当前上下文去检索相关的历史记忆，把检索结果注入到短期记忆或直接注入到系统提示词里。

这就是为什么长期记忆和 RAG 有非常深的关系。在 Day8 到 Day13 我们讲的 RAG 是面向外部知识库的——把企业文档切成块存起来，用户提问时检索相关块。长期记忆是面向用户个人的 RAG——把用户的事实、偏好、历史决策存起来，对话需要时检索。两者用了完全相同的技术栈——Embedding 模型编码、向量数据库存储、相似度检索。区别只是数据的来源和范围不同。

当前最受关注的长期记忆系统之一是 Mem0。Mem0 的设计思路非常值得在面试中引用，它体现了工程化的记忆系统需要解决的几个核心问题。

第一个问题是自动提取。你不能指望用户在对话中每次说一句记住我喜欢蓝色 Agent 就真的记住。你需要一套机制自动从对话中抽取值得记住的信息。这通常用另一个 LLM 调用来做——把最近的几轮对话发给 LLM，让它判断这段对话中出现了哪些应该被记住的用户事实，返回一个结构化的事实列表。每条事实包含内容本身、事实类型，比如偏好、背景、决定，还有一个重要性评分。

第二个问题是去重和更新。用户可能在不同时间说了类似的信息——第一次说我住在北京，第二次说我搬到上海了。新信息应该覆盖旧信息，而不是在数据库里存两条矛盾的记录。Mem0 的做法是每次提取新事实时，先在已有记忆中搜索相似的内容，如果发现冲突或更新，自动用新的替换旧的。这个机制叫记忆合并。

第三个问题是层级化存储。不是所有记忆都是一样的。用户说我讨厌香菜可能比用户说我昨天喝了一杯拿铁更重要。香菜偏好是长期有效的偏好信息，喝拿铁是一次性的事件。层级化存储把记忆分成不同的重要性层级——频繁使用的偏好放最前面，偶尔有用的事件放后面。检索时自动优先召回高层级的记忆。

面试中如果你提到 Mem0 的这些设计逻辑，最好顺便讲一个具体的工程案例。比如说——实现一个智能健身教练 Agent。用户每次训练完说今天深蹲做了一百公斤五组五下。Agent 需要记住这个数据，下次训练时自动建议适当加重。同时还要记住用户三个月前受过膝伤，所以虽然力量在增长，训练计划需要避开高冲击动作。这里需要两层记忆——训练数据是精确数值要存下来，伤病情况是高优先级的约束条件不能丢。然后在下一次 ReAct 循环开始前，Agent 先检索出最近的训练数据和伤病约束，注入到 Thought 里，再开始推理今天的训练方案。

这就是长期记忆和 ReAct 循环的整合方式。记忆不是配角，是 Thought 环节的燃料。LLM 在每一次推理前，不仅看到当前对话历史，还看到从长期记忆中检索出来的相关上下文。推理的质量、工具选择的准确性，都依赖于记忆检索的精度。如果检索出来的记忆是错的或者过时的，Agent 的整个推理链都会跑偏。

这里就引出一个面试中非常有深度的话题——记忆质量怎么评估。你不能只看 Agent 的最终答案对不对，还要看它调用的记忆对不对。这和 Day15 我们讲的轨迹评估是一个道理——推理质量和结果质量必须分开评估。记忆质量评估需要考察三个指标。第一个是召回率——该记的事实记了没有。第二个是精度——记下来的事实有没有错的。第三个是时效性——用户搬家后旧地址是应该被更新还是保留。

工程上还有一个非常棘手的问题，很多面试官喜欢问——长期记忆会无限膨胀怎么办？用户用了一年 Agent，每天产生几十条新记忆，一年下来几万条。检索效率下降，存储成本增长，旧记忆可能已经过时了。你怎么处理？

答案是遗忘策略加分级存储。遗忘不是简单的删除，而是基于几个维度来决策。时间维度——三个月前的某次对话细节大概率已经没有价值了，但用户的核心偏好不受时间影响，比如用户说我对花生过敏是一年前的，这条绝不能删。重要性维度——每条记忆被 LLM 提取出来的时候就附带了一个重要性分数。高频被检索的记忆加分，从未被检索过的降分。降到一个阈值就自动归档到冷存储或者直接删除。容量维度——当向量数据库的存储量超过设定上限时，按重要性分数从低到高淘汰。这个机制跟操作系统的内存页面置换算法在思想上完全一致。

遗忘策略的设计还有一个非常容易被忽略的细节——用户主动删除。根据隐私法规，用户有权利要求 Agent 忘记某些信息。你的记忆系统必须支持按用户 ID 精确删除所有相关记忆。而且不仅是删数据库里的记录，还要考虑记忆可能已经被转换成了摘要注入到了某次对话历史中，那些已经发生的对话你改不了但你至少可以确保下次新对话不会再检索到。

再多讲一个面试高频话题——记忆注入的时机和方式。记忆到底什么时候注入？是每次 LLM 调用前都查一次，还是只在对话开始时查一次，还是在 ReAct 的每一步 Thought 之前查一次？

每次 LLM 调用前都查一次是最安全但也是最贵的。如果一次对话中有五次 ReAct 循环，那就是五次检索。大部分场景不需要这样。最实用的做法是分两层。在对话开始时做一次全量检索，把所有高度相关的长期记忆一次性注入到系统提示词里。中途如果对话方向发生了明显的改变——比如用户从咨询健身切换到咨询饮食——再做一次增量检索补充新的记忆。对于 ReAct 循环中的关键决策点，比如 Agent 准备调用一个写操作的工具，做最后一次记忆校验确保没有违背用户的偏好约束。

这个设计问题的核心是延迟和准确性的平衡。检索越多，信息越全，但延迟越高成本越大。没有标准答案，看场景。客服 Agent 用户通常一次对话只问一个主题，对话开始查一次就够。私人助理 Agent 用户可能在一次对话里处理三个完全不相关的任务，需要多次检索。

好，把今天的内容做一个系统总结。

Agent 的记忆系统分为三层。工作记忆是 ReAct 循环中的中间变量，会话结束就释放。短期记忆是当前对话的完整历史，受上下文窗口限制。长期记忆是跨会话的持久化用户信息，存在向量数据库里。

短期记忆有四种实现模式——全量缓冲最简单但成本高，滑动窗口最省但丢旧信息，摘要压缩平衡但质量依赖 LLM，token 截断最精确。生产系统通常是混合模式。

长期记忆本质上是用户专属的 RAG。自动提取事实，去重合并，层级存储，按需检索。Mem0 的设计是当前最佳实践的集中体现。

遗忘策略是长期记忆必须面对的问题。时间衰减、重要性评分、容量淘汰、用户主动删除，四道防线各自解决不同层面的膨胀。

记忆质量的评估需要看召回率、精度和时效性。推理轨迹评估比只看最终结果更能发现记忆的问题。

最后也是最重要的——记忆注入的时机是工程决策。全量检索最全但最贵，按需检索最省但可能遗漏。方案取决于场景和延迟预算。

再多讲几个面试中最容易被追问的细节。

第一个，记忆更新的时机。在 ReAct 循环中，什么时候把新信息写入长期记忆？是每次工具调用完就更新，还是整个对话结束才更新，还是每隔几轮批量更新？

每次工具调用完就更新是最实时的，但会引入一个严重的问题——如果在多轮循环的中间步骤更新了记忆，LLM 下一次 Thought 就会看到这些新记忆，导致循环行为发生变化。这种变化可能是好的，也可能是坏的。如果你的记忆提取出了错误的事实——比如把用户说过的假设当成事实存进去了——后续的循环就会被污染。所以大多数系统采用的是对话结束后批量异步更新。整个对话结束，LLM 确认了最终答案，然后一个后台任务把整段对话发给另一个 LLM 来提取事实，写入长期记忆。异步更新不会影响当前对话的循环行为，也更安全。

但有一个例外。如果用户在当前对话中说了非常关键的信息——比如我是公司的管理员账号——而这个信息可能在本轮对话后续的循环中就需要用到，那必须同步更新。这种高优先级事实应该立即写入记忆缓存，让后续的 Thought 能马上看到。

第二个细节，多用户场景下的记忆隔离。如果你的 Agent 是服务于多个用户的，每个用户的记忆必须严格隔离。你不能让用户 A 的记忆出现在用户 B 的检索结果中。技术上有两种实现方式。第一种是在每条记忆的元数据里打上用户 ID 的标签，检索时加上用户 ID 过滤。第二种是每个用户独立的向量空间，物理隔离。小规模用标签过滤就够了，大规模或者有严格隐私要求的场景用物理隔离。

第三个细节，记忆检索的结果太多了怎么办。一次检索可能返回几十条相关的历史记忆。你不能全部塞进系统提示词。你需要一个排序和截断的机制。相关性加重要性加权打分，取 top K。而且 K 的值不是固定的——对话越复杂，需要越多的记忆上下文。一个工程化的做法是根据当前对话的复杂度动态决定 K 的值。如果用户只是在聊天，K 取三到五个就够了。如果用户在做复杂的决策，比如帮我对比三种保险方案的优劣，K 可能需要十到十五个。

还有一个更高阶的概念值得了解——记忆的自我进化。一个真正成熟的记忆系统不只是被动地存储和检索，而是能主动从记忆中提炼出更高层次的知识。比如用户在过去三个月里五次提到喜欢简洁的设计风格，记忆系统不只存储这五条独立事实，而是自动提炼出一条总结——该用户偏好简洁设计风格。这个提炼过程本身也可以用 LLM 来做。当某个话题相关的记忆数量超过一个阈值，触发一次记忆总结，生成一条高层级的元记忆。这就是记忆从数据到知识到智慧的跃迁。

这个概念在 Mem0 里叫 Memory Layers，在 Zep 里叫 Memory Synthesis。面试中如果你能从这个角度讨论记忆系统，展现的不仅是工程能力，还有对产品设计意图的理解。

来讲一个非常实用的面试题。面试官问你——请设计一个医疗问诊 Agent 的记忆系统，你觉得最关键的三个设计决策是什么？

这个问题考的是你是否能把通用记忆框架映射到具体的行业场景。好的回答是这样的。第一个关键决策是隐私和合规。医疗数据的隐私保护是法律红线。记忆系统必须在用户首次使用时明确告知哪些数据会被记录，记录的目的是什么，并且支持用户随时查看自己的记忆数据和请求删除。技术上需要实现记忆的细粒度权限控制和完整的审计日志。

第二个关键决策是信息的准确性和时效性。用户可能说上个月血糖有点高，也可能说三年前做过一个手术。这两类信息的时间敏感性完全不同。血糖数据三个月前的已经无效了，但手术史是终身有效的。记忆系统需要给不同类型的信息打上不同的时效性标签。

第三个关键决策是记忆的类型划分。医疗对话中的信息至少分三类——个人基本信息，比如年龄性别过敏史，需要永久精准保留。症状描述，比如头疼三天了，需要对时间敏感，旧症状自动降权。对话中的决策和推论，比如 Agent 根据症状建议你去医院检查，这条应该保存但不是永久性的，一个月后可以归档。

这三个决策分别对应记忆系统的安全性、时效性和粒度设计，展现的是有行业落地经验的工程思维。

最后我们来建立一些知识连接。

Day8 到 Day13 讲的整个 RAG 系统，从 Chunking 策略到向量数据库选型到 Rerank 精排，全部可以直接复用到长期记忆的检索管道中。你把用户事实当成文档块，用同样的 Embedding 编码，用同样的向量数据库存储，用同样的 Rerank 做精排。唯一的区别是存储器里的内容是外部文档还是用户事实。所以你在 RAG 上积累的工程经验，改两行配置就能搭出一个长期记忆系统。

Day14 写的 ReAct 循环中，短期记忆的载体就是消息列表。而长期记忆需要在每次循环开始前或每个关键 Thought 之前注入。注入时机直接影响循环的质量，需要在延迟和准确性之间做权衡。

Day15 讲的 Agent 评估三层次对记忆系统同样适用。组件级别评估——每个记忆提取和检索的模块单独测。轨迹级别评估——检查 Agent 的每一步推理是否使用了正确的记忆。端到端评估——对话质量是否因为记忆而提升。特别要注意的是，记忆系统的提升在端到端评估上是滞后的——你今天改善了记忆提取逻辑，可能要一周后用户再来对话才能真正评估效果。

明天我们要进入 Agent Loop 设计——ReAct 循环只是一个起点。真正的 Agent 系统如何管理多次循环中的上下文、如何工程化 Context Window 的分层利用，以及 Google 提出的 Context Engineering 四大策略到底是什么。这些是你面试中回答系统设计题的真正弹药。
